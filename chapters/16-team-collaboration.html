团队协作、MLOps 与面试准备

从个人开发者到团队工程化：掌握团队协作模式、CI/CD 流程、评估体系，并为职业发展做好全面准备

👥
Agent 开发的团队角色与分工
构建高效 Agent 团队的完整角色图谱与协作模式

Agent 开发不同于传统的软件工程——它要求团队成员在提示词设计、模型行为理解、工具集成和产品思维之间灵活切换。理解每个角色的职责和协作方式，是构建高质量 Agent 产品的组织基础。以下是典型 Agent 团队中的核心角色及其职责：

角色 核心职责 关键技能 与谁协作最密切
Agent 工程师 设计系统提示词、定义工具接口、编排 Agent 循环、选择模型、调优行为、调试 LLM 输出异常 提示词工程、Python/TypeScript、LLM API 使用、工具设计、模型行为调试 Agent Ops、Agent PM、领域专家
Agent 运维 (Agent Ops) 部署 Agent 到生产环境、监控性能指标、管理成本预算、处理线上故障、实施安全策略、管理 CI/CD K8s/Docker、CI/CD、可观测性工具、成本优化、事件响应 Agent 工程师、SRE 团队
Agent 产品经理 (Agent PM) 定义 Agent 的使用场景、设定成功指标、收集用户反馈、优先级排序、管理 Agent 的行为边界和"人格" 用户研究、数据分析、LLM 能力边界理解、评估集设计、原型验证 Agent 工程师、领域专家、UX 设计师
领域专家 (Domain Expert) 提供专业领域知识、审核 Agent 输出的准确性、构建评估集、标注训练数据、定义"好的回答"的标准 特定领域的深厚知识（如法律、医疗、金融）、评估与标注能力 Agent 工程师、Agent PM
Agent 安全工程师 提示注入防护、输出内容审核、工具权限控制、数据合规审查、红队测试、威胁建模 安全工程、渗透测试、LLM 安全研究、数据隐私法规 Agent 工程师、Agent Ops、法务团队

典型团队协作工作流

在一个成熟的 Agent 团队中，工作流通常如下：Agent PM 与领域专家一起定义用户故事和成功标准。Agent 工程师基于这些需求设计提示词和工具。安全工程师对设计进行威胁建模审查。Agent Ops 设置 CI/CD 管道和监控。每两周一个迭代，通过评估集量化进展。这种跨角色的紧密协作是 Agent 产品质量的根本保障。

小团队如何运作

在早期创业团队中，一个人可能同时承担 2-3 个角色。Agent 工程师往往同时是 Agent Ops，Agent PM 同时是领域专家。关键是意识到每个角色承担的不同责任，而不是忽视它们。即使是单人项目，也应该在"Agent 工程师模式"和"Agent PM 模式"之间有意识地切换——先戴上 PM 的帽子定义成功标准，再戴上工程师的帽子去实现。

提示词版本管理 — Treat Prompts as Code
用 Git 管理提示词生命周期，实现团队协作、语义化版本与 A/B 测试

提示词是 Agent 的"源代码"中最关键但也最容易被忽视的部分。将提示词纳入版本管理体系是专业 Agent 团队的第一步。与代码不同，提示词的一个词的变化可能导致完全不同的行为——"请"改为"务必"可能使 Agent 从建议型变为命令型，"你应该"改为"你可以"可能使其从权威型变为建议型。

提示词 Git 工作流

  • 独立文件存储：提示词存储在独立文件中（如 prompts/system-prompt-v3.txt），与代码一起进行版本控制。避免将提示词硬编码在 Python/JS 源码中
  • PR 审查流程：每次修改提示词都需要 Pull Request，由至少一名同事进行 Code Review。PR 描述中必须包含：修改原因、预期行为变化、在评估集上的对比结果
  • 语义化版本控制 (Semantic Prompt Versioning)：Major 版本改变核心行为模式（如从"热情主动"改为"专业简洁"）；Minor 版本增加新功能指导或工具使用说明；Patch 版本修复措辞问题和特定场景下的错误行为
  • 回滚机制：每次提示词变更都应可独立回滚。提示词版本与代码版本解耦，通过配置中心（如 LaunchDarkly、自建配置服务）动态切换
  • A/B 测试集成：在生产环境中同时运行两个版本的提示词，将流量按 10%/90% 或 50%/50% 分配，通过统计显著性检验后决定推广或回退
Python — PromptRegistry 类：提示词版本化与 A/B 测试 
import hashlib
import random
from datetime import datetime
from dataclasses import dataclass, field
from typing import Optional, Dict, List
import difflib

@dataclass
class PromptVersion:
    version: str
    content: str
    author: str
    description: str
    created_at: datetime
    eval_score: Optional[float] = None

class PromptRegistry:
    """提示词注册中心 — 管理提示词的全生命周期

    功能：创建新版本、回滚到历史版本、A/B 测试流量分配、
    语义化 Diff、版本统计、基于用户哈希的一致性分组
    """

    def __init__(self, prompt_name: str):
        self.name = prompt_name
        self.versions: List[PromptVersion] = []
        self.current_version: Optional[str] = None
        self.ab_config: Optional[Dict] = None
        self.rollback_history: List[Dict] = []

    def register_version(self, version: str, content: str,
                          author: str, description: str):
        if self.get_version(version):
            raise ValueError(f"版本 {version} 已存在")
        pv = PromptVersion(version=version, content=content,
                              author=author, description=description,
                              created_at=datetime.now())
        self.versions.append(pv)
        if self.current_version is None:
            self.current_version = version
        print(f"[PromptRegistry] 注册版本 {version} — {description}")
        return pv

    def get_version(self, version: str) -> Optional[PromptVersion]:
        for pv in self.versions:
            if pv.version == version:
                return pv
        return None

    def activate(self, version: str):
        pv = self.get_version(version)
        if not pv:
            raise ValueError(f"版本 {version} 不存在")
        prev = self.current_version
        self.current_version = version
        self.ab_config = None
        print(f"[PromptRegistry] 版本激活: {prev} → {version}")

    def rollback(self, reason: str = "手动回滚") -> Optional[str]:
        if len(self.versions) < 2:
            return None
        idx = next(i for i, pv in enumerate(self.versions) if pv.version == self.current_version)
        target = self.versions[idx - 1]
        self.rollback_history.append({
            "from": self.current_version,
            "to": target.version,
            "reason": reason,
            "time": datetime.now().isoformat()
        })
        self.activate(target.version)
        return target.version

    def setup_ab_test(self, version_a: str, version_b: str,
                         b_percentage: float = 0.1):
        if not (self.get_version(version_a) and self.get_version(version_b)):
            raise ValueError("A/B 版本不存在")
        self.ab_config = {
            "version_a": version_a, "version_b": version_b,
            "b_percentage": b_percentage,
            "started_at": datetime.now()
        }

    def get_prompt_for_request(self, user_id: str) -> str:
        """根据当前版本和A/B测试配置返回提示词。使用用户哈希保证Sticky Bucketing"""
        if self.ab_config:
            hash_val = int(hashlib.md5(user_id.encode()).hexdigest(), 16)
            bucket = (hash_val % 10000) / 10000
            version = self.ab_config["version_b"] if bucket < self.ab_config["b_percentage"] else self.ab_config["version_a"]
        else:
            version = self.current_version
        return self.get_version(version).content

    def diff_versions(self, v1: str, v2: str) -> str:
        pv1, pv2 = self.get_version(v1), self.get_version(v2)
        if not (pv1 and pv2):
            return "无法比较"
        diff = difflib.unified_diff(
            pv1.content.splitlines(), pv2.content.splitlines(),
            fromfile=v1, tofile=v2, lineterm=""
        )
        return "\n".join(diff)

# 使用示例
registry = PromptRegistry("customer-support-agent")
registry.register_version("1.0.0", "你是一个乐于助人的客服助手...", "张三", "初始版本")
registry.register_version("1.1.0", "你是一个专业高效的客服助手...", "李四", "调整语气为简洁专业")
registry.setup_ab_test("1.0.0", "1.1.0", b_percentage=0.10)
提示词变更的风险

与传统代码不同，提示词的变更无法通过单元测试完全覆盖。同一个提示词在面对不同用户输入时可能表现出截然不同的行为。因此，提示词的发布必须经过比代码更严格的金丝雀部署和更长的观察期——建议至少观察 24 小时的线上指标（错误率、完成率、满意度）后才能全量发布。

Agent 的 CI/CD 管道 — 从提交到生产的自动化
提示词 Linting → 评估 → Staging 部署 → 冒烟测试 → 金丝雀 → 全量发布

Agent 的 CI/CD 需要比传统软件更丰富的检查环节，因为"正确性"对 Agent 来说是一个模糊的概念——没有"预期输出等于实际输出"的简单断言。需要构建多阶段的自动化管道：

CI/CD 流水线的六个阶段

  1. Stage 1 — 提示词 Linting：检查 Token 数量是否在预算内、占位符变量是否完整、工具名称引用是否有效、是否存在硬编码的测试值或过期信息。这是一个纯静态分析阶段，成本几乎为零
  2. Stage 2 — 工具单元测试：测试工具函数的输入输出格式、异常处理逻辑、Mock 外部依赖验证 Agent 循环逻辑
  3. Stage 3 — 评估集运行：在评估集上运行 Agent，计算任务完成率、工具调用准确率、幻觉率。每个评估类别的通过率都必须在阈值以上（如 basic > 95%, edge-case > 85%, adversarial > 70%）
  4. Stage 4 — Staging 部署：部署到 Staging 环境，运行冒烟测试——一组精心挑选的真实用户场景，验证 Agent 在类生产环境中的端到端行为
  5. Stage 5 — 金丝雀部署：将 5% 生产流量导向新版本，自动监控关键指标 1-4 小时。指标包括：错误率（不超过基线 1.5x）、P95 延迟（不超过基线 1.3x）、任务完成率（不低于基线 0.9x）
  6. Stage 6 — 全量发布或自动回滚：金丝雀通过后全量发布；如任何指标超阈值，自动触发回滚到上一个稳定版本，并创建 Incident Ticket 通知值班工程师
YAML — GitHub Actions: Agent CI/CD Pipeline 
name: Agent CI/CD Pipeline

on:
  push:
    branches: [main, staging]
    paths:
      - 'prompts/**'
      - 'tools/**'
      - 'agents/**'
      - 'evals/**'
  pull_request:
    branches: [main]

env:
  ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}

jobs:

  prompt-lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with: python-version: '3.12'
      - name: Install deps
        run: pip install pyyaml tiktoken
      - name: Lint prompts
        run: |
          python scripts/lint_prompts.py \
            --prompts-dir ./prompts \
            --max-tokens 8000 \
            --check-placeholders \
            --check-tool-refs

  run-evals:
    needs: prompt-lint
    runs-on: ubuntu-latest
    strategy:
      matrix:
        eval-set: [basic, edge-cases, adversarial, real-user-queries]
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with: python-version: '3.12'
      - name: Install deps
        run: pip install -r requirements.txt
      - name: Run evals
        run: |
          python scripts/run_evals.py \
            --eval-set evals/${{ matrix.eval-set }}.jsonl \
            --agent-config agents/current.yaml \
            --parallel 4
      - name: Check thresholds
        run: |
          python scripts/check_eval_thresholds.py \
            --results-dir eval-results/${{ matrix.eval-set }} \
            --min-completion-rate 0.85 \
            --min-tool-accuracy 0.90 \
            --max-hallucination-rate 0.05

  deploy-staging:
    needs: run-evals
    if: github.ref == 'refs/heads/staging'
    runs-on: ubuntu-latest
    steps:
      - name: Deploy to staging
        run: |
          curl -X POST https://api.internal/deploy \
            -H "Authorization: Bearer ${{ secrets.DEPLOY_TOKEN }}" \
            -d '{"environment":"staging","version":"'"${GITHUB_SHA::7}"'"}'
      - name: Smoke test
        run: python scripts/smoke_test.py --test-cases tests/smoke.jsonl

  deploy-production:
    needs: deploy-staging
    if: github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
    steps:
      - name: Canary deploy (5% traffic)
        run: |
          python scripts/canary_deploy.py \
            --version "${GITHUB_SHA::7}" \
            --traffic-percentage 5 \
            --observation-minutes 60 \
            --auto-rollback true
      - name: Full rollout (100% traffic)
        run: |
          python scripts/canary_deploy.py \
            --version "${GITHUB_SHA::7}" \
            --traffic-percentage 100

  notify-failure:
    needs: [prompt-lint, run-evals, deploy-staging, deploy-production]
    if: failure()
    runs-on: ubuntu-latest
    steps:
      - name: Alert on failure
        run: |
          curl -X POST ${{ secrets.SLACK_WEBHOOK }} \
            -H "Content-Type: application/json" \
            -d '{"text":"Agent CI/CD 流水线失败\nCommit: ${{ github.sha }}\n请查看: ${{ github.server_url }}/${{ github.repository }}/actions/runs/${{ github.run_id }}"}'

Agent 设计审查清单 — 30+ 项全面检查
确保每个 Agent 在发布前通过安全、可靠性、成本和 UX 审查

以下是一个全面的 Agent 设计审查清单，覆盖五大维度共 32 项检查。在每次 Agent 发布前应对照此清单逐项检查：

一、安全性 (Safety) — 9 项

  1. 系统提示词是否包含明确的注入防护指令？
  2. 是否对用户输入进行了内容审核和敏感词过滤？
  3. 工具权限是否遵循最小权限原则（每个工具只授予必要权限）？
  4. 高风险操作（删除、支付、发送邮件）是否需要用户二次确认？
  5. Agent 输出是否经过 PII（个人身份信息）检测和脱敏？
  6. 是否设置了请求频率限制（Rate Limiting）？
  7. 代码执行是否在隔离沙箱中进行（Docker/gVisor）？
  8. 是否通过了提示注入和越狱红队测试？
  9. 是否记录并保留了完整的审计日志？

二、可靠性 (Reliability) — 7 项

  1. 所有 LLM 调用是否有指数退避重试机制？
  2. 是否设置了合理的超时时间（工具调用 30s，LLM 调用 60s）？
  3. 是否有主模型不可用时的降级策略（Fallback Model）？
  4. Agent 循环是否有 max_steps 硬限制？
  5. 是否检测并防止工具调用死循环？
  6. 是否对 LLM 返回的 JSON/结构化输出进行了格式校验？
  7. 会话状态丢失时是否有优雅的降级提示？

三、成本控制 (Cost) — 6 项

  1. 是否启用了 Prompt Caching（系统提示词和工具定义前置）？
  2. 是否设置了每会话/每用户的 Token 预算上限？
  3. 是否对简单任务使用了模型分层路由（弱模型处理简单问题）？
  4. 是否实现了语义缓存以复用相似查询的响应？
  5. 上下文管理策略是否有效控制了上下文膨胀？
  6. 是否配置了每日/每月成本告警？

四、用户体验 (UX) — 5 项

  1. Agent 的回复是否清晰、有帮助、没有无意义的冗长输出？
  2. 在长时间操作期间是否有适当的进度提示？
  3. 错误消息是否友好、可操作（而非纯技术错误码）？
  4. 是否支持流式输出以降低体感延迟？
  5. 是否明确告知用户正在与 AI Agent 交互？

五、合规性 (Compliance) — 5 项

  1. 用户数据存储和传输是否加密？
  2. 是否满足 GDPR / 个人信息保护法的要求？
  3. 是否设置了数据保留和自动清理策略？
  4. Agent 的输出是否有可能违反相关法律或平台政策？
  5. 是否提供了用户投诉和人工复核通道？
如何使用此清单

将此清单集成到你的 Pull Request 模板中。在每次提交 Agent 变更时，要求开发者逐项勾选。对于标记为"未完成"的项，必须在 PR 描述中说明原因和计划完成时间。安全类项目（第1-9项）不允许"跳过"——必须全部通过才可合并。

评估驱动的 Agent 开发 — Eval-Driven Development (EDD)
先定义评估集，再构建 Agent，用数据驱动迭代改进

传统软件开发有 TDD (Test-Driven Development)，Agent 开发应该有 EDD (Eval-Driven Development)——用评估驱动 Agent 的迭代改进。在 Agent 开发中，"先写测试"的含义是：先与领域专家和产品经理一起定义"什么样的 Agent 行为是好的"，将这些标准转化为可量化的评估指标和评估用例，然后再开始构建 Agent。

EDD 五步工作流

  1. 定义成功标准：与产品和领域专家一起，明确"好的 Agent 行为"是什么。将模糊的描述（"Agent 应该专业"）转化为可测量的指标（"回复中不应出现口语化表达；对不确定的答案应明确告知用户"）
  2. 构建评估集：收集多样化的测试用例——典型场景（80%）、边界情况（15%）、对抗性输入（5%）。每个用例包含输入和期望输出（或输出条件）。评估集应反映真实用户分布
  3. 构建最小可行 Agent (MVA)：基于最简单的提示词和最少工具构建第一个版本。目标不是完美，而是在评估集上建立一个量化的基线分数
  4. 迭代循环：评估 → 分析失败 → 改进 → 重新评估：每次迭代都针对具体的失败模式——工具选择错误？幻觉？回复不完整？每个失败模式都可能需要不同的修复策略
  5. 达到目标 → 冻结评估集 → 持续监控 → 定期更新：在评估集上达到满意表现后，将评估集成入 CI/CD 防止回归。每季度从生产数据中采样新用例更新评估集，防止过拟合
Python — Agent 评估运行器 (Eval Harness) 
import json
import asyncio
import time
from dataclasses import dataclass, field
from typing import List, Dict, Optional

@dataclass
class EvalCase:
    id: str
    category: str              # basic / edge-case / adversarial / real-user
    difficulty: str            # easy / medium / hard
    user_input: str            # 模拟用户输入
    expected_tools: Optional[List[str]] = None
    expected_keywords: Optional[List[str]] = None
    forbidden_keywords: Optional[List[str]] = None  # 幻觉检测
    max_steps: int = 10

@dataclass
class EvalResult:
    case_id: str
    passed: bool
    actual_output: str
    tool_calls_made: List[str]
    steps_taken: int
    tokens_used: int
    latency_ms: float
    failure_reason: Optional[str] = None

class EvalHarness:
    """评估运行器 — 在评估集上运行 Agent 并生成量化报告"""

    def __init__(self, agent_factory, judge_llm=None):
        self.agent_factory = agent_factory
        self.judge_llm = judge_llm
        self.results: List[EvalResult] = []

    async def run_evals(self, eval_cases: List[EvalCase],
                         parallel: int = 4) -> List[EvalResult]:
        print(f"开始运行 {len(eval_cases)} 个评估用例 (并行度: {parallel})")
        start_time = time.time()
        for i in range(0, len(eval_cases), parallel):
            batch = eval_cases[i:i + parallel]
            tasks = [self._run_single_eval(case) for case in batch]
            batch_results = await asyncio.gather(*tasks, return_exceptions=True)
            for case, result in zip(batch, batch_results):
                if isinstance(result, Exception):
                    result = EvalResult(case_id=case.id, passed=False, actual_output="", tool_calls_made=[], steps_taken=0, tokens_used=0, latency_ms=0, failure_reason=f"异常: {str(result)}")
                self.results.append(result)
            print(f"  进度: {min(i+parallel, len(eval_cases))}/{len(eval_cases)}")
        print(f"评估完成，总耗时 {time.time()-start_time:.1f}s")
        return self.results

    async def _run_single_eval(self, case: EvalCase) -> EvalResult:
        agent = self.agent_factory()
        eval_start = time.time()
        try:
            result = await agent.run(task=case.user_input, session_id=f"eval-{case.id}", max_steps=case.max_steps)
            passed, failure = self._evaluate(case, result.output, result.tool_calls, result.steps)
        except Exception as e:
            passed, failure, result = False, f"执行失败: {e}", None
        latency = (time.time() - eval_start) * 1000
        return EvalResult(case_id=case.id, passed=passed, actual_output=result.output if result else "", tool_calls_made=result.tool_calls if result else [], steps_taken=result.steps if result else 0, tokens_used=result.total_tokens if result else 0, latency_ms=latency, failure_reason=failure)

    def _evaluate(self, case: EvalCase, output: str, tools: List[str], steps: int) -> tuple:
        failures = []
        if case.expected_tools:
            missing = set(case.expected_tools) - set(tools)
            if missing: failures.append(f"缺少工具调用: {missing}")
        if case.expected_keywords:
            for kw in case.expected_keywords:
                if kw.lower() not in output.lower():
                    failures.append(f"缺少关键词: '{kw}'")
        if case.forbidden_keywords:
            for kw in case.forbidden_keywords:
                if kw.lower() in output.lower():
                    failures.append(f"出现禁止关键词: '{kw}' (幻觉)")
        if steps > case.max_steps:
            failures.append(f"步数超限: {steps} > {case.max_steps}")
        return len(failures) == 0, "; ".join(failures) if failures else None

    def get_report(self) -> Dict:
        total = len(self.results)
        passed = sum(1 for r in self.results if r.passed)
        avg_latency = sum(r.latency_ms for r in self.results) / total if total > 0 else 0
        failures = [{"case_id": r.case_id, "reason": r.failure_reason} for r in self.results if not r.passed]
        return {"total_cases": total, "passed": passed, "failed": total - passed, "pass_rate": f"{passed/total*100:.1f}%" if total > 0 else "N/A", "avg_latency_ms": f"{avg_latency:.0f}", "failures": failures}
好的评估集特征

多样性：覆盖不同类型用户、不同复杂度任务、不同表达方式。有难度梯度：从 easy 到 hard，确保 Agent 在各个级别都有合理表现。包含对抗性示例：故意构造容易让 Agent 出错的输入。反映真实分布：从生产日志中采样，确保评估集代表实际场景。定期更新：评估集会"过拟合"——每季度从生产数据中补充新用例。

知识共享与文档 — 让 Agent 知识在团队中流动
ADR、工具 API 文档、提示词 rationale、故障模式与 runbook 模板

Agent 系统的"隐性知识"远比传统软件多——为什么选择这个提示词结构？为什么这个工具被设计为只读？为什么这个场景下 Agent 会表现出特定行为？这些决策背后的推理如果不被记录下来，当团队成员变动时，知识就会流失。

必备文档类型

  • 架构决策记录 (ADR)：记录重要的架构决策及其背景。例如："选择不使用 LangChain 而使用原生 Anthropic SDK——原因：LangChain 的抽象层导致工具调用调试困难，且我们的场景不需要 LangChain 的复杂编排功能"
  • 工具 API 文档：每个工具的作用、参数说明、返回值格式、权限要求、速率限制、典型使用场景、已知的限制和边界条件
  • 提示词 Rationale：提示词中每个部分的设计意图。例如："在系统提示词的第3段中明确告知 Agent 不要编造数据——因为我们在客服场景中观察到 Agent 有 12% 的概率在找不到数据时编造答案"
  • 故障模式 (Failure Mode) 目录：记录已知的 Agent 失败模式、触发条件、影响范围和缓解措施。这是一个"活文档"，每次线上事故后更新
  • Runbook (操作手册)：常见运维操作的步骤说明——如何紧急关闭某个工具、如何回滚提示词、如何切换到备用模型、如何从备份恢复对话历史

文档共享机制

文档不应只存在于某人的本地文件夹中。推荐做法：(1) 将所有文档纳入 Git 仓库的 docs/ 目录，与代码一起版本控制；(2) 每周进行一次"Agent 行为回顾会议"——团队一起查看本周的 Agent 异常行为日志，讨论并记录新的故障模式；(3) 使用内部 Wiki 或 Notion 维护一个可搜索的知识库，方便新成员快速上手。

文档的关键原则

好的文档不是堆砌文字——它应该回答"为什么"而不仅仅是"是什么"。"这个工具的 timeout 是 30 秒"是信息，"这个工具的 timeout 是 30 秒，因为下游 API 的 P95 响应时间是 15 秒，30 秒提供了 2 倍的安全余量"是知识。后者让读者理解决策逻辑，从而在条件变化时能做出正确的调整。

Runbook 模板示例

一个完备的 Agent Runbook 应包含以下结构化条目：(1) 操作名称与触发条件；(2) 影响范围——此操作会影响哪些用户或哪些 Agent 功能；(3) 前置检查——执行前需要确认的条件；(4) 操作步骤——具体的 CLI 命令或 Dashboard 操作；(5) 回滚步骤——如果操作导致问题如何恢复；(6) 预期结果——操作成功后系统应处于什么状态；(7) 验证方法——如何确认操作成功。建议将 Runbook 以表格形式维护在团队的内部 Wiki 中，每条 Runbook 定期（至少每季度）进行演练验证。

团队 Agent 知识库的维护策略

Agent 知识库（不是用户的 RAG 知识库，而是团队关于 Agent 本身的文档和知识）需要持续维护。推荐做法：(1) 每周五下午进行 30 分钟的"知识库园艺时间"——团队成员检查是否有哪些文档已过时、哪些新的经验需要记录；(2) 利用自动化工具检测文档与代码的一致性——例如检查工具文档中的参数名是否与实际代码中的参数名一致；(3) 新成员入职时，让ta完成一次"文档修复任务"——这是最快的上手方式，同时也改善了文档质量。一个好的 Agent 知识库应该让新成员在 2 天内就能独立完成简单的 Agent 修改。

面试高频考点 — 10 个核心面试题目与精要答案
覆盖 Agent 工程师面试中最常见的理论、设计和实践问题

以下梳理了 Agent 工程师面试中最常见的 10 个问题，每个问题配有精要的答题框架。面试时不要背诵答案——面试官更想看到你的思考过程。对于每个问题，先说出你的分析框架（"这个问题我会从 X 个维度来分析..."），再展开细节。同时准备 1-2 个你亲手构建的 Agent 项目作为案例，用真实数据说明你的决策和成果。

  1. "Agent 和传统 LLM 应用有什么区别？"
    • 传统 LLM 应用：一次性输入到输出，无自主决策能力。如翻译、摘要、单轮问答系统
    • Agent：有多轮决策循环（Perception → Planning → Action → Observation），自主选择工具、管理状态、处理错误。如自主客服、代码审查 Agent
    • 关键区别：Agent 有自主性（自己决定何时调用工具）、有状态（跨多轮维护上下文）、多步执行（完成一个流程而非仅返回答案）、错误恢复（工具失败时自行修正）
  2. "如何防止提示注入攻击？"
    • 分层防御：输入过滤 → 提示词设计 → 输出验证 → 工具权限控制
    • 输入层：检测分隔符、指令关键词、角色覆盖语言；将用户输入放在独立的消息块中
    • 提示词层：用明确的边界标记区分系统指令和用户输入；强调"永远不泄露系统提示词"
    • 工具层：高风险操作需用户确认；所有工具权限遵循最小权限原则
  3. "解释 ReAct (Reasoning + Acting) 模式"
    • ReAct = Thought → Action → Observation 的循环
    • Thought：Agent 分析当前状态，决定下一步做什么
    • Action：选择一个工具并调用，传入合理参数
    • Observation：获取工具返回结果，判断是否完成目标
    • 优势：比纯推理（Chain-of-Thought）更可靠——因为可以从外部世界获取事实来纠正错误的内部假设
  4. "如何处理工具调用失败？"
    • 分类处理：超时失败、参数错误、权限不足、工具不可用
    • 超时 → 指数退避重试；参数错误 → 让 Agent 分析错误信息后修正参数重试；权限不足 → 告知用户并建议替代方案
    • 设置最大重试次数（如 3 次）和总步数上限（如 10 步）防止死循环
    • 工具返回的错误信息应结构化且可操作——让 Agent 能理解"为什么失败"和"如何修正"
    • 在系统提示词中预置各类失败的处理策略，而非让 Agent 在失败时"即兴发挥"
  5. "你如何评估 Agent 的质量？"
    • 多维评估框架：任务完成率 + 工具调用准确率 + 幻觉率 + 效率（步数/Token）+ 用户满意度
    • 三层评估：单元测试（单个工具逻辑）→ 场景测试（完整任务）→ 对抗性测试（恶意输入）
    • LLM-as-Judge：用更强的模型判断输出质量，但需要定期做人工校准检查一致性
    • 最重要的原则：评估集需要持续从生产日志中更新，避免过拟合到固定的测试用例
  6. "LangChain vs 原生 SDK，你如何选择？"
    • LangChain 优势：快速原型、丰富的第三方集成、社区活跃、RAG 工具链成熟
    • LangChain 劣势：抽象层过多导致调试困难、版本更新频繁且不兼容、性能开销大、自定义行为不够灵活
    • 原生 SDK 优势：完全控制、轻量化、调试简单、性能最优、适合生产环境
    • 推荐策略：原型阶段用 LangChain 快速验证想法（1-2 周）；生产阶段用原生 SDK 重写以获得更好的控制、性能和可维护性
  7. "如何管理 Agent 的记忆？"
    • 短期记忆（会话内）：对话历史。使用滑动窗口 + 自动摘要压缩管理上下文长度。超过 N 轮时，前面的对话自动压缩为摘要
    • 长期记忆（跨会话）：用户偏好、历史行为模式、关键事实。存储在外部数据库（PostgreSQL + 向量数据库），通过语义检索在需要时加载
    • 工作记忆（当前任务）：中间推理结果和工具返回值。在 Agent 循环的各步之间传递，不做持久化
    • 核心权衡：保留更多上下文提供更好体验但增加 Token 成本和延迟。需要自动化的记忆管理策略而非硬性的"保留最近 K 轮"
  8. "Agent 可观测性的关键指标有哪些？"
    • 业务指标：任务完成率、用户满意度 (CSAT)、自动化解决率
    • 性能指标：P50/P95/P99 延迟、TTFT (Time To First Token)、每任务平均步数
    • 成本指标：每请求 Token 消耗、缓存命中率、模型分布（Haiku vs Sonnet 使用比例）
    • 质量指标：工具调用成功率、重试率、幻觉率、用户纠错率
    • 安全指标：PII 泄露事件数、提示注入尝试数、异常调用模式
  9. "设计一个多 Agent 客服系统"
    • 架构：Triage Agent (路由) → Specialist Agent (FAQ/订单/技术/退款) → Escalation Agent → 人工
    • Triage Agent 分析用户意图并路由。传递结构化的上下文对象（分类 + 置信度 + 关键信息提取）而非原始对话文本
    • 每个 Specialist 有独立的工具集、知识库和提示词。Agent 之间不直接通信，通过共享的上下文对象交换信息
    • 设置明确的升级条件：置信度低于阈值、涉及敏感操作、三次尝试后仍未解决
  10. "如何优化 Agent 的成本？"
    • 第一优先：Prompt Caching——零代码改动，立即节省 60-90% 输入成本
    • 第二优先：模型分层路由——简单任务用 Haiku/Mini（更便宜），复杂任务用 Sonnet/Opus
    • 第三优先：并行工具调用——识别独立工具调用并同时执行，减少步数 = 减少 LLM 调用
    • 第四优先：上下文窗口管理——智能摘要 + 滑动窗口裁剪，控制上下文膨胀
    • 第五优先：语义缓存——对高频相似查询跳过 LLM 调用，缓存命中时零成本
    • 终极策略：批处理 API 用于离线任务，节省 50%

3 个项目创意 + 如何有效展示你的 Agent 开发能力

项目 1：开源代码审查 Agent

  • 构建什么：一个自动审查 GitHub Pull Request 的 Agent，提供代码质量、安全性、性能方面的建议
  • 技术栈：GitHub API + Claude API + Prompt Caching + 自定义代码分析工具（AST 解析、diff 分析）
  • 展示能力：工具集成、提示词工程（代码审查需要高度结构化的输出格式）、异步处理、结果格式化
  • 出彩点：添加"与人类审查者意见的对比"仪表盘，用数据证明 Agent 的审查质量。开源并获得 GitHub Stars

项目 2：个人知识管理 Agent (RAG Knowledge Agent)

  • 构建什么：一个带 RAG 的个人知识管理 Agent，能搜索、整理、关联你的笔记、书签和文档
  • 技术栈：向量数据库（ChromaDB/Pinecone）+ Claude API + Embedding API + 本地文件系统工具 + 网页抓取工具
  • 展示能力：RAG 管道设计、记忆管理、多模态输入处理、用户个性化
  • 出彩点：展示"跨文档关联"能力——Agent 自动发现笔记 A 和存档 B 讨论同一主题但从未被关联。用可视化的知识图谱展示关联结果

项目 3：多 Agent 自动化研究系统

  • 构建什么：多个 Agent 协作完成一个研究任务——Search Agent 搜索、Analysis Agent 分析、Writing Agent 撰写报告
  • 技术栈：自定义 Multi-Agent 编排框架 + 搜索 API + LLM 多模型路由 + 消息队列协调
  • 展示能力：多 Agent 编排、任务分解与分配、Agent 间通信协议、复杂工作流管理
  • 出彩点：与单 Agent 版本做详细的对比分析（完成质量、耗时、成本），用数据说明多 Agent 在准确性、完整性上的优势
展示作品的技巧

写一篇高质量的博客文章比只放代码库有效 10 倍。文章结构：问题定义 → 架构设计 → 关键挑战 → 解决方案 → 性能数据 → 经验教训。在 GitHub README 中加入可运行的 Demo 链接或视频演示。用具体数字说话："Agent 完成准确率 92%，比基线提升 15 个百分点"比"Agent 表现良好"有说服力得多。

行业趋势与未来机会 — 未来 3-5 年的职业定位

AI Agent 正在从"实验性技术"发展为"企业核心基础设施"。以下是未来 3-5 年最值得关注的趋势和机会：

五大关键趋势

  1. Agent-Native 公司的崛起：新一代创业公司的核心产品本身就是一个 Agent（而非在现有产品上添加 AI 功能）。这些公司的团队结构、商业模式、评估体系都以 Agent 为中心设计。例如：AI 客服公司用 Agent 替代 80% 的人工客服；AI 法律服务公司用 Agent 完成合同审查和尽职调查。机会：加入这样的公司作为早期工程师，积累 Agent-first 的工程经验
  2. Agent 市场与生态形成：类似 App Store 的 Agent 市场正在形成——专门化的 Agent 可以发布、交易和组合。MCP (Model Context Protocol) 正在成为连接 Agent 和工具的行业标准协议。出现新型中间件：Agent 开发平台、Agent 托管服务、Agent 评估服务。机会：成为 MCP Server 或 Agent 工具的开发者，或构建 Agent 编排平台
  3. "Agent 工程"作为独立学科：就像"前端工程"从"软件工程"中分化出来一样，"Agent 工程"正在成为一个独立的工程学科。有独特的技能要求：提示词设计、模型行为理解、工具编排、评估设计——这些不是传统软件工程的核心技能。大学开始开设专门的 Agent 开发课程，企业开始设立"Agent 工程师"专职岗位。机会：成为第一代"Agent 工程师"的先行者，帮助定义这个职业的标准和最佳实践
  4. 监管与合规成为刚需：随着 Agent 在金融、医疗、法律等受监管行业的应用增多，合规能力成为核心竞争壁垒。Agent 审计（记录和审查决策过程）、Agent 红队测试、Agent 偏见检测与公平性评估将成为标配。机会：专注 Agent 安全与合规的工程师将非常稀缺和高价值
  5. Agent-to-Agent 协议标准化：随着多 Agent 系统的普及，Agent 之间的通信协议将走向标准化。类似 HTTP 之于 Web 服务、gRPC 之于微服务——未来可能出现专门的 Agent 间通信协议（如 Google 的 A2A 协议），定义 Agent 之间如何发现彼此、协商能力、委托任务和交换结果

如何为未来定位自己

  • 深耕垂直领域 + Agent 技术：如"金融 + Agent"、"医疗 + Agent"、"法律 + Agent"。纯粹的 Agent 通才会越来越多，但在某个垂直领域有深厚积累的 Agent 工程师始终稀缺
  • 成为"Agent 全栈工程师"：不仅会写提示词，还理解部署、评估、安全、产品设计和成本优化。能从商业问题出发端到端地交付 Agent 解决方案
  • 建立技术品牌：写技术博客、在 GitHub 上维护开源 Agent 项目、在社区分享经验。在这个快速发展的领域，你的公共技术足迹是最有力的简历
  • 持续追踪前沿：arXiv 订阅 AI Agent 相关论文、关注 Anthropic/OpenAI/Google DeepMind 的技术博客、参与 Agent 相关的 Discord 和线上 meetup
