构建第一个生产Agent

从零开始，用 Python 完整构建一个可运行的 AI Agent，涵盖环境搭建、工具定义、核心循环、流式输出和 CLI 部署

环境准备 — 5分钟快速搭建开发环境

在开始编写 Agent 代码之前，我们需要准备好开发环境。以下是完整的、可复现的环境搭建步骤：

第一步：安装 Python 3.10+

Agent 开发需要 Python 3.10 或更高版本（推荐 3.11+），因为我们需要使用 TypeAlias、match-case 等新特性，以及更好的类型提示支持。

Bash — 检查并安装 Python 
# 检查当前 Python 版本
python --version  # 应输出 Python 3.10+

# 如果版本过低，使用 pyenv 安装
# macOS
brew install python@3.11

# Ubuntu/Debian
sudo add-apt-repository ppa:deadsnakes/ppa -y
sudo apt update
sudo apt install python3.11 python3.11-venv python3.11-dev -y

# Windows: 从 python.org 下载安装包
# 安装时务必勾选 "Add Python to PATH"

第二步：创建虚拟环境

永远在虚拟环境中开发——这不仅是 Python 最佳实践，对于 Agent 项目来说尤为重要，因为依赖关系复杂且版本敏感。

Bash — 创建并激活虚拟环境 
# 创建项目目录
mkdir my-first-agent && cd my-first-agent

# 创建虚拟环境
python -m venv .venv

# 激活虚拟环境
# Linux/macOS:
source .venv/bin/activate
# Windows (PowerShell):
.venv\Scripts\Activate.ps1
# Windows (CMD):
.venv\Scripts\activate.bat

# 确认虚拟环境已激活（路径前应显示 (.venv)）
which python  # 应指向 .venv/bin/python

第三步：安装依赖包

Bash — 安装核心依赖 
# 安装 Anthropic SDK（支持 Claude 系列模型）
pip install anthropic

# 安装 OpenAI SDK（支持 GPT 系列模型）

# 安装辅助工具
pip install python-dotenv  # 管理环境变量
pip install rich            # 彩色终端输出

# 生成 requirements.txt
pip freeze > requirements.txt

第四步：配置 API 密钥

绝对不要将 API Key 硬编码在代码中。使用 .env 文件存储敏感信息：

项目结构设计 — 可维护的 Agent 代码组织

一个单文件的 Agent 在原型阶段可行，但进入生产之前必须合理组织代码。以下是推荐的项目结构：

目录树 
my-first-agent/

├── README.md                 # 项目说明
│
├── agent/                    # 核心Agent模块
│   ├── __init__.py
│   ├── agent.py              # Agent主类：核心循环逻辑
│   ├── tools.py              # 工具定义：所有工具的JSON Schema和实现
│   ├── prompts.py            # 系统提示词：各场景的提示词模板
│   └── config.py             # 配置：从.env加载配置项
│
├── cli.py                    # 命令行入口
├── tests/                    # 测试目录
│   ├── __init__.py
│   ├── test_agent.py         # Agent核心逻辑测试
│   └── test_tools.py         # 工具函数单元测试
│
└── logs/                     # 日志目录（运行时不提交）
    └── agent.log

各模块的职责划分：

模块 职责 关键原则
agent.py Agent 主循环：接收用户消息 → 调用LLM → 执行工具 → 返回结果 不包含工具实现细节；不包含特定领域的业务逻辑
tools.py 所有工具函数的实现 + JSON Schema 定义 每个工具是纯函数（输入→输出）；不依赖 Agent 状态
prompts.py 系统提示词模板、few-shot 示例、角色定义 提示词与代码分离；支持模板变量替换
config.py 统一管理所有配置项（API Key、模型名、超时时间等） 提供默认值；支持环境变量覆盖；类型安全
cli.py 命令行接口：参数解析、交互模式、单次查询模式 轻量入口；不包含业务逻辑
为什么这样划分？

这种模块划分遵循单一职责原则 (SRP)——每个模块只有一个明确的职责。当你的工具从3个增长到30个，或需要支持不同模型的 API 时，这种结构让修改只影响局部，不会牵一发而动全身。

定义Agent的工具 — 从函数到JSON Schema

工具是 Agent 的"双手"——让 Agent 能够执行实际操作而不仅仅是生成文本。我们将定义三个实用的工具：

工具1: get_current_time() — 获取当前时间

看似简单，但这是 Agent 最常需要的基础能力——没有它，Agent 无法回答"现在是几点"这样的问题，因为 LLM 的训练数据有时间截止。

工具2: calculate(expression) — 安全地执行数学计算

LLM 擅长推理但不擅长精确计算。给 Agent 一个计算器，它会大幅提升处理数值问题的准确性。

工具3: search_web(query) — 搜索最新信息

让 Agent 能够获取训练数据之外的最新信息，突破知识截止日期的限制。

以下是完整的工具实现代码：

每个工具包含两部分:
1. JSON Schema 定义 — 供 LLM 理解工具的用途和参数
2. Python 函数实现 — 实际执行的逻辑

关键原则:
- 函数签名必须与 Schema 中定义的参数一致
- 每个工具返回字符串或可序列化的数据结构
- 工具必须是幂等的（多次调用结果一致，无副作用）
- 工具内部必须做好异常处理，不要让异常传播到 Agent 循环
"""

import math
from datetime import datetime, timezone
from typing import Any, Dict

# 3. 给出参数的约束条件（required、enum、minimum/maximum）

def get_current_time() -> str:
    """获取当前日期和时间"""
    try:
        now = datetime.now(timezone.utc)
        beijing_now = now.replace(tzinfo=timezone.utc).astimezone(
            timezone(datetime.strptime("+08:00", "%z").utcoffset(None))
        )
        # 使用更简单的方式获取北京时间
        from datetime import timedelta
        beijing_tz = timezone(timedelta(hours=8))
        beijing_now = now.astimezone(beijing_tz)

        return json.dumps({
            "utc_time": now.strftime("%Y-%m-%d %H:%M:%S UTC"),
            "beijing_time": beijing_now.strftime("%Y-%m-%d %H:%M:%S CST"),
            "weekday": beijing_now.strftime("%A"),
            "timestamp": now.timestamp()
        }, ensure_ascii=False)
    except Exception as e:
        return json.dumps({"error": f"获取时间失败: {e}"}, ensure_ascii=False)


def calculate(expression: str) -> str:
    """安全地计算数学表达式

    安全性说明:
    - 使用白名单机制，只允许数学相关的函数和常量
    - 不能访问内置函数、模块导入、文件系统
    - 表达式超过1000字符时拒绝执行（防止注入攻击）
    """
    try:
        # 长度限制
        if len(expression) > 1000:
            return json.dumps({"error": "表达式过长，拒绝执行"}, ensure_ascii=False)

        # 构建安全的执行环境（白名单）
        safe_names = {
            "abs": abs, "round": round,
            "min": min, "max": max,
            "sum": sum,
            "sqrt": math.sqrt, "pow": math.pow,
            "sin": math.sin, "cos": math.cos,
            "tan": math.tan, "log": math.log,
            "log10": math.log10, "log2": math.log2,
            "exp": math.exp,
            "pi": math.pi, "e": math.e,
            "degrees": math.degrees, "radians": math.radians,
            "ceil": math.ceil, "floor": math.floor,
        }

        # 使用 eval 但限制命名空间
        result = eval(expression, {"__builtins__": {}}, safe_names)

        return json.dumps({
            "expression": expression,
            "result": result
        }, ensure_ascii=False)
    except SyntaxError as e:
        return json.dumps({"error": f"表达式语法错误: {e}"}, ensure_ascii=False)
    except ZeroDivisionError:
        return json.dumps({"error": "除数不能为零"}, ensure_ascii=False)
    except Exception as e:
        return json.dumps({"error": f"计算失败: {e}"}, ensure_ascii=False)


def search_web(query: str) -> str:
    """搜索互联网获取信息

    在生产环境中，这里应该调用真实的搜索API（如 Tavily、Brave Search、
    Google Custom Search 等）。为保持演示代码的可运行性，这里返回模拟结果。

    实际集成搜索API的代码示例:
    ```
    import requests
    response = requests.post(
        "https://api.tavily.com/search",
        json={"api_key": TAVILY_API_KEY, "query": query, "search_depth": "basic"}
    )
    results = response.json()["results"]
    ```
    """
    try:
        if not query.strip():
            return json.dumps({"error": "搜索关键词不能为空"}, ensure_ascii=False)

        # 模拟搜索 — 生产环境替换为真实API调用
        return json.dumps({
            "query": query,
            "results": [
                {
                    "title": f"关于'{query}'的搜索结果",
                    "snippet": f"这是一个模拟的搜索结果，与'{query}'相关。在生产环境中，这里将返回来自搜索API的真实数据。",
                    "url": f"https://example.com/search?q={query}"
                }
            ],
            "_note": "模拟数据 — 生产环境需集成真实搜索API"
        }, ensure_ascii=False)
    except Exception as e:
        return json.dumps({"error": f"搜索失败: {e}"}, ensure_ascii=False)

如何写出好的工具描述 (Tool Description)?

工具描述是 LLM 决定是否调用某个工具的唯一依据。好的描述应该：(1) 说明工具做什么——用一句清晰的话概括功能；(2) 说明何时使用——给出使用场景示例；(3) 说明参数含义——每个参数要解释其作用和格式；(4) 不要过度描述——描述越长越容易被模型忽略关键信息，保持50-150字最佳。

常见错误：工具 Schema 定义不合格

以下是一个糟糕的工具定义（不要这样做）：

{
  "name": "calc",
  "description": "计算一些东西",
  "parameters": {"type": "object", "properties": {"x": {"type": "string"}}}
}

问题：名称不直观、描述过于模糊、参数命名无意义、缺少 required 字段、缺少类型说明。这种定义会让 LLM 困惑——它不知道该在什么时候调用这个工具，以及应该传什么参数。

编写系统提示词 — Agent的"灵魂"

系统提示词定义了 Agent 的身份、能力和行为边界。一个优秀的系统提示词是 Agent 表现好坏的决定性因素之一。下面是我们的 Agent 的系统提示词及逐段解析：

Python — agent/prompts.py 系统提示词 
"""系统提示词定义"""

SYSTEM_PROMPT = """你是一个智能AI助手，名为 "Agent-13"。

## 你的角色
你是一个乐于助人、精确且诚实的助手。你的目标是尽最大努力理解用户的需求，
并提供准确、有用的帮助。你对能力范围内的问题给出详尽回答，对不确定的
问题明确告知你的局限。

## 核心行为准则
1. **准确性优先**：当进行数学计算时，必须使用 calculate 工具而非心算。
2. **时间感知**：当用户询问时间、日期时，使用 get_current_time 工具。
3. **信息时效**：当问题涉及最新资讯时，使用 search_web 工具获取信息。
4. **诚实透明**：当你不知道答案或不确定时，直接告知用户，不要编造。
5. **逐步推理**：对于复杂问题，先思考再行动。解释你的推理过程。
6. **安全边界**：拒绝执行任何恶意代码、生成有害内容或侵犯隐私的请求。

## 工具使用指南
你可以使用以下工具来扩展你的能力：
- `get_current_time`: 获取当前时间，无需参数
- `calculate`: 安全计算数学表达式，参数 expression
- `search_web`: 搜索网络信息，参数 query

**何时使用工具**：
- 需要精确计算 → calculate
- 需要当前时间 → get_current_time
- 需要最新信息 → search_web
- 所有你知识范围外的操作 → 对应的工具

**工具使用原则**：
- 一次只调用一个工具（除非多个工具之间无依赖关系）
- 工具返回结果后，基于结果生成最终回答
- 如果工具返回错误，尝试分析原因并给用户有用的建议

## 输出格式
- 使用清晰的结构（段落、列表、标题等）组织信息
- 代码用 Markdown 代码块包裹，并标注语言
- 当引用工具返回的数据时，注明数据来源
- 保持友好但不啰嗦的语气

## 示例交互

用户: 现在是几点？
助手: [调用 get_current_time 工具]
      [基于工具返回] 现在是北京时间 2026年5月10日 14:30:00，星期日。

用户: 计算 (456 + 789) * 23 / 2
助手: [调用 calculate 工具，传入表达式 "(456 + 789) * 23 / 2"]
      [基于工具返回] 计算结果为 14317.5。计算过程：(456+789)=1245,
      1245×23=28635, 28635÷2=14317.5

用户: 你好！请介绍下你自己。
助手: 你好！我是 Agent-13，一个智能AI助手。我可以帮你进行计算、
      查询时间、搜索信息等任务。有什么我可以帮你的吗？
"""

系统提示词的关键组成部分分析

组成部分 作用 设计要点
角色定义 设定Agent的"人格"和定位 简洁明确，不要太长；明确Agent的职责范围
行为准则 规定Agent的基本原则 用数字编号，每条一条规则；正面描述（做什么）优于负面描述（不做什么）
工具使用指南 教会Agent何时和如何使用工具 给出具体的触发条件；明确"何时使用" vs "何时不用"
输出格式 控制Agent的输出风格 具体（"用Markdown代码块"）优于抽象（"输出美观"）
示例交互 通过Few-shot引导Agent行为 覆盖主要的使用场景；示例要真实、自然
提示词迭代策略

不要期望一次写出完美的系统提示词。推荐的迭代流程：写出初版 → 测试10个典型用例 → 记录失败案例 → 分析失败原因（工具未调用？调用了错误的工具？回复格式不对？）→ 针对性修改提示词 → 再测试。通常3-5轮迭代后提示词趋于稳定。

实现Agent核心循环 — Think → Act → Observe

这是整个 Agent 的心脏——核心循环 (Core Loop)。它的设计直接决定了 Agent 的智能程度和稳定性。以下代码展示了最经典的 ReAct 循环（Reasoning + Acting）的完整实现：

Python — agent/agent.py 核心Agent类（约100行） 
"""Agent 核心循环模块

实现基于 Anthropic SDK 的 Agent 主循环。
循环模式: 接收用户输入 → 调用LLM → [工具调用?] → 执行工具
→ 将结果返回LLM → 生成最终回复 → 返回给用户
"""

import json
import logging
from typing import Any, Callable, Dict, List, Optional

from anthropic import Anthropic
from anthropic.types import Message, MessageParam, ToolUseBlock

from .config import settings
from .prompts import SYSTEM_PROMPT
from .tools import TOOL_EXECUTORS, TOOL_SCHEMAS

logger = logging.getLogger("agent")


class AgentError(Exception):
    """Agent运行时的自定义异常"""
    pass


class Agent:
    """智能Agent主类

    职责:
    - 管理与LLM的交互
    - 处理工具调用的完整生命周期
    - 维护对话历史和消息管理
    - 实现安全防护（步数限制、令牌限制）

    使用示例:
        agent = Agent()
        response = agent.run("帮我计算 123 * 456")
        print(response)

        # 使用流式输出
        async for chunk in agent.run_stream("现在几点了"):
            print(chunk, end="", flush=True)
    """

    def __init__(
        self,
        model: Optional[str] = None,
        max_steps: int = 10,
        verbose: bool = False,
    ):
        """初始化Agent

        Args:
            model: 使用的模型名称，默认从config读取
            max_steps: 最大工具调用步数，防止无限循环
            verbose: 是否输出详细日志（调试时开启）
        """
        self.client = Anthropic(api_key=settings.ANTHROPIC_API_KEY)
        self.model = model or settings.DEFAULT_MODEL
        self.max_steps = max_steps
        self.verbose = verbose
        self.system_prompt = SYSTEM_PROMPT

        # 对话历史
        self.messages: List[MessageParam] = []

        # 统计信息
        self.total_tool_calls = 0
        self.total_tokens_used = 0

        if self.verbose:
            logger.info(f"Agent 初始化完成 | 模型: {self.model} | 最大步数: {max_steps}")

    def run(self, user_input: str, reset_history: bool = False) -> str:
        """执行Agent的主循环

        Args:
            user_input: 用户的消息
            reset_history: 是否重置对话历史（新会话时设为True）

        Returns:
            Agent的文本回复

        Raises:
            AgentError: 当循环超过最大步数或其他异常时
        """
        if reset_history:
            self.messages = []

        # 添加用户消息到历史
        self.messages.append({
            "role": "user",
            "content": user_input
        })

        for step in range(self.max_steps):
            if self.verbose:
                logger.info(f"--- 步骤 {step + 1}/{self.max_steps} ---")

            # 调用LLM — 这是"Think"阶段
            response = self.client.messages.create(
                model=self.model,
                max_tokens=4096,
                system=self.system_prompt,
                tools=TOOL_SCHEMAS,
                messages=self.messages
            )

            # 更新令牌统计
            self.total_tokens_used += response.usage.input_tokens + response.usage.output_tokens

            # 检查LLM是否要调用工具
            tool_use_blocks = [
                block for block in response.content
                if block.type == "tool_use"
            ]

            if not tool_use_blocks:
                # 没有工具调用 -> LLM已经生成了最终答案
                text_blocks = [
                    block.text for block in response.content
                    if block.type == "text"
                ]
                final_answer = "".join(text_blocks)

                # 将助手消息加入历史
                self.messages.append({
                    "role": "assistant",
                    "content": final_answer
                })

                if self.verbose:
                    logger.info(f"Agent完成 | 步骤: {step+1} | 令牌: {self.total_tokens_used}")

                return final_answer

            # 处理工具调用 — 这是"Act"和"Observe"阶段
            tool_results = self._execute_tools(tool_use_blocks)

            # 将工具调用和结果加入对话历史
            assistant_content = []
            for block in response.content:
                if block.type == "text":
                    assistant_content.append({"type": "text", "text": block.text})
                elif block.type == "tool_use":
                    assistant_content.append({
                        "type": "tool_use",
                        "id": block.id,
                        "name": block.name,
                        "input": block.input
                    })

            self.messages.append({
                "role": "assistant",
                "content": assistant_content
            })

            # 添加工具执行结果
            self.messages.append({
                "role": "user",
                "content": tool_results
            })

        # 超过最大步数
        raise AgentError(
            f"Agent在{self.max_steps}步内未能完成任务。"
            f"考虑增加max_steps或检查工具是否有循环调用。"
        )

    def _execute_tools(self, tool_use_blocks: List[ToolUseBlock]) -> List[Dict]:
        """执行LLM请求的工具调用"""
        results = []

        for block in tool_use_blocks:
            tool_name = block.name
            tool_input = block.input
            self.total_tool_calls += 1

            if self.verbose:
                logger.info(f"调用工具: {tool_name}({json.dumps(tool_input, ensure_ascii=False)})")

            # 查找并执行工具函数
            executor = TOOL_EXECUTORS.get(tool_name)
            if executor is None:
                result = json.dumps({"error": f"未知工具: {tool_name}"}, ensure_ascii=False)
            else:
                try:
                    result = executor(**tool_input) if tool_input else executor()
                except Exception as e:
                    result = json.dumps({"error": f"工具执行异常: {e}"}, ensure_ascii=False)
                    logger.error(f"工具 {tool_name} 执行失败: {e}", exc_info=True)

            results.append({
                "type": "tool_result",
                "tool_use_id": block.id,
                "content": result
            })

        return results

    def clear_history(self):
        """清空对话历史，开始新会话"""
        self.messages = []
        self.total_tool_calls = 0

    def get_stats(self) -> Dict[str, Any]:
        """获取Agent的运行统计"""
        return {
            "model": self.model,
            "total_tool_calls": self.total_tool_calls,
            "total_tokens_used": self.total_tokens_used,
            "messages_in_history": len(self.messages),
            "max_steps": self.max_steps,
        }

核心循环的设计要点

  • max_steps 安全阀：防止 Agent 陷入无限循环（如工具返回异常导致 LLM 反复重试）。10步对于大多数场景足够，复杂任务可调整到20-30步。
  • 消息历史的正确维护：每次 LLM 调用后，必须将 assistant 消息（包含 tool_use blocks）和 tool_result 消息都加入历史。这是 Anthropic API 的规范要求，消息顺序错误会导致 API 报错。
  • 异常隔离：工具执行中的异常不会传播到 Agent 循环——捕获后作为错误结果返回给 LLM，让 LLM 自行判断如何向用户解释。
  • 统计信息：记录工具调用次数和令牌消耗，方便监控和成本分析。
常见陷阱：消息格式错误

Anthropic API 对消息格式有严格要求：tool_use 和 tool_result 必须成对出现，且 tool_result 后面必须跟 user 消息。如果消息序列格式不对，API会返回400错误。开发时开启 verbose=True 可以帮你排查这类问题。

添加流式输出 — 提升用户体验

用户等待 Agent 回复时，看着空白的屏幕是一种糟糕的体验。流式输出 (Streaming) 让用户实时看到 Agent 的思考过程：

Python — 流式输出的 Agent.run_stream() 方法 
"""添加到 Agent 类中的流式输出方法"""

实现流式输出的关键区别:
1. 使用 stream=True 参数
2. 使用 async for 迭代响应流
3. 需要处理 streaming 中的 tool_use 块（增量累积）
4. 将文本块实时 yield 给调用者
"""

import json
from typing import AsyncGenerator


async def run_stream(
    self, user_input: str, reset_history: bool = False
) -> AsyncGenerator[str, None]:
    """流式执行 Agent 循环

    与 run() 的逻辑相同，但通过 async generator 实时输出。
    调用方式:
        async for chunk in agent.run_stream("你好"):
            print(chunk, end="", flush=True)
    """
    if reset_history:
        self.messages = []

    self.messages.append({
        "role": "user", "content": user_input
    })

    for step in range(self.max_steps):
        # 流式调用LLM
        async with self.client.messages.stream(
            model=self.model,
            max_tokens=4096,
            system=self.system_prompt,
            tools=TOOL_SCHEMAS,
            messages=self.messages
        ) as stream:
            # 累积器：用于处理流式中的 tool_use 块
            current_text = ""
            tool_use_accumulator = {}  # {block_index: {"name":..., "input":..., "id":...}}

            async for event in stream:
                if event.type == "content_block_delta":
                    delta = event.delta

                    if delta.type == "text_delta":
                        # 文本内容：实时 yield
                        text = delta.text
                        current_text += text
                        yield text

                    elif delta.type == "input_json_delta":
                        # 工具调用的JSON参数：累积而非yield
                        idx = event.index
                        if idx not in tool_use_accumulator:
                            tool_use_accumulator[idx] = {
                                "name": "",
                                "input_json": "",
                                "id": ""
                            }
                        tool_use_accumulator[idx]["input_json"] += delta.partial_json

                elif event.type == "content_block_start":
                    block = event.content_block
                    if block.type == "tool_use":
                        idx = event.index
                        tool_use_accumulator[idx] = {
                            "name": block.name,
                            "input_json": "",
                            "id": block.id
                        }

            # 流结束，获取最终的完整消息
            final_message = await stream.get_final_message()

        # 检查是否需要工具调用
        tool_use_blocks = [
            block for block in final_message.content
            if block.type == "tool_use"
        ]

        if not tool_use_blocks:
            # 没有工具调用，任务完成
            yield "\n"  # 结束标记

            # 记录助手消息到历史
            text_content = [
                block.text for block in final_message.content
                if block.type == "text"
            ]
            self.messages.append({
                "role": "assistant",
                "content": "".join(text_content)
            })
            return

        # 有工具调用 — 执行工具并继续循环
        yield f"\n\n[调用工具中...]\n"

        tool_results = self._execute_tools(tool_use_blocks)

        # 更新消息历史
        assistant_content = []
        for block in final_message.content:
            if block.type == "text":
                assistant_content.append({"type": "text", "text": block.text})
            elif block.type == "tool_use":
                assistant_content.append({
                    "type": "tool_use",
                    "id": block.id,
                    "name": block.name,
                    "input": block.input
                })

        self.messages.append({
            "role": "assistant",
            "content": assistant_content
        })
        self.messages.append({
            "role": "user",
            "content": tool_results
        })

    yield f"\n[达到最大步数限制 ({self.max_steps})]\n"
流式输出 vs 非流式输出

非流式 (run)：适合 CLI 单次查询、批处理、API 服务后端——简单直接。流式 (run_stream)：适合聊天界面、REPL 交互——用户体验更好但实现更复杂。生产环境中建议两种模式都支持，让调用者选择。

部署为命令行工具 — 打造专业CLI

将 Agent 封装为命令行工具，支持交互式 REPL 和单次查询两种模式：

Python — cli.py 命令行入口 
"""Agent 命令行工具

用法:
  python cli.py                          # 进入交互式REPL模式
  python cli.py --query "计算 123*456"   # 单次查询模式
  python cli.py --stream "现在几点了"     # 流式输出模式
  python cli.py --model claude-opus-4-20250514 --query "你好"
  python cli.py --verbose                # 显示调试日志
"""

import argparse
import asyncio
import sys
from rich.console import Console
from rich.markdown import Markdown

from agent.agent import Agent
from agent.config import settings

console = Console()


def parse_args():
    parser = argparse.ArgumentParser(
        description="Agent-13: 智能AI助手命令行工具",
        formatter_class=argparse.RawDescriptionHelpFormatter,
        epilog="""
示例:
  %(prog)s                                      # 交互式模式
  %(prog)s -q "39度算发烧吗？"                  # 单次查询
  %(prog)s -s -q "2026年GDP排名前3的国家"       # 流式查询
  %(prog)s -m claude-opus-4-20250514 -v         # 指定模型+详细日志
        """
    )
    parser.add_argument("-q", "--query", type=str, help="单次查询内容（不进入REPL）")
    parser.add_argument("-s", "--stream", action="store_true", help="使用流式输出")
    parser.add_argument("-m", "--model", type=str, help="指定模型名称")
    parser.add_argument("-v", "--verbose", action="store_true", help="详细日志输出")
    parser.add_argument("--max-steps", type=int, default=settings.MAX_STEPS, help="最大工具调用步数")
    return parser.parse_args()


def main():
    """主入口"""
    args = parse_args()

    try:
        settings.validate()
    except ValueError as e:
        console.print(f"[red]❌ 配置错误: {e}[/red]")
        console.print("[yellow]提示: 确保 .env 文件存在并包含 ANTHROPIC_API_KEY[/yellow]")
        sys.exit(1)

    agent = Agent(
        model=args.model,
        max_steps=args.max_steps,
        verbose=args.verbose
    )

    if args.query:
        # 单次查询模式
        if args.stream:
            asyncio.run(_run_stream_query(agent, args.query))
        else:
            response = agent.run(args.query)
            console.print(Markdown(response))
    else:
        # 交互式REPL模式
        console.print("[bold cyan]🤖 Agent-13 交互模式[/bold cyan]")
        console.print("[dim]输入 help 查看帮助 | quit/exit 退出 | clear 清空对话[/dim]\n")

        while True:
            try:
                user_input = console.input("[bold green]你:[/bold green] ").strip()
            except (KeyboardInterrupt, EOFError):
                console.print("\n[yellow]再见！👋[/yellow]")
                break

            if not user_input:
                continue

            if user_input.lower() in ("quit", "exit", "q"):
                console.print("[yellow]再见！👋[/yellow]")
                break
            elif user_input.lower() == "clear":
                agent.clear_history()
                console.print("[dim]对话历史已清空[/dim]")
                continue
            elif user_input.lower() == "help":
                console.print("[dim]命令: quit/exit 退出 | clear 清空对话 | help 帮助[/dim]")
                continue
            elif user_input.lower() == "stats":
                import json
                console.print(json.dumps(agent.get_stats(), indent=2, ensure_ascii=False))
                continue

            # 处理查询
            console.print("[bold blue]Agent:[/bold blue] ", end="")
            if args.stream:
                asyncio.run(_run_stream_query(agent, user_input))
            else:
                response = agent.run(user_input)
                console.print(Markdown(response))
            console.print()  # 空行分隔


async def _run_stream_query(agent: Agent, query: str):
    """执行流式查询"""
    async for chunk in agent.run_stream(query):
        console.print(chunk, end="")
    console.print()


if __name__ == "__main__":
    main()
CLI 设计最佳实践

一个好的命令行工具应该：(1) 支持交互模式和单次查询模式两种场景；(2) 使用 argparse 提供标准化的参数解析和帮助文档；(3) 用 rich 库美化终端输出（Markdown、颜色、进度条）；(4) 优雅处理 Ctrl+C 和 EOF；(5) 提供快捷命令（help、clear、stats）提升使用体验。

常见问题排查 — 新手必读

以下是开发 Agent 过程中最常见的5类问题，了解它们可以帮你节省大量调试时间：

问题1: API Key 未找到或无效

症状：运行时报错 AuthenticationError: Incorrect API key provided 或 ValueError: ANTHROPIC_API_KEY 未设置

诊断：echo $ANTHROPIC_API_KEY 检查环境变量；确认 .env 文件在项目根目录

修复：确保 load_dotenv() 在模块顶层被调用；在 IDE 中运行时可能需要手动指定 .env 路径；重新生成 API Key 如果怀疑泄露

问题2: 速率限制 (Rate Limiting)

症状：RateLimitError: Too many requests，通常在连续快速调用时出现

诊断：检查 Anthropic Console 中的 Usage 页面，查看当前的 RPM (Requests Per Minute) 和 TPM (Tokens Per Minute) 限制

修复：(1) 在 Agent 循环中加入 time.sleep(1) 避免连续调用；(2) 实现指数退避重试机制；(3) 联系 Anthropic 提升限额

问题3: 上下文窗口溢出

症状：对话进行一段时间后，API 返回 400 context_length_exceeded 错误

诊断：检查 agent.messages 的长度。每次工具调用会让历史消息增长2条（assistant + user/tool_result），10次工具调用就有20条消息

修复：(1) 限制 max_steps 防止无限增长；(2) 实现消息历史的滑动窗口（保留最近N条）；(3) 对长对话进行定期摘要压缩；(4) 在工具返回结果时截断过长内容

问题4: 工具 Schema 格式错误

症状：API 返回 400 invalid tools 或 Agent 从不调用某个工具

诊断：检查 JSON Schema 是否符合 Anthropic API 规范——必须包含 type: "function" 外层包装和 function.name/description/parameters 三个字段

修复：(1) 用 JSON Schema 验证器检查格式；(2) 确保 parameters.properties 中的每个属性都有 type 字段；(3) required 数组中的字段必须在 properties 中定义；(4) 检查 description 字段是否存在且有意义

问题5: Python 导入路径错误

症状：ModuleNotFoundError: No module named 'agent'

诊断：确保项目根目录在 Python 路径中；确保 agent/__init__.py 文件存在（即使是空文件）

修复：(1) 在项目根目录运行 pip install -e . 以开发模式安装；(2) 或创建 setup.py/pyproject.toml 声明包结构；(3) 确保从项目根目录执行 Python 脚本

下一步学习路径 — 从入门到精通的路线图

你刚刚构建了人生中第一个完整的 AI Agent。但这只是开始。以下是从这里出发的学习路线：

  1. 丰富工具生态

  2. 尝试不同的模型
    • 对比 Claude Sonnet/Opus 在处理复杂工具调用时的差异

    • 了解何时使用强模型（复杂推理）vs 快模型（简单任务）的混合策略
  3. 添加记忆系统
    • 集成向量数据库（ChromaDB、Pinecone）实现长期记忆
    • 实现会话摘要——将长对话压缩为摘要保留关键信息
    • 添加用户偏好记忆——让 Agent 越来越"懂"使用者
  4. 部署为 Web 服务
    • 将 Agent 包装为 FastAPI/Flask 服务，提供 REST API
    • 添加 WebSocket 支持以实现实时流式通信
    • 使用 Docker 容器化部署，配合 Nginx 反向代理
  5. 探索多 Agent 系统
    • 学习第6章中的多 Agent 协作模式
    • 尝试 CrewAI 或 AutoGen 框架构建多 Agent 团队

  6. 深入企业集成
    • 学习下一章（第14章：企业集成模式）
    • 了解 Agent 如何安全地连接数据库、API、消息队列
    • 掌握企业级的安全性、监控和合规实践
