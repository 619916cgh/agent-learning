Agent 性能优化

深入掌握 Token 经济学、缓存策略、模型路由、并行执行与上下文优化，将 Agent 成本降低 80% 同时提升 3 倍速度

💰
Token 经济学基础 — 理解 LLM 的计价体系
从微观 Token 成本到宏观总拥有成本 (TCO)

Agent 系统的运营成本主要来源于 LLM API 调用。深入理解 Token 经济学是性能优化的第一步：

  • Token 的基本计价模型
    • 输入 Token (Input Token)：发送给模型的全部上下文，包括系统提示词、工具定义、对话历史、用户输入。通常价格较低，但在多轮 Agent 对话中占比很大
    • 输出 Token (Output Token)：模型生成的文本，包括推理过程、工具调用指令、最终回复。价格通常是输入 Token 的 3-5 倍
    • Cache Write Token：将内容写入缓存时产生的费用，通常比正常输入贵 25%
    • Cache Read Token：从缓存中读取的 Token，通常比正常输入便宜 90%
  • 单次请求成本估算公式
    • 成本 = input_tokens × 输入单价 + output_tokens × 输出单价 + cache_write_tokens × 写入单价 + cache_read_tokens × 读取单价
    • 对于典型的 Agent 请求 (~2000 input, ~500 output, ~1000 cache read)，使用 Claude Sonnet 的成本约 $0.01-0.03/次
    • 多步 Agent (5-10 轮循环) 的 Token 消耗远高于单次聊天，每步都需要携带工具定义和累积的上下文
  • 总拥有成本 (TCO) 的组成部分
    • LLM API 费用 (通常占 50-70%)
    • Embedding API 费用 (用于 RAG，占 5-15%)
    • 基础设施费用：服务器、数据库、Redis 缓存、CDN
    • 监控/日志/追踪工具费用
    • 人力成本：开发、运维、评估

Prompt 缓存策略 — 将重复 Token 成本降低 90%
Anthropic Prompt Caching 深度解析与语义缓存实战

Agent 系统有一个显著特征：系统提示词、工具定义和多轮对话中的上下文前缀在每次请求中高度重复。Prompt Caching 正是利用这一特征，将大量重复的输入 Token 成本降低 90%：

  • Anthropic Prompt Caching 工作原理
    • 在消息内容中设置 cache_control: {"type": "ephemeral"} 断点
    • 断点之前的全部内容会被缓存，后续请求如果前缀匹配则直接从缓存读取
    • 缓存有 5 分钟 TTL (Time-To-Live)，每次命中后自动续期 5 分钟
    • 缓存读取价格为正常输入价格的 10%，大幅节省成本
    • 最多可在一条消息中设置 4 个缓存断点
  • 缓存最佳实践
    • 将静态内容放在最前面：系统提示词、工具定义、角色设定等不随请求变化的内容优先放置，并在末尾加 cache_control 断点
    • 多断点策略：断点1在系统提示词后，断点2在工具定义后，断点3在对话历史摘要后，断点4在最近几条消息后
    • 保持前缀稳定：缓存的命中依赖前缀精确匹配，不要在缓存区域前插入任何动态内容
    • 对长文档使用缓存：如果 Agent 需要反复参考一份长文档（如产品手册），将其放在缓存区域
  • 语义缓存 (Semantic Caching)
    • 不同于 Prompt Caching 的精确前缀匹配，语义缓存通过向量相似度匹配来复用历史响应
    • 流程：新用户问题 → 计算 Embedding → 在向量数据库中搜索相似问题 → 如果相似度 > 0.95，直接返回缓存的响应 → 否则调用 LLM
    • 适合场景：FAQ 型客服、相似度高的模板化查询、热门问题的快速响应
    • 核心权衡：缓存命中率 vs 准确性（相似度阈值设太高，命中率低；设太低，可能返回不准确的答案）

Prompt 缓存与语义缓存 — 双缓存实现
Anthropic Prompt Caching + Embedding 语义缓存的生产级实现
Python — 双层缓存系统：Prompt Caching + Semantic Caching 
import hashlib
import json
import time
import numpy as np
from dataclasses import dataclass, field
from typing import Optional, List, Dict
from anthropic import Anthropic, CacheControlEphemeral

@dataclass
class CacheEntry:
    """语义缓存条目"""
    query_embedding: np.ndarray
    response: str
    tokens_saved: int
    created_at: float = field(default_factory=time.time)

class DualLayerCache:
    """
    双层缓存系统：
    Layer 1 — Anthropic Prompt Caching (精确前缀匹配，自动托管)
    Layer 2 — Semantic Caching (语义相似匹配，自管理)
    """
    def __init__(
        self,
        anthropic_client: Anthropic,
        embedding_model="text-embedding-3-small",
        similarity_threshold=0.95,
        semantic_cache_size=10000
    ):
        self.client = anthropic_client
        self.embedding_model = embedding_model
        self.similarity_threshold = similarity_threshold
        self.semantic_cache: List[CacheEntry] = []
        self.max_cache_size = semantic_cache_size

    def build_messages_with_cache(
        self,
        system_prompt: str,
        tools: List[Dict],
        conversation_history: List[Dict],
        user_message: str
    ) -> List[Dict]:
        """
        构建带 Prompt Caching 断点的消息列表。
        缓存策略：
        - 断点1：系统提示词之后 → 缓存所有系统提示词
        - 断点2：工具定义之后 → 缓存所有工具定义
        后续请求将命中这些缓存，节省 90% 输入成本。
        """
        system_block = {
            "type": "text",
            "text": system_prompt,
            "cache_control": {"type": "ephemeral"}
        }

        tools_text = json.dumps(tools, ensure_ascii=False, indent=2)
        tools_block = {
            "type": "text",
            "text": f"## Available Tools\n{tools_text}",
            "cache_control": {"type": "ephemeral"}
        }

        messages = [system_block, tools_block] + conversation_history + [
            {"role": "user", "content": user_message}
        ]
        return messages

    async def query_with_semantic_cache(
        self,
        user_query: str,
        llm_call_func: callable
    ) -> Dict:
        """
        语义缓存查询：
        1. 计算查询的 embedding
        2. 在缓存中搜索最相似的条目
        3. 如果相似度 >= 阈值，直接返回缓存响应
        4. 否则调用 LLM 并缓存结果
        """
        # 1. 计算查询 embedding
        query_embedding = self._get_embedding(user_query)

        # 2. 搜索相似缓存
        best_match, best_similarity = self._find_best_match(query_embedding)

        # 3. 命中：直接返回
        if best_match and best_similarity >= self.similarity_threshold:
            return {
                "response": best_match.response,
                "cached": True,
                "similarity": best_similarity,
                "tokens_saved": best_match.tokens_saved
            }

        # 4. 未命中：调用 LLM
        response = await llm_call_func(user_query)

        # 5. 缓存新结果
        self._add_to_cache(query_embedding, response, tokens_saved=500)

        return {"response": response, "cached": False}

    def _get_embedding(self, text: str) -> np.ndarray:
        """调用 Embedding API 获取向量表示"""
        # 实际实现需调用 OpenAI/Anthropic Embedding API
        cached_hash = hashlib.md5(text.encode()).hexdigest()
        np.random.seed(int(cached_hash, 16) % (2**32))
        return np.random.randn(1536)

    def _find_best_match(self, query_emb: np.ndarray) -> tuple:
        """在语义缓存中查找最相似条目，使用余弦相似度"""
        if not self.semantic_cache:
            return None, 0.0

        best_match = None
        best_similarity = 0.0

        for entry in self.semantic_cache:
            similarity = self._cosine_similarity(query_emb, entry.query_embedding)
            if similarity > best_similarity:
                best_similarity = similarity
                best_match = entry

        return best_match, best_similarity

    def _cosine_similarity(self, a: np.ndarray, b: np.ndarray) -> float:
        """计算两个向量之间的余弦相似度"""
        return np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b))

    def _add_to_cache(self, embedding: np.ndarray, response: str, tokens_saved: int):
        """添加条目到语义缓存，超出容量时淘汰最旧的条目"""
        if len(self.semantic_cache) >= self.max_cache_size:
            self.semantic_cache.pop(0)  # FIFO 淘汰策略
        self.semantic_cache.append(
            CacheEntry(query_embedding=embedding, response=response, tokens_saved=tokens_saved)
        )

# ===== 使用示例 =====
client = Anthropic()
cache = DualLayerCache(anthropic_client=client)

# 第1次请求 — Prompt Cache MISS, Semantic Cache MISS → 全额 LLM 调用
messages = cache.build_messages_with_cache(
    system_prompt="你是一个专业的客服 Agent...",
    tools=[{"name": "search", "description": "..."}],
    conversation_history=[],
    user_message="如何退货？"
)

# 第2次请求 — Prompt Cache HIT (系统提示词+工具定义命中) → 节省 90%
messages2 = cache.build_messages_with_cache(
    system_prompt="你是一个专业的客服 Agent...",  # 相同前缀 → 缓存命中!
    tools=[{"name": "search", "description": "..."}],
    conversation_history=[],
    user_message="如何换货？"
)

模型分层路由 — 用对模型，省对钱
将简单任务路由到便宜模型，复杂任务升级到强大模型

模型分层 (Model Tiering) 是 Agent 成本优化的核心策略：并非所有任务都需要最强的模型。通过智能路由将任务分配到最适合的模型层级，可以在保持质量的同时大幅降低成本：

  • 三层模型架构
    • Tier 1 — 快速层：Claude Haiku 或 GPT-4o-mini。处理简单分类、意图识别、问候回复、状态查询等低复杂度任务。响应时间 < 1s，成本 < $0.001/次
    • Tier 2 — 标准层：Claude Sonnet 或 GPT-4o。处理大部分 Agent 任务：工具调用、中等复杂度推理、RAG 问答、代码生成。响应时间 2-5s，成本约 $0.01-0.03/次
    • Tier 3 — 高级层：Claude Opus。处理最复杂的任务：多步推理链、复杂代码架构设计、多文档综合分析。响应时间 5-15s，成本约 $0.05-0.15/次
  • 路由决策因素
    • 查询复杂度评分：基于用户输入的长度、关键词、历史失败模式
    • 任务类型识别：分类任务 → Tier 1；工具调用 → Tier 2；开放式推理 → Tier 3
    • 用户等级：免费用户优先路由到 Tier 1/2，付费用户可使用 Tier 3
    • 预算状态：接近预算上限时降级到低层模型
  • 路由实现方式
    • 基于规则的路由：正则表达式 + 关键词匹配识别简单问题
    • 基于分类器的路由：用小模型 (如 Haiku) 判断问题复杂度，决定升级到哪个层级
    • 动态路由：运行时监测 Agent 行为，如果 Haiku 多次失败则自动升级到 Sonnet
模型分层的风险

路由器的判断本身也可能出错——把一个需要深度推理的问题误判为简单问题，导致用户得到低质量回答。建议在路由器中设置"不确定即升级"策略：当分类器对复杂度判断的置信度低于 70% 时，默认升级到高层模型。

ModelRouter 类 — 智能模型路由器实现
基于查询复杂度的自适应模型选择与自动升级机制
Python — 智能模型路由器 (规则 + 分类器 + 动态升级) 
import re
from enum import Enum
from dataclasses import dataclass
from typing import Callable, Optional

class ModelTier(Enum):
    FAST = "haiku"      # $0.80/$4.00 per 1M — 简单任务
    STANDARD = "sonnet"   # $3.00/$15.00 per 1M — 日常 Agent 任务
    ADVANCED = "opus"     # $15.00/$75.00 per 1M — 复杂推理

@dataclass
class RoutingDecision:
    tier: ModelTier
    reason: str
    confidence: float  # 0.0 - 1.0
    estimated_tokens: int
    estimated_cost: float

class ModelRouter:
    """
    模型路由器 — 根据查询复杂度智能选择模型层级。
    包含三个决策层：规则层、分类器层、动态升级层。
    """

    # 简单任务的规则特征
    SIMPLE_PATTERNS = [
        r"^(你好|hi|hello|早上好|晚上好|再见)",    # 问候
        r"^(谢谢|感谢|thanks)",                      # 感谢
        r"^(是|否|对|不对|yes|no)$",                  # 简单确认
        r"^现在几点|今天周几|今天是几号",               # 简单查询
        r"^什么是\s*\w{1,6}$"                          # 极短的"What is X"
    ]

    # 复杂任务的特征
    COMPLEX_PATTERNS = [
        r"(设计|架构|分析|评估|比较|对比|重构|优化方案)",
        r"(代码审查|code review|代码review)",
        r"(多步|复杂|综合|全面|深度)",
        r"(帮我写|帮我实现|帮我设计).{30,}",
        r".*(法律|金融|医疗|安全漏洞).*"
    ]

    def __init__(self, complexity_classifier: Optional[Callable] = None):
        self.classifier = complexity_classifier
        self.routing_history: list = []
        self.cost_breakdown = {tier: 0.0 for tier in ModelTier}

    def route(self, query: str, conversation_context: Optional[str] = None) -> RoutingDecision:
        """
        三步决策流程：
        1. 规则层：快速匹配明显简单/复杂的查询
        2. 分类器层：用小模型评估查询复杂度
        3. 动态升级层：检查上下文，判断是否需要升级
        """

        # Layer 1: 规则层
        rule_result = self._rule_based_route(query)
        if rule_result and rule_result.confidence > 0.9:
            return rule_result

        # Layer 2: 分类器层（使用小模型评估复杂度）
        if self.classifier:
            complexity = self._classify_complexity(query, conversation_context)
        else:
            complexity = self._heuristic_complexity(query)

        # Layer 3: 动态升级 — 根据上下文调整
        complexity = self._apply_contextual_upgrade(complexity, conversation_context)

        # 决策
        if complexity < 0.25:
            tier = ModelTier.FAST
            reason = "低复杂度，使用快速模型"
        elif complexity < 0.65:
            tier = ModelTier.STANDARD
            reason = "中等复杂度，使用标准模型"
        else:
            tier = ModelTier.ADVANCED
            reason = "高复杂度，使用高级模型"

        decision = RoutingDecision(
            tier=tier,
            reason=reason,
            confidence=complexity if complexity > 0.75 else 1 - complexity,
            estimated_tokens=self._estimate_tokens(query),
            estimated_cost=0.0
        )

        self.routing_history.append({"query": query[:80], "tier": tier.value, "complexity": complexity})
        return decision

    def _rule_based_route(self, query: str) -> Optional[RoutingDecision]:
        """规则引擎 — 快速识别明确简单/复杂的查询"""
        for pattern in self.SIMPLE_PATTERNS:
            if re.match(pattern, query.strip()):
                return RoutingDecision(
                    tier=ModelTier.FAST, reason="规则匹配：简单查询",
                    confidence=0.95, estimated_tokens=100, estimated_cost=0.0005
                )

        for pattern in self.COMPLEX_PATTERNS:
            if re.search(pattern, query):
                return RoutingDecision(
                    tier=ModelTier.ADVANCED, reason="规则匹配：复杂查询",
                    confidence=0.9, estimated_tokens=3000, estimated_cost=0.08
                )

        return None  # 规则无法判断，升级到分类器层

    def _heuristic_complexity(self, query: str) -> float:
        """
        启发式复杂度评估（不需要 LLM 调用的快速近似）：
        - 查询长度
        - 多步推理关键词
        - 代码相关度
        - 列表/结构化请求
        """
        score = 0.0

        # 长度因子
        if len(query) > 200: score += 0.3
        elif len(query) > 100: score += 0.15

        # 推理关键词
        reasoning_keywords = ["为什么", "怎么", "原因", "分析", "方案", "步骤"]
        score += 0.05 * sum(1 for kw in reasoning_keywords if kw in query)

        # 代码相关
        if "```" in query or "def " in query or "class " in query: score += 0.25

        # 多部分请求
        if len(re.findall(r"\d+[\.、)]", query)) >= 3: score += 0.2

        return min(score, 1.0)

    def _apply_contextual_upgrade(self, complexity: float, context: Optional[str]) -> float:
        """根据对话上下文动态调整复杂度评分"""
        if context and len(context) > 2000:
            complexity = min(complexity + 0.2, 1.0)  # 长上下文增加复杂度
        return complexity

    def _estimate_tokens(self, query: str) -> int:
        """快速估算 Token 数 (简化为字符/4)"""
        return max(len(query) // 2, 50)

    def get_stats(self) -> Dict:
        """获取路由统计信息"""
        total = len(self.routing_history)
        if total == 0: return {}
        return {
            "total_requests": total,
            "tier_distribution": {
                tier.value: sum(1 for r in self.routing_history if r["tier"] == tier.value) / total
                for tier in ModelTier
            }
        }

# ===== 使用示例 =====
router = ModelRouter()

# 1. 简单问候 → FAST 层

print(f"路由: {decision.tier.value} | 原因: {decision.reason}")
# 输出: 路由: haiku | 原因: 规则匹配：简单查询

# 2. 复杂架构分析 → ADVANCED 层

print(f"路由: {decision.tier.value} | 原因: {decision.reason}")
# 输出: 路由: opus | 原因: 规则匹配：复杂查询

# 3. 中等查询 → STANDARD 层
decision = router.route("帮我查一下北京的天气，然后告诉我今天适合穿什么")
print(f"路由: {decision.tier.value} | 复杂度: {decision.confidence:.2f}")

并行工具调用 — 从串行到并发的加速
识别独立性，将Agent响应时间从 15s 降到 5s

当 Agent 需要调用多个工具时，如果这些工具之间没有数据依赖关系，就应该并行执行。这是性能优化中投入产出比最高的策略——零额外成本，显著减少延迟：

  • 识别可并行的工具调用
    • 独立查询："查询北京、上海、广州的天气" → 3 次独立的天气 API 调用，完全可以并行
    • 多源检索：同时搜索内部知识库、外部文档、代码仓库 → 3 个独立的检索操作
    • 多维度分析：同时进行情感分析、实体提取、关键词提取 → 3 个独立的 NLP 任务
    • 数据验证：同时校验字段A、B、C的合法性 → 3 个独立的验证
  • 依赖感知调度
    • 必须串行的情况：工具 B 的输入参数来自工具 A 的输出（如：先搜索文档ID，再用ID获取文档内容）
    • 部分并行：A、B 各自独立但都依赖 C 的结果 → 先并行执行 A、B，等 C 完成后决定下一步
    • 构建依赖图 (DAG)：工具 A → 工具 C → 工具 E；工具 B → 工具 D。则可以 [A||B] → [C||D] → [E]
  • 性能对比（典型 Agent 任务）
    执行模式3 个工具各 3s5 个工具各 2s10 个工具各 1s
    串行执行9 秒10 秒10 秒
    并行执行3 秒 (3x)2 秒 (5x)1 秒 (10x)
实践建议

在 Agent 循环中，LLM 每次可以返回多个工具调用 (tool_calls 数组)。现代的 LLM SDK 支持同时执行这些工具调用。确保每个工具都是线程安全的（没有全局副作用），这是并行执行的前提条件。

依赖感知并行工具执行器
自动构建依赖图，最大化并发执行独立工具
Python — 依赖感知并行工具调度器 
import asyncio
import time
from typing import List, Dict, Set, Tuple
from dataclasses import dataclass

@dataclass
class ToolCall:
    name: str
    arguments: Dict
    depends_on: Set[str]  # 依赖的工具调用名称集合
    result: str = ""

class ParallelToolExecutor:
    """
    依赖感知的并行工具执行器。
    自动构建 DAG (有向无环图)，在每个层级上最大化并行执行。
    """

    def __init__(self, tool_registry: Dict):
        """tool_registry: {name: async_callable}"""
        self.tools = tool_registry

    async def execute_all(self, tool_calls: List[ToolCall]) -> Dict[str, str]:
        """
        按依赖关系分层执行所有工具调用。
        同层无依赖的工具并行执行，不同层按依赖顺序执行。

        例：A→C→E, B→D→E, F(独立)
        Layer 0: [A, B, F] 并行
        Layer 1: [C, D] 并行
        Layer 2: [E]
        """
        remaining = {tc.name: tc for tc in tool_calls}
        completed: Dict[str, str] = {}
        start_time = time.time()

        while remaining:
            # 找出所有依赖已满足的工具调用
            ready = [
                tc for tc in remaining.values()
                if tc.depends_on.issubset(completed.keys())
            ]

            if not ready:
                unresolved = [
                    f"{tc.name} (等待: {tc.depends_on - set(completed.keys())})"
                    for tc in remaining.values()
                ]
                raise RuntimeError(f"检测到循环依赖或缺失工具: {unresolved}")

            # 并行执行所有就绪的工具
            layer_start = time.time()
            tasks = [self._execute_one(tc) for tc in ready]
            results = await asyncio.gather(*tasks, return_exceptions=True)

            layer_time = time.time() - layer_start
            print(f"Layer [{len(ready)} tools] completed in {layer_time:.2f}s")

            for tc, result in zip(ready, results):
                if isinstance(result, Exception):
                    completed[tc.name] = f"Error: {str(result)}"
                else:
                    completed[tc.name] = str(result)
                    tc.result = str(result)

                del remaining[tc.name]

        total_time = time.time() - start_time
        print(f"全部 {len(tool_calls)} 个工具执行完成，总耗时 {total_time:.2f}s")
        return completed

    async def _execute_one(self, tc: ToolCall) -> str:
        """执行单个工具调用"""
        func = self.tools.get(tc.name)
        if not func:
            return f"Unknown tool: {tc.name}"
        return await func(**tc.arguments)

    def estimate_speedup(self, tool_calls: List[ToolCall]) -> float:
        """估算并行执行相比串行的加速比"""
        if not tool_calls:
            return 1.0

        serial_time = len(tool_calls)  # 假设每个工具1单位时间

        # 计算拓扑层数
        completed = set()
        remaining = {tc.name: tc for tc in tool_calls}
        layers = 0
        while remaining:
            ready = {name for name, tc in remaining.items()
                     if tc.depends_on.issubset(completed)}
            completed.update(ready)
            for name in ready: del remaining[name]
            layers += 1

        parallel_time = layers
        return serial_time / parallel_time

# ===== 使用示例 =====
async def main():
    # 模拟工具注册表
    async def weather(city: str): await asyncio.sleep(1); return f"{city}天气: 晴 22°C"
    async def news(topic: str): await asyncio.sleep(1.2); return f"{topic}最新新闻: ..."
    async def summarize(text: str): await asyncio.sleep(0.8); return f"摘要: {text[:30]}..."

    executor = ParallelToolExecutor({
        "weather": weather,
        "news": news,
        "summarize": summarize
    })

    # 定义工具调用及其依赖关系
    calls = [
        ToolCall(name="weather_bj", arguments={"city": "北京"}, depends_on=set()),
        ToolCall(name="weather_sh", arguments={"city": "上海"}, depends_on=set()),
        ToolCall(name="news_ai",   arguments={"topic": "AI"},   depends_on=set()),
        ToolCall(name="summary",   arguments={"text": "...合成文本..."}, depends_on={"weather_bj", "weather_sh"}),
    ]

    # 估算加速比
    speedup = executor.estimate_speedup(calls)
    print(f"预计并行加速: {speedup:.1f}x (4 个工具，2 个依赖层级)")

    # 并行执行
    results = await executor.execute_all(calls)
    print(f"结果: {list(results.keys())}")
    # 串行需 4s，并行约 2s (Layer0: 3个并行，Layer1: 1个)

asyncio.run(main())

上下文窗口优化 — 在有限空间内保留最大价值
智能摘要、滑动窗口、重要性评分三管齐下

虽然模型的上下文窗口不断增长（Claude 200K, Gemini 2M），但越大的上下文意味着越高的延迟和成本。而且 Agent 的多步循环会快速累积对话历史和工具结果，上下文膨胀是 Agent 性能的头号隐形杀手：

  • 智能摘要压缩 (Intelligent Summarization)
    • 使用 LLM 对历史对话进行阶段性摘要：每 N 轮或上下文达到一定阈值时，将之前的内容压缩为摘要
    • 摘要应保留：当前任务目标、关键决策点、已获取的重要信息、待完成的事项
    • 摘要可以丢弃：冗余的中间推理过程、失败的尝试（保留教训即可）、重复的工具调用结果
    • 推荐摘要间隔：每 10-15 轮对话自动触发摘要
  • 滑动窗口 + 重要性评分 (Sliding Window with Importance Scoring)
    • 为每条消息（包括用户消息、助手消息、工具结果）计算"重要性分数"
    • 重要性评分维度：消息对后续决策的帮助程度、包含的信息是否可从其他消息中推断、时效性
    • 当上下文接近窗��上限时，按重要性分数从低到高裁剪消息
    • 始终保留：系统提示词、工具定义、最近 N 条消息（N 通常为 5-10）
  • 分块处理大文档 (Chunked Processing)
    • 不要将整个 100 页 PDF 塞进上下文
    • 使用检索增强 (RAG)：先通过向量检索找到相关段落，只将相关段落放入上下文
    • 对超长文档使用 Map-Reduce 模式：分段分析 → 合并结果
  • 上下文使用率监控
    • 在每次 LLM 调用前检查 input_tokens 数量
    • 设置警戒线：当上下文使用超过 70% 时触发摘要，超过 90% 时强制执行裁剪
    • 记录上下文使用率的时序变化，识别哪些类型的任务最容易导致上下文膨胀

上下文窗口管理器 — 自动摘要与裁剪
智能维护上下文在预算范围内，摘要低价值消息
Python — 上下文窗口管理器 (摘要 + 重要性评分 + 自动裁剪) 
import tiktoken
from dataclasses import dataclass, field
from typing import List, Dict, Optional

@dataclass
class ContextMessage:
    role: str
    content: str
    importance: float = 0.5   # 0.0 (可丢弃) ~ 1.0 (必须保留)
    token_count: int = 0
    is_summary: bool = False

class ContextWindowManager:
    """
    上下文窗口管理器：
    - 追踪 Token 使用量
    - 当接近预算时自动触发摘要
    - 按重要性评分裁剪低价值消息
    - 保证核心信息始终保留
    """

    def __init__(
        self,
        max_tokens: int = 100000,          # 上下文窗口总 Token 预算
        system_prompt_tokens: int = 2000,
        tool_def_tokens: int = 3000,
        summary_trigger_ratio: float = 0.7,   # 70% 触发摘要
        force_prune_ratio: float = 0.9,       # 90% 强制裁剪
        keep_recent: int = 6                  # 始终保留最近 N 条消息
    ):
        self.max_tokens = max_tokens
        self.reserved_tokens = system_prompt_tokens + tool_def_tokens
        self.available_tokens = max_tokens - self.reserved_tokens
        self.summary_trigger = int(self.available_tokens * summary_trigger_ratio)
        self.force_prune = int(self.available_tokens * force_prune_ratio)
        self.keep_recent = keep_recent
        self.messages: List[ContextMessage] = []
        self.summary: Optional[str] = None
        self.encoder = tiktoken.get_encoding("cl100k_base")
        self.total_tokens_used = 0

    def add_message(self, role: str, content: str, importance: float = 0.5) -> List[ContextMessage]:
        """添加消息并自动管理上下文窗口"""
        token_count = len(self.encoder.encode(content))
        msg = ContextMessage(role=role, content=content, importance=importance, token_count=token_count)
        self.messages.append(msg)
        self.total_tokens_used += token_count

        # 检查是否需要管理上下文
        if self.total_tokens_used > self.force_prune:
            return self.force_prune_context()
        elif self.total_tokens_used > self.summary_trigger:
            return self.create_summary_and_trim()

        return self.messages

    def create_summary_and_trim(self) -> List[ContextMessage]:
        """
        创建摘要 + 裁剪低价值消息：
        1. 取最旧的 N 条消息生成摘要
        2. 用摘要替换这些消息
        3. 保留最近 keep_recent 条消息不变
        """
        if len(self.messages) <= self.keep_recent + 3:
            return self.messages  # 消息太少，不值得摘要

        # 分离待摘要的消息和应保留的消息
        to_summarize = self.messages[:-self.keep_recent]
        to_keep = self.messages[-self.keep_recent:]

        # 生成摘要 (实际应调用 LLM，此处简化为拼接)
        summary_content = self._generate_summary(to_summarize)

        # 替换：摘要消息 + 最近消息
        summary_msg = ContextMessage(
            role="system",
            content=f"[对话历史摘要]\n{summary_content}",
            importance=0.9,
            token_count=len(self.encoder.encode(summary_content)),
            is_summary=True
        )

        self.messages = [summary_msg] + to_keep
        self.total_tokens_used = sum(m.token_count for m in self.messages)
        self.summary = summary_content

        print(f"[ContextManager] 摘要完成: 压缩 {len(to_summarize)} 条消息 → Token 使用: {self.total_tokens_used}")
        return self.messages

    def force_prune_context(self) -> List[ContextMessage]:
        """
        强制裁剪：按重要性分数从低到高删除消息，直到 Token 预算回到安全线以下。
        最近 keep_recent 条消息始终保留。
        """
        if len(self.messages) <= self.keep_recent:
            return self.messages

        recent = self.messages[-self.keep_recent:]
        candidates = self.messages[:-self.keep_recent]

        # 按重要性升序排列（最低的在前面）
        candidates.sort(key=lambda m: m.importance)

        target_tokens = int(self.available_tokens * 0.6)
        current_tokens = sum(m.token_count for m in recent)

        kept = []
        removed_count = 0
        for msg in candidates:
            if current_tokens + msg.token_count <= target_tokens and msg.importance >= 0.5:
                kept.append(msg)
                current_tokens += msg.token_count
            else:
                removed_count += 1

        self.messages = kept + recent
        self.total_tokens_used = current_tokens

        print(f"[ContextManager] 强制裁剪: 删除 {removed_count} 条低价值消息，当前 Token: {self.total_tokens_used}")
        return self.messages

    def _generate_summary(self, messages: List[ContextMessage]) -> str:
        """生成消息摘要 (简化版，实际应调用 LLM)"""
        key_points = []
        for msg in messages:
            if msg.importance > 0.6 and not msg.is_summary:
                prefix = msg.content[:50].replace("\n", " ")
                key_points.append(f"- [{msg.role}] {prefix}...")
        return "\n".join(key_points[:15])  # 最多15条要点

    def get_context_stats(self) -> Dict:
        """获取上下文使用统计"""
        return {
            "total_messages": len(self.messages),
            "tokens_used": self.total_tokens_used,
            "tokens_available": self.available_tokens,
            "usage_percent": self.total_tokens_used / self.available_tokens * 100,
            "has_summary": self.summary is not None
        }

# ===== 使用示例 =====
cwm = ContextWindowManager(max_tokens=50000)

# 模拟 20 轮对话
for i in range(20):
    importance = 0.3 + (0.03 * i)  # 模拟逐渐增加的重要性
    msgs = cwm.add_message(
        role="user" if i % 2 == 0 else "assistant",
        content=f"消息 #{i}: " + "分析数据 " * (10 + i * 5),
        importance=importance
    )

print(cwm.get_context_stats())
# {'total_messages': 15, 'tokens_used': 12500, 'usage_percent': 25.0, ...}

批处理与高吞吐 — 离线场景的成本利器
Batch API 50% 折扣的适用场景与实践

对于不需要实时响应的任务（数据标注、评估运行、报告生成），使用 Batch API 可以节省 50% 的 API 费用：

  • Batch API 工作原理
    • Anthropic Message Batches API：提交一批请求（最多 10,000 个），Anthropic 在 24 小时内异步处理完成。价格是实时 API 的 50%
    • OpenAI Batch API：类似机制，提交批量请求后在 24 小时内完成，输出 Token 享受 50% 折扣
    • 请求在后台队列中排队，资源闲置时处理，不会抢占实时用户的资源
  • 适合批处理的场景
    • 评估运行：对评估集的数百个测试用例批量运行 Agent，获取准确率、延迟、成本指标
    • 数据标注：用 LLM 对大量文本进行分类、实体提取、摘要等标注任务
    • 预计算：为常见的查询提前生成回复并缓存（如 FAQ 答案），当用户实际提问时直接返回缓存结果
    • 离线分析：日志分析、用户反馈分类、错误模式挖掘
  • 不适合批处理的场景
    • 实时对话交互（用户等待响应）
    • 多步 Agent 任务（每步依赖上一步结果，无法批处理）
    • 需要上下文延续的会话（批处理请求之间独立，无法共享会话状态）
批处理吞吐量计算

假设使用 Claude Sonnet 批处理 ($1.50/$7.50 per 1M tokens)，每个评估用例消耗 3000 tokens。1000 个用例的总成本 = 1000 x 3000 x $1.50/1M = $4.50。同样的任务使用实时 API 需要 $9.00。如果每天运行一次评估，每月节省 $135。

延迟优化技巧与性能监控仪表盘
TTFT、预计算、边缘部署与全面的可观测性指标

延迟优化的目标是在不改变 LLM 推理速度的前提下，让用户感觉 Agent 响应更快：

  • 流式输出 (Streaming)
    • TTFT (Time To First Token)：从发送请求到收到第一个 Token 的时间。流式输出让用户在 200-500ms 内看到第一个字，大幅改善体感延迟
    • 对于 Agent 工具调用，流式输出可以让用户实时看到 Agent 的"思考过程"（思考 → 决定调用什么工具 → 执行中 → 返回结果），形成更好的交互体验
  • 预计算与预热
    • 对高频查询（如热门问题）预计算答案并缓存
    • 保持与 LLM API 的常驻连接（HTTP Keep-Alive），减少 TCP/TLS 握手开销
    • 预热 GPU 实例：在预期流量高峰前预先启动 GPU 实例，避免冷启动

优化阶段 策略 每请求成本 平均延迟 备注
