第 3 章

工具调用机制

工具是 Agent 的"手"，掌握 Function Calling 的原理、Schema 设计和执行生命周期，是构建实用 Agent 的核心技能。

⚙
Function Calling 深度机制：从用户输入到工具执行
一步一步追踪 Function Calling 的完整链路，理解每个环节的作用

Function Calling 是连接 LLM 的"语言世界"和外部系统的"行动世界"的桥梁。理解它不能停留在"模型能调用函数"的表面认知上，而需要深入到每个环节的技术细节中。

完整链路追踪

当用户说"帮我查一下北京今天的天气"时，以下步骤依次发生：

步骤1 — 用户输入到达编排器

编排器将用户输入包装为 messages[0] = {"role": "user", "content": "帮我查一下北京今天的天气"}

步骤2 — 工具 Schema 注入

编排器将所有可用工具的 JSON Schema 附加到请求中。LLM 此时"看到"了 get_weather、search_news 等多个工具的定义。

步骤3 — LLM 语义匹配推理

LLM 在内部做"语义匹配"：用户的"查天气"与 get_weather 的 description (获取天气) 高度匹配；"北京"对应 city 参数；"今天"意味着不需要额外的时间参数。

步骤4 — LLM 输出工具调用请求

LLM 不生成文本回答，而是输出一个结构化的 JSON：{function: "get_weather", arguments: {city: "北京", unit: "celsius"}}。注意模型甚至合理推断出默认使用摄氏度。

步骤5 — 编排器解析并执行

编排器接收工具调用请求，从工具注册表中查找 get_weather，传入参数执行。实际的 API 调用由工具函数完成。

步骤6 — 结果回传与二次推理

工具返回 {"temperature": 22, "condition": "晴"}。编排器将结果追加到 messages，再次调用 LLM。LLM 此时基于工具结果生成自然语言回答："北京今天晴天，温度22°C..."

核心理解

Function Calling 的精妙之处在于：LLM 不"知道"函数做了什么，它只"知道"函数的签名和描述。LLM 的角色是"翻译官"——将用户的自然语言请求翻译为结构化的工具调用指令，然后将工具返回的结构化结果翻译回自然语言回答。LLM 从未真正"执行"任何代码。

JSON Schema 设计完整指南
工具 Schema 的质量直接决定 Agent 调用工具的能力，这是一门需要精心打磨的技术

JSON Schema 是 LLM 理解工具的"唯一窗口"。Schema 写得好的工具，LLM 几乎从不误用；Schema 写得差的工具，LLM 要么不用、要么乱用。以下是经过实践验证的设计规范。

Schema 设计的黄金法则

法则 好的实践 坏的实践 原因
描述要具体 "description": "根据城市名称获取实时天气数据，包括温度、湿度、天气状况和风速。适用于当前天气查询，不适用于天气预报。" "description": "获取天气" 具体描述帮助 LLM 准确判断何时应该调用此工具，何时不应该
类型要精确 "city": {"type": "string", "description": "城市名称，如 '北京' 或 'Beijing'。支持中英文。"} "city": {"type": "object"} 精确的类型约束防止 LLM 传入错误格式的参数
枚举优于自由文本 "unit": {"type": "string", "enum": ["celsius", "fahrenheit"]} "unit": {"type": "string"} enum 消除歧义，防止 LLM 生成 "度"、"C"、"摄氏"等不规范值
标注必填参数 "required": ["city"] "required": [] 或不写 required 明确告诉 LLM 哪些参数不可或缺
值域要限制 "limit": {"type": "integer", "minimum": 1, "maximum": 100, "default": 20} "limit": {"type": "integer"} minimum/maximum 防止参数越界，default 提供合理默认值
一工具一职责 get_weather、get_forecast、get_air_quality 分三个工具 一个 get_all_weather_data 工具包含所有功能 细粒度工具降低 LLM 的调用出错率

完整 Schema 示例：一个设计良好的天气查询工具

JSON — 高质量工具 Schema

  "function": {
    "name": "get_current_weather",
    "description": "获取指定城市的实时天气数据。返回温度、湿度、天气状况（晴/雨/阴等）、风速和风向。注意：此工具仅返回当前天气，如需未来天气预报请使用 get_weather_forecast 工具。支持全球主要城市。",
    "parameters": {
      "type": "object",
      "properties": {
        "city": {
          "type": "string",
          "description": "城市名称。支持中英文，如 '北京'、'Beijing'、'New York'。不支持经纬度坐标查询。",
          "examples": ["北京", "Shanghai", "Tokyo"]
        },
        "unit": {
          "type": "string",
          "enum": ["celsius", "fahrenheit"],
          "description": "温度单位。celsius 为摄氏度，fahrenheit 为华氏度。如果用户没有特别说明，默认使用 celsius。",
          "default": "celsius"
        },
        "include_details": {
          "type": "boolean",
          "description": "是否返回详细的附加信息（如紫外线指数、体感温度、日出日落时间等）。默认 false，仅在用户明确要求详细信息时设为 true。",
          "default": false
        }
      },
      "required": ["city"],
      "additionalProperties": false
    }
  }
}
Schema 设计中常见的3个致命错误
  1. description 与另一个工具雷同：当两个工具有相似的 description 时，LLM 无法区分何时用哪个。确保每个工具的 description 都明确说明了它的"独特适用场景"。
  2. 参数类型设为 "object" 或 "string" 来 "兜底"：这看似万能，实际上让 LLM 无所适从。精确的类型约束是给 LLM 的"路标"。
  3. 忘记 additionalProperties: false：如果不设置这个，LLM 可能会在参数中添加你未定义的额外字段，导致后端解析失败。

工具生命周期：从注册到退役
每个工具在 Agent 系统中都经历完整的创建-使用-维护-退役周期

工具不是写完了就完事。在 Agent 系统中，每个工具都有一个完整的生命周期，理解这个生命周期有助于你设计更健壮的工具。

阶段1 — 定义 (Define)

明确工具的功能边界、输入输出规范、适用场景和限制。这个阶段的关键产出是清晰的功能描述和完整的 JSON Schema。问自己三个问题：这个工具解决什么问题？它与其他工具的边界在哪里？什么情况下不应该使用这个工具？

阶段2 — 注册 (Register)

将工具添加到 Agent 的工具注册表中。注册不仅仅是"加进去"，还包括：设置工具的权限级别（只读/可写/危险）、配置速率限制（每分钟最多调用多少次）、指定超时时间（单次调用最长等待多久）。

阶段3 — 发现 (Discover)

LLM 根据当前上下文"发现"需要调用哪个工具。这一步完全由 LLM 的语义理解能力驱动。优化方向：让工具的 description 在语义上与众不同，使 LLM 在正确的场景下能"一眼认出"正确的工具。

阶段4 — 选择 (Select)

当多个工具都能完成任务时，LLM 需要做出选择。影响选择的因素包括：description 的匹配度、参数复杂度、工具在上下文中的位置（靠前的工具有一定优势）。这也是为什么"一工具一职责"如此重要的原因。

阶段5 — 验证 (Validate)

编排器收到 LLM 的工具调用请求后，进行参数验证：参数类型是否正确？必填参数是否齐全？参数值是否在合法范围内？是否有越权操作？验证通过后才进入执行。

阶段6 — 执行 (Execute)

实际调用工具函数，传入参数，等待返回。这个阶段要做：超时控制、异常捕获、结果截断（防止返回过大的数据撑满上下文）、日志记录。

阶段7 — 观察 (Observe)

编排器将工具返回结果格式化后反馈给 LLM。结果应该：结构化（JSON 或 Markdown 表格）、精简（去掉无关字段）、带元信息（执行耗时、数据来源、记录数）。

阶段8 — 退役 (Retire)

当工具不再需要、被更好的工具替代、或发现安全漏洞时，应该及时从注册表中移除。退役前需要：检查是否有 Agent 流程依赖此工具、提供替代方案、更新文档。

工具分类体系：四类工具的各自之道
不同类型的工具有不同的设计考量、性能特征和风险点

Agent 可以调用的工具种类繁多，但都可以归入以下四个类别。理解每个类别的特征，有助于你为每类工具采用最合适的设计策略。

🔍
数据检索类

做什么：从数据库、API、文件系统、搜索引擎中获取信息。

典型工具：SQL查询、REST API调用、向量检索、网页抓取

关键考量：

  • 返回数据量控制（截断策略）
  • 查询性能（索引、缓存）
  • 数据新鲜度（实时 vs 缓存）

风险等级：低（只读操作）

低风险
🔢
计算处理类

做什么：执行计算、转换、分析操作。

典型工具：代码执行器、数据处理（pandas）、数学计算、文本分析

关键考量：

  • 执行环境隔离（沙箱）
  • 资源限制（CPU/内存/时间）
  • 输入输出的大小限制

风险等级：中（需要沙箱保护）

中等风险
🚀
操作/执行类

做什么：对环境产生实际变更。

典型工具：文件写入、数据库更新、发送邮件、部署操作、创建资源

关键考量：

  • 操作前确认机制
  • 幂等性（重复执行安全）
  • 回滚/撤销能力
  • 操作审计日志

风险等级：高（不可逆操作）

高风险
🎬
多模态类

做什么：处理图片、音频、视频等非文本数据。

典型工具：图像生成/识别、语音转文字、视频分析、OCR

关键考量：

  • 二进制数据传输
  • 格式转换和兼容性
  • 处理和传输成本

错误处理模式：让 Agent 优雅地失败
工具执行总会出错，关键在于出错后的处理策略

在 Agent 系统中，工具执行错误是常态而非例外。网络抖动、API 限流、参数错误、数据变化——任何环节都可能出错。优秀的 Agent 不是不会出错，而是知道在出错后如何优雅恢复。

三层错误处理金字塔

L1 — 自动重试 (Automatic Retry)

适用场景：网络超时、临时性服务不可用、限流错误

策略：指数退避重试（1s -> 2s -> 4s -> 8s），最多3次。重试时参数不变，因为错误是临时性的而非参数错误。

关键细节：只在幂等操作上重试。对于非幂等操作（如发送邮件），重试可能导致重复执行。

L2 — 智能降级 (Intelligent Fallback)

适用场景：工具不可用、数据源不可达、权限不足

策略：切换到备用方案。例如：主数据库查询失败 -> 查询只读副本；实时 API 超时 -> 返回缓存数据（标注"非实时"）；文件系统不可写 -> 改为返回内容让用户手动保存。

关键细节：降级方案必须明确告知 LLM 和用户当前的降级状态和数据的可靠性等级。

L3 — 优雅失败 (Graceful Failure)

适用场景：所有重试和降级都失败

策略：向 LLM 返回结构化的错误信息，而不是抛出异常。错误信息应包含：发生了什么（what）、为什么（why）、建议怎么做（suggestion）。让 LLM 能够基于这些信息调整策略或向用户解释。

错误信息模板：

{"error": true, "type": "execution_failed", "tool": "query_database",
"message": "数据库连接超时（已重试3次）",
"suggestion": "请尝试以下替代方案：1) 缩小查询范围减少数据量；2) 使用 read_only_replica 工具查询只读副本；3) 如果不需要实时数据，可以使用 cached_query 工具获取昨天的缓存数据。"}

错误处理的核心原则

不要向 LLM 抛异常。LLM 不理解 Java/Python 的异常堆栈。所有的错误都应该被捕获并转换为LLM 可理解的结构化反馈。给 LLM 的信息越有用，它越可能自动找到解决方案。

案例1：数据库查询工具

描述内容分析
差 "查询数据库" 太模糊。什么数据库？查询什么？LLM 无法判断何时该用
好 "在 PostgreSQL 数据库中执行只读 SQL 查询。支持的查询类型：SELECT、EXPLAIN、SHOW。返回结果最多 500 行。如果用户涉及销量、订单、用户等业务数据且需要精确统计，优先使用此工具。禁止执行 INSERT/UPDATE/DELETE。" 明确：数据库类型、支持的SQL、结果上限、适用场景、禁止操作

好 "从指定路径读取文件内容。支持纯文本(.txt)、Markdown(.md)、代码文件(.py/.js/.ts等)和JSON/CSV文件。单次最大读取 100KB。如果文件大于100KB，请先使用 file_preview 工具查看前1000字符，再决定是否需要分段读取。路径必须是绝对路径。" 明确：支持格式、大小限制、超大文件处理策略、路径要求

描述写作的5个黄金问题

写完一个工具的 description 后，用这5个问题自检：

  1. 何时用？ — LLM 读完描述后，能准确判断什么场景下该调用这个工具吗？
  2. 何时不用？ — 描述是否暗示了此工具的边界（什么场景下应该用别的工具）？
  3. 有何限制？ — 大小限制、速率限制、权限要求是否说清楚了？
  4. 输入什么？ — 每个参数的类型、含义、示例值是否都明确了？
  5. 输出什么？ — 返回结果的结构和内容是否有说明？
一个有用的思维实验

想象你只能通过工具描述来理解一个工具——你看不到源代码，不能运行它，也没有其他文档。在这种情况下，你能准确判断何时该用这个工具、应该传什么参数、能拿到什么返回值吗？如果不能，说明你的工具描述还需要完善。

def execute_sql_query(query: str, timeout_ms: int = 30000) -> dict:
    """执行 SQL 查询"""
    try:
        result = db_connection.execute(query, timeout=timeout_ms/1000)
        rows = result.fetchmany(500)
        return {
            "success": True,
            "row_count": len(rows),
            "columns": result.keys(),
            "data": [dict(zip(result.keys(), row)) for row in rows],
            "truncated": result.rowcount > 500
        }
    except Exception as e:
        return {
            "error": True,
            "message": f"SQL 查询失败: {e}",
            "suggestion": "请检查 SQL 语法。如果查询超时，尝试添加更精确的 WHERE 条件或使用 LIMIT 限制返回行数。"
        }

        proc = subprocess.run(
            ["python", "-c", code],
            capture_output=True, text=True, timeout=30
        )
        return {
            "success": True,
            "stdout": proc.stdout,
            "stderr": proc.stderr if proc.stderr else None,
            "return_code": proc.returncode
        }
    except subprocess.TimeoutExpired:
        return {"error": True, "message": "代码执行超时（30秒）"}


def execute_web_search(query: str, max_results: int = 10) -> dict:
    """执行网络搜索"""
    results = search_engine.search(query, num=max_results)
    return {
        "success": True,
        "query": query,
        "total_results": len(results),
        "results": [
            {"title": r.title, "snippet": r.snippet, "url": r.url}
            for r in results
        ]
    }

# 3. 工具执行循环（集成到编排器中）

class ToolExecutor:
    """工具执行器：负责工具调用的全生命周期管理"""

    def __init__(self, registry: dict, max_retries: int = 3):
        self.registry = registry
        self.max_retries = max_retries
        self.call_history = []  # 调用历史（可观测性）

    def execute(self, tool_name: str, args: dict) -> dict:
        """带完整错误处理的工具执行"""
        if tool_name not in self.registry:
            return {"error": True, "message": f"工具 '{tool_name}' 未注册"}

        func = self.registry[tool_name]

        for attempt in range(1, self.max_retries + 1):
            try:
                result = func(**args)
                self.call_history.append({
                    "tool": tool_name, "args": args,
                    "success": True, "attempt": attempt
                })
                return result

            except Exception as e:
                if attempt == self.max_retries:
                    # 所有重试失败：返回结构化的错误信息
                    self.call_history.append({
                        "tool": tool_name, "args": args,
                        "success": False, "error": str(e)
                    })
                    return {
                        "error": True,
                        "message": f"工具 '{tool_name}' 执行失败（已重试 {self.max_retries} 次）: {str(e)}",
                        "suggestion": "请尝试修改参数或使用其他工具完成此任务。"
                    }
                time.sleep(2 ** (attempt - 1))  # 指数退避

进阶话题：并行工具调用与工具编排
掌握高级工具使用技巧，让 Agent 更快、更智能

并行工具调用

当任务中的多个工具调用相互独立时，应该并行执行而非顺序执行。例如：用户问"帮我比较一下北京和上海的天气"，查询北京天气和查询上海天气是两个完全独立的操作。

顺序执行耗时：200ms（北京）+ 200ms（上海）= 400ms

并行执行耗时：max(200ms, 200ms) = 200ms — 节省50%

支持并行调用的 LLM 可以在一次响应中返回多个 tool_calls，编排器收到后并行执行，将所有结果一次性返回给 LLM。

工具链与工具组合

复杂的任务往往需要多个工具协同工作。常见模式：

  • 检索-分析-输出链：sql_query -> run_python -> 自然语言总结。先用 SQL 取数据，再用 Python 分析，最后 LLM 总结。
  • 搜索-验证-综合链：web_search 多次 -> 交叉验证 -> 综合报告。多次搜索不同关键词，验证信息一致性，生成综合报告。
  • 读取-处理-写入链：file_read -> run_python -> file_write。读取文件、处理内容、写回结果。

工具使用的反模式

  • 过度工具化：把任何小功能都做成独立工具。5个工具可能就够了，50个工具会让 LLM 患上"选择困难症"。
  • 虚假能力：工具名称听起来很强大，但实际实现很弱（如名为 "analyze_market_trends" 的工具只做了简单的平均计算）。这会严重损害 LLM 对工具的信任。
  • 缺乏超时机制：工具执行没有时间限制，一个慢查询卡住整个 Agent 循环。
工具设计的终极检验标准

一个好的工具集会让 LLM 在 90% 的场景下第一次就选对工具。如果你的 Agent 经常在多个工具之间犹豫、频繁调用"错误"的工具然后重试，那说明你的工具描述存在重叠或模糊。回到 Schema 设计上重新打磨。
