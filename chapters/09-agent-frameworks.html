第 9 章

Agent 开发框架

全面了解主流 Agent 开发框架的特点、优劣势与适用场景，帮助你做出正确的技术选型

🗺️
Agent 框架全景概览

随着 AI Agent 技术的快速演进，市面上涌现出了大量 Agent 开发框架。它们各自有不同的设计哲学、抽象层级和适用场景。理解这些框架的核心差异，是做出正确技术选型的前提。

当前主流的 Agent 框架可以分为以下几类：

  • 通用编排框架：LangChain、Semantic Kernel — 提供完整的工具链、记忆管理、RAG 支持，抽象层级较高，适合快速构建原型和复杂工作流。
  • 多 Agent 协作框架：CrewAI、AutoGen、Swarm — 专注于多个 Agent 之间的角色定义、任务分配和通信协调。
  • 模型原生 SDK：Anthropic SDK、OpenAI Assistants API — 直接使用模型厂商提供的第一方工具调用和 Agent 能力，抽象层级低，灵活性最高。
  • 低代码/可视化平台：Dify、Coze、Flowise — 通过拖拽界面构建 Agent 工作流，降低开发门槛。
框架选择的本质

选择框架不是在选"谁最强"，而是在选谁的抽象层级和设计哲学最匹配你的项目需求。高抽象框架开发效率高但灵活性受限，低抽象 SDK 灵活但需要更多底层代码。理解这种权衡是高级 Agent 开发者的基本功。

🦜
LangChain 深度解析

LangChain 是目前生态最丰富的 Agent 开发框架，由 Harrison Chase 于 2022 年创立。它提供了从 LLM 调用到复杂 Agent 编排的完整工具链。核心模块包括：

  • Chains（链）：将多个 LLM 调用和工具调用串联成可复用的工作流。LCEL (LangChain Expression Language) 提供了声明式的链定义方式，使用 | 管道符串联组件。
  • Agents（智能体）：在 Chain 之上增加了动态决策能力。Agent 可以自主选择使用哪个工具、以什么顺序调用。LangChain 提供了 OpenAI Functions Agent、ReAct Agent、JSON Chat Agent 等多种 Agent 类型。
  • Tools（工具）：预定义了数百种工具集成，涵盖搜索引擎（Google、Bing）、数据库（SQL、Vector Store）、API 集成（GitHub、Slack、Gmail）等。
  • Memory（记忆）：提供了 ConversationBufferMemory、ConversationSummaryMemory、VectorStoreRetrieverMemory 等多种记忆实现，支持短期和长期记忆。
  • LangGraph：LangChain 生态中专门用于构建有状态、多步骤 Agent的库。将 Agent 逻辑建模为有向图（Graph），节点代表操作，边代表状态转移。支持条件分支、循环和人机交互节点，比传统的 AgentExecutor 提供更精细的控制。
LangChain 的争议

LangChain 以"抽象过多、调试困难"而受到一些开发者的批评。它的版本迭代较快，API 破坏性变更频繁。建议在使用前评估项目的复杂度：简单的 Agent 可能直接用模型 SDK 更合适，复杂工作流则可受益于 LangChain 的编排能力。

Python — LangChain Agent 示例 
from langchain_openai import ChatOpenAI
from langgraph.prebuilt import create_react_agent
from langchain.tools import tool

# 1. 定义工具
@tool
def get_weather(city: str) -> str:
    """获取指定城市的实时天气信息"""
    return f"{city} 当前晴，温度22°C"

@tool
def search_knowledge(query: str) -> str:
    """搜索知识库获取专业信息"""
    return f"关于 '{query}' 的搜索结果: ..."

# 2. 创建模型实例
model = ChatOpenAI(
    model="gpt-4o",
    temperature=0.1
)

# 3. 使用 LangGraph 构建 Agent（支持状态管理）
agent = create_react_agent(
    model=model,
    tools=[get_weather, search_knowledge],
    prompt="你是一个乐于助人的AI助手。"
)

# 4. 执行任务 — LangGraph 支持流式输出和中断
config = {"configurable": {"thread_id": "session-001"}}
for event in agent.stream(
    {"messages": ["北京今天天气怎么样？适合旅游吗？"]},
    config=config
):
    if "agent" in event:
        print(event["agent"]["messages"][-1].content)

CrewAI — 基于角色的多 Agent 协作

CrewAI 的设计灵感来源于公司团队协作模式。它的核心理念是：定义不同的 AI Agent 角色，每个角色拥有特定的目标、背景故事和能力，然后通过"任务"（Task）和"Crew"（团队）来编排它们的工作流程。

  • Agent 角色定义：每个 Agent 有 role（角色名）、goal（目标）、backstory（背景故事）三个核心属性。这些描述会被注入到系统提示词中，使 Agent 表现出符合角色的行为模式。
  • Task 委托机制：任务可以被分配给特定 Agent，也可以由多个 Agent 协作完成。任务支持异步执行、依赖关系和上下文传递。
  • Crew 编排：Crew 定义了 Agent 团队的工作流程：sequential（顺序执行）或 hierarchical（层级调度，由管理者 Agent 分配任务）。
  • 内置工具：集成了搜索、代码解释器、文件读写等常用工具，也支持自定义工具。
  • 适用场景：特别适合内容创作（写手+编辑+出版商）、市场分析（研究员+策略师+文案）、软件开发（需求分析师+架构师+开发者+测试）等需要多角色协作的场景。
Python — CrewAI 多 Agent 协作示例 
from crewai import Agent, Task, Crew, Process

# 1. 定义角色
researcher = Agent(
    role="资深技术研究员",
    goal="深入调研新技术趋势并输出结构化分析",
    backstory="你拥有10年的技术研究经验，擅长快速掌握新技术本质",
    verbose=True
)

writer = Agent(
    role="技术内容创作者",
    goal="将复杂的技术概念转化为清晰易懂的内容",
    backstory="你是一名顶尖技术博客作者，擅长用生动的语言解释技术",
    verbose=True
)

reviewer = Agent(
    role="内容质量审核",
    goal="确保内容准确、全面且结构清晰",
    backstory="你是一名严谨的编辑，对技术细节一丝不苟",
    verbose=True
)

# 2. 定义任务
research_task = Task(
    description="调研 AI Agent 在 2025 年的 5 大技术趋势",
    expected_output="包含5个趋势的详细分析报告，每项不少于200字",
    agent=researcher
)

writing_task = Task(
    description="基于研究报告，撰写一篇面向大众的科普文章",
    expected_output="一篇1500字的科普文章，语言通俗易懂",
    agent=writer
)

review_task = Task(
    description="审核文章的技术准确性和可读性，提供改进建议",
    expected_output="审核报告，列出修改建议和最终评分",
    agent=reviewer
)

# 3. 创建 Crew 并执行
crew = Crew(
    agents=[researcher, writer, reviewer],
    tasks=[research_task, writing_task, review_task],
    process=Process.sequential,  # 顺序执行模式
    verbose=True
)

result = crew.kickoff()
print(result)

AutoGen (Microsoft) — 对话驱动的多 Agent 框架

AutoGen 是微软开源的 Agent 框架，核心理念是"对话即编程" — Agent 之间通过自然的对话来协调和完成任务。其核心概念包括：

  • ConversableAgent：所有 Agent 的基类，拥有发送和接收消息的能力。可以配置不同的 LLM 后端、系统提示词和工具。
  • GroupChat：允许多个 Agent 在一个共享的对话空间中协作。当一条消息发送到群组时，所有 Agent 都能看到，并由 GroupChatManager 根据规则选择下一个发言者。
  • Two-Agent Chat Pattern：最常见的模式是助手 Agent（AssistantAgent）与用户代理（UserProxyAgent）之间的双向对话。UserProxyAgent 可以代表人类执行代码和工具调用。
  • 嵌套对话：一个对话中触发另一个子对话，例如：主对话中的 Agent 可能需要与另一个专门的 Agent 进行子对话来解决特定子问题。
  • 代码执行沙箱：AutoGen 内置了 Docker 沙箱中的代码执行能力，Agent 可以生成代码、执行代码、根据执行结果修正代码，形成自动的代码迭代循环。
  • 核心优势：微软生态原生支持（Azure OpenAI 集成）、对话模式灵活可组合、代码执行能力强大、支持人类介入（Human-in-the-Loop）。
AutoGen vs CrewAI

AutoGen 更强调对话驱动 — Agent 通过聊天来协调，适合需要大量交互和代码执行的场景。CrewAI 更强调角色驱动 — Agent 有明确的角色定义和任务分配，适合结构化的流水线工作。两者可以互补：AutoGen 在灵活性和代码能力上更强，CrewAI 在角色建模和任务编排上更直观。

Anthropic Claude — 模型原生 Agent 能力

Anthropic 的 Claude 系列模型为 Agent 开发提供了一流的原生能力，并通过官方 SDK 提供简洁的 API。核心特性包括：

  • Tool Use (Function Calling)：Claude 原生支持工具调用，可以定义工具的 name、description 和 input_schema。支持并行工具调用（同时调用多个不相互依赖的工具），显著提升效率。
  • Extended Thinking（扩展思考）：Claude 3.5 Sonnet 及以上模型支持"思考模式"，在回答前进行透明的、结构化的多步推理。可以设置 thinking budget（如 16000 tokens）来控制思考深度。思考过程对开发者可见，便于调试优化。
  • Computer Use（计算机操作）：Claude 可以直接操作计算机界面——移动鼠标、点击按钮、截图分析、输入文字。这为构建操作 GUI 应用的 Agent 提供了强大的基础能力。
  • MCP (Model Context Protocol)：Anthropic 提出的开放协议，标准化了模型与外部工具/数据源之间的连接方式。MCP 服务器可以暴露 Resources（数据）、Tools（操作）、Prompts（模板），Client 通过标准协议调用。目前已形成丰富的 MCP 服务生态。
  • Prompt Caching：对重复使用的系统提示词和工具定义进行缓存，大幅降低延迟和成本（最高可节省 90% 的输入 token 成本）。
  • 高安全标准：Claude 在安全对齐方面投入巨大，由 Constitutional AI 方法训练，天然适合需要高安全合规的 Agent 场景。

同一任务的两种实现对比：LangChain vs Anthropic SDK

以下展示用两种框架实现同一个任务——"查询天气并根据结果给用户出行建议"的代码对比：

方案A: Anthropic SDK（直接、类型安全） 
import anthropic

client = anthropic.Anthropic()

# 定义工具
tools = [{
    "name": "get_weather",
    "description": "获取城市天气",
    "input_schema": {
        "type": "object",
        "properties": {
            "city": {"type": "string"}
        },
        "required": ["city"]
    }
}]

# 执行 Agent 循环
messages = [{"role": "user", "content": "北京天气如何？"}]

for _ in range(5):  # 最大循环 5 次
    response = client.messages.create(
        model="claude-sonnet-4-20250514",
        max_tokens=1024,
        system="你是天气助手，根据天气给出行建议。",
        messages=messages,
        tools=tools
    )

    if response.stop_reason == "end_turn":
        print(response.content[0].text)
        break

    # 处理工具调用
    for block in response.content:
        if block.type == "tool_use":
            result = execute_tool(block.name, block.input)
            messages.append({
                "role": "user",
                "content": [{"type": "tool_result",
                               "tool_use_id": block.id,
                               "content": result}]
            })
方案B: LangChain + LangGraph（抽象、可复用） 
from langgraph.prebuilt import create_react_agent
from langchain_anthropic import ChatAnthropic
from langchain.tools import tool

@tool
def get_weather(city: str) -> str:
    """获取城市天气"""
    return f"{city}: 晴，22°C"

model = ChatAnthropic(
    model="claude-sonnet-4-20250514"
)

agent = create_react_agent(
    model=model,
    tools=[get_weather],
    prompt="你是天气助手，根据天气给出行建议。"
)

# 一行代码触发完整 Agent 循环
result = agent.invoke(
    {"messages": ["北京天气如何？"]}
)
两种方案的特点对比

Anthropic SDK：代码量稍多但逻辑完全透明，你可以精确控制每一步。适合需要精细控制、高性能、低依赖的场景。
LangChain + LangGraph：代码量少，内置会话管理、状态持久化，适合快速开发和复杂工作流。但调试和理解内部行为需要额外学习成本。

如何选择适合你的框架

选择 Agent 框架需要综合考虑以下因素。以下是一个结构化的决策框架：

  1. 评估项目复杂度
    • 简单 Agent（1-2个工具、简单循环） → 直接用 Anthropic SDK 或 OpenAI SDK
    • 中等复杂度（多工具、RAG、记忆系统） → LangChain 或 Semantic Kernel
    • 高复杂度（多 Agent 协作、复杂状态管理） → LangGraph + LangChain 或 AutoGen
  2. 考虑团队技术栈
    • Python 为主 → LangChain、CrewAI、AutoGen
    • .NET / C# → Semantic Kernel（微软官方支持）
    • TypeScript / Node.js → LangChain.js 或 Vercel AI SDK
  3. 模型锁定策略
    • 深度绑定 Anthropic → Anthropic SDK（利用 Tool Use、Extended Thinking 等独特特性）
    • 深度绑定 OpenAI → OpenAI Assistants API（最省心，但灵活性受限）
    • 模型无关/多模型切换 → LangChain（统一的模型接口）
  4. 安全与合规要求
    • 金融/医疗等高合规场景 → Anthropic Claude（Constitutional AI 安全对齐）+ 自建编排层
    • 通用场景 → LangChain（内置输入/输出安全过滤器）
  5. 不要过度设计
    • "框架饥渴症"是常见陷阱 — 花三周选框架不如花三天用 SDK 写原型
    • 可以先从 Anthropic SDK 或 OpenAI SDK 做原型验证，确认需求后再引入 LangChain 等框架来提升可维护性
实践建议

起步建议：先用 Anthropic SDK 或 OpenAI SDK 写一个最小可行的 Agent 原型（不超过200行代码）。这个过程会让你深刻理解 Agent 循环的本质。之后再用框架重构，你会更清楚什么抽象是你需要的，什么是多余的。

Semantic Kernel (Microsoft) — 企业级 AI 编排 SDK

Semantic Kernel (SK) 是微软推出的轻量级 AI 编排 SDK，与 LangChain 定位相似但设计哲学不同：

  • 多语言支持：原生支持 C#、Python 和 Java，对 .NET 生态的企业用户特别友好。
  • Plugin 架构：将功能封装为"插件"（Plugin），分为 Native Plugin（原生代码函数）和 Semantic Plugin（基于提示词模板）。插件可以被 Planner 自动编排调用。
  • Planner：类似于 LangChain 的 Agent，但更轻量化。从用户的自然语言目标自动生成执行计划，选择要调用的插件和参数。
  • Process Framework：SK 最近引入的流程编排能力，允许定义带有分支、循环、条件逻辑的业务流程，这些流程可以调用 AI 步骤作为其中一环。
  • 与 Azure 生态深度集成：天然支持 Azure OpenAI、Azure AI Search、Azure Cosmos DB 等服务，适合 Azure 云用户。
  • 适用场景：企业内部工具集成（通过 Plugin 模型轻松连接现有的企业 API）、.NET/C# 项目、需要将 AI 调用嵌入现有业务流程的场景。
🤖
OpenAI Assistants API — 零运维的托管 Agent 服务

OpenAI 的 Assistants API 是一种全托管的 Agent 服务，你只需定义 Assistants 的配置，OpenAI 负责 Agent 循环、工具执行和状态管理：

  • 核心抽象：Assistant（助手配置：指令+模型+工具）、Thread（对话线程：自动管理上下文窗口）、Run（一次执行：OpenAI 自动处理 Agent 循环）。
  • 内置工具：Code Interpreter（代码解释器，沙箱执行 Python）、File Search（文件搜索，基于向量检索）、Function Calling（自定义函数调用）。
  • 流式支持：支持 Server-Sent Events 实时推送 Agent 的执行状态和中间步骤。
  • 优势：零运维、无需手动管理 Agent 循环和上下文窗口、OpenAI 持续优化底层模型和 Agent 执行策略。
  • 局限：仅支持 OpenAI 模型、工具执行逻辑受限于 OpenAI 平台提供的框架、自定义程度有限、成本不可控（Agent 循环次数由 OpenAI 决定）。
选择提醒

Assistants API 非常适合快速验证想法和构建原型，但生产环境中需要评估：你是否愿意将 Agent 执行逻辑的控制权完全交给 OpenAI？如果你的回答是"不确定"，那么建议使用更灵活的 SDK + 自建编排层方案。
