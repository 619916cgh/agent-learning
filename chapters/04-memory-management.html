第 4 章

记忆与状态管理

深入掌握三层记忆架构、RAG 检索增强生成管线、向量数据库选型与上下文窗口精细化管理

1. 三层记忆架构：Agent 认知的基石

三层记忆架构受认知科学启发，Agent 记忆系统分为三层：短期记忆(STM)承载当前对话上下文，受 LLM 上下文窗口限制；工作记忆(WM)维护任务执行状态和中间结果，由编排器管理；长期记忆(LTM)基于向量数据库持久化存储跨会话知识。三层各司其职，通过信息流动保持协同。是 Agent 信息管理系统的核心框架。人类的记忆并非单一的"硬盘"——我们有瞬时感知的感官记忆、正在处理的"思维便签"工作记忆、以及跨越大脑皮层终生存留的长期记忆。Agent 的记忆系统借鉴了同样的分层设计：

L1 — 短期记忆 (Short-Term Memory)
定位：当前对话的实时上下文，是 LLM 直接"看见"的信息。数据形态：消息列表 (messages array)，每条含 role/content。容量：严格受限于 LLM 上下文窗口 (4K-1M tokens)。管理：滑动窗口淘汰 + 智能摘要压缩 + 重要性剪枝。生命周期：单次会话。
L2 — 工作记忆 (Working Memory)
定位：任务执行过程中的结构化状态追踪。数据形态：结构化 JSON——任务状态机、规划 DAG、中间结果缓存。核心组件：任务阶段追踪、便签区 (Scratchpad)、事实缓存 (Fact Cache)、执行追踪 (Plan Trace)。生命周期：跨多轮 LLM 调用，任务完成后归档或丢弃。
L3 — 长期记忆 (Long-Term Memory)
定位：跨越会话和任务的持久化知识库。数据形态：文本块 + 向量嵌入，存储在向量数据库中。典型内容：用户偏好、历史任务经验、领域知识文档的语义索引、成功案例的执行方案。管理：语义检索 + 多因子评分(新鲜度/重要性/频率) + 记忆压缩淘汰。生命周期：永久。

三层间的数据流动

在一次典型的 Agent 任务中，记忆系统按以下顺序运转：(1) 用户输入驱动长期记忆进行语义检索，召回相关知识；(2) 工作记忆加载当前任务状态；(3) 两者与短期记忆(对话历史)融合为完整的 LLM 上下文；(4) LLM 推理并执行工具调用；(5) 结果更新工作记忆中的任务状态和中间结果；(6) 重要发现异步写入长期记忆；(7) 短期记忆通过滑动/压缩机制管理 token 预算。

核心洞察

三层记忆架构的本质是成本-能力的动态平衡。短期记忆最"贵"(token 即金钱，也是延迟)，长期记忆最"便宜"(磁盘存储成本几乎可忽略)，工作记忆居中。优秀的设计懂得：在正确的时间把正确的信息放到正确的层级，而非盲目追求"所有信息都在线"。

2. 短期记忆精要：上下文窗口的精细化管理

短期记忆的管理直接决定 Agent 在多轮交互中的稳定性。现代 LLM 的上下文窗口可达 128K-1M tokens，表面上充裕，但在经过多轮复杂任务后——系统提示词(500-3000 tokens)、工具 Schema(200-2000 tokens)、多轮对话历史(累积数千至数万 tokens)、工具调用返回结果(单次可达数千 tokens)、RAG 检索注入的文档片段(500-4000 tokens)——窗口很容易被填满。此时若不主动管理，Agent 将出现"记忆力衰退"：遗忘早期对话中的关键约束和决策。

三种管理策略的对比

策略原理优势劣势适用场景
滑动窗口 只保留最近 N 条消息 实现极简，无额外 LLM 调用 可能丢失关键早期信息 时效性强的任务(客服、实时问答)
智能摘要 对旧消息生成摘要替代原文(压缩比 5:1~20:1) 信息密度高，不轻易丢失关键内容 需要额外 LLM 调用生成摘要 长对话、复杂多轮任务
重要性剪枝 对每条消息评重要性分，淘汰低分内容 精细控制，保留高价值情报 评分机制本身有开销和误差 异构对话(重要约束+大量闲聊混合)
窗口错觉：大不等于好

即使有 1M token 的窗口，大量低质量内容也会稀释 LLM 的注意力——这就是"大海捞针"(Needle in a Haystack) 问题的本质。LLM 对长文本中后部内容的关注度会自然衰减。主动管理短期记忆不仅是为了"节省空间"，更是为了提高 LLM 从上下文中提取关键信息的效率。

3. 工作记忆：任务执行的"方向盘"

工作记忆是三层架构中最容易被忽视但最关键的环节。它不像短期记忆那样直接受限于 token 窗口，也不像长期记忆那样依赖复杂的向量检索——它的核心使命是：让 Agent 在跨越数十轮的复杂任务中始终知道"我是谁、我在哪、我要去哪"。

工作记忆的四大组件

组件职责数据形态示例
任务状态机 追踪当前执行到哪个阶段、哪一步 枚举 + 计数 {"phase": "execution", "current_step": 4, "total_steps": 10}
便签区 (Scratchpad) 存放中间计算结果和临时变量 Key-Value 字典 {"beijing_temp": 22, "shanghai_temp": 25, "temp_diff": 3}
事实缓存 已验证的事实，避免重复查询 查询→结果 映射表 "weather_2026-05-10_BJ" → {"temp":22,"condition":"晴"}
规划追踪 (Plan Trace) 记录当前执行计划的每一步状态 DAG + 状态标记 步骤1-3已完成(绿)，步骤4进行中(黄)，步骤5-10待执行(灰)

注入策略：工作记忆不是"全部塞入"

工作记忆的内容不应全量灌入 LLM 上下文。正确做法是通过模板化的"状态摘要"将最相关的部分注入——通常仅 50-200 tokens。例如在步骤 5 执行前，摘要包含：(1) 前 4 步各自做了什么、结果如何；(2) 当前步骤需要的前置数据；(3) 步骤在整体计划中的位置。这好比汽车仪表盘——你不需要看到引擎内部每个零件，只需要速度、油量和故障灯。

跨任务上下文切换

同一次会话中用户可能连续提出多个任务。工作记忆需支持类似操作系统的"上下文切换"：(1) 将当前任务状态序列化保存到任务栈；(2) 初始化新任务的干净状态；(3) 用户切回旧任务时从栈中恢复。这是被大多数 Agent 框架忽视的关键能力。

设计原则

工作记忆的内容应该结构化且可查询。不要把所有信息揉成一个长字符串——用明确的 JSON Schema 定义结构，让任何组件都能高效读写它所关心的部分。结构化的另一大好处是可调试：你能随时查看 Agent 的"大脑状态"，快速定位"Agent 为什么做了那个决策"。

4. 长期记忆：向量嵌入与向量数据库深度对比

长期记忆使 Agent 拥有"跨会话的知识"，是区分高级 Agent 和普通聊天机器人的核心能力。向量嵌入 (Embeddings)将文本等非结构化数据映射到高维向量空间(通常 768-3072 维)的技术。核心属性：语义相近的内容在向量空间中距离近，语义无关的则远。这使得"语义搜索"成为可能——通过计算向量间余弦相似度找到相关内容，无需关键词匹配。主流模型包括 OpenAI text-embedding-3、Cohere Embed v3、BGE-large-zh 等。是这一能力的技术支柱。

ANN 索引算法：HNSW vs IVF vs PQ

  • HNSW (Hierarchical Navigable Small World)：构建多层图结构，搜索从顶层粗粒度逐层下降到底层细粒度。查询延迟 1-10ms，是多数向量数据库的默认选择，但内存消耗较高。
  • IVF (Inverted File Index)：用 K-Means 将向量空间划分为多个聚类，查询时仅在最近聚类中搜索。内存友好，适合内存受限场景，但精度略低于 HNSW。
  • PQ (Product Quantization)：高维向量分解为多个低维子向量并量化压缩。大幅减少内存(压缩比可达 10:1-30:1)，代价是轻微精度损失。适合十亿级别的大规模场景。
选型决策

原型阶段用 ChromaDB(简单快速)；中等生产用 Weaviate 或 Qdrant(混合搜索和高性能)；已有 PG 基础设施用 PGVector(减少运维)；十亿级和大规模生产用 Pinecone 或 Milvus。中文场景强烈推荐 BGE-large-zh-v1.5 作为嵌入模型。

5. RAG 完整管线：从文档到答案的七步解析

RAG (Retrieval-Augmented Generation)由 Lewis et al.(2020) 提出。在 LLM 生成回答前，从外部知识库检索相关信息并注入上下文。核心管线：文档加载→文本分块→向量化→索引存储→语义检索→重排序→上下文注入。RAG 大幅降低了幻觉，因为 LLM 可以"参考"知识库中的真实信息而非依赖其参数量中的不确定记忆。是连接长期记忆和 LLM 推理的核心管线。理解每个环节才能在检索质量出问题时准确定位根因。

1
文档加载 (Document Loading)
从 PDF、Markdown、网页、Notion、Confluence 等多源加载。关键难点：PDF 表格解析(容易乱码/断裂)。推荐工具：Unstructured.io、LlamaParse、PyMuPDF。
2
文本分块 (Chunking)
将长文档切分。固定大小(512-1024 tokens, 10-15% overlap)最简单；语义分块按段落/句子边界切分质量更高；递归分块兼顾效率和质量；层级分块(父块+子块)平衡精度和上下文完整性。
3
向量化 (Embedding)
批量调用嵌入模型将文本块转为向量。这是管线中成本最高的批量操作——对于百万级文档需并行处理。关键优化：嵌入缓存(相同文本 SHA256 哈希去重，避免重复 API 调用)。
4
索引存储 (Indexing)
向量存入向量数据库并构建 ANN 索引(如 HNSW)。索引构建是一次性成本，品质决定后续所有检索的速度和精度。同时存入元数据(来源、时间、标签)便于过滤。
5
语义检索 (Retrieval)
用户查询向量化后在向量库中搜索 top-K(通常 K=5-20)。混合检索(向量+BM25关键词)在大多数场景优于纯向量检索。元数据过滤(如"仅搜索最近30天文档")是提精利器。
6
重排序 (Reranking)
向量相似度不等于语义相关性！重排序模型(Cohere Rerank / bge-reranker)对上一步的 top-K 精细再排序，保留 top-3~5 注入上下文。这是性价比最高的精度提升手段。
7
上下文注入 (Context Injection)
将最终选定的文档片段格式化，标注来源后注入 LLM 上下文。Prompts 模板化："请基于[参考资料]回答。若信息不足请明确说明，不要编造。"

分块策略深度对比

策略方法优势劣势

RAG 三大失败模式
  1. "看到了但没理解"：相关文档被检索到并注入上下文，但 LLM 未正确使用信息——通常是表面相似，语义不匹配。解法：更好的重排序。
  2. "应该找到但没找到"：答案在知识库中但未检索到。根因：chunk 不合理(关键信息被拆分)、嵌入模型对领域术语理解不足、查询改写不到位。
  3. "找到了不该找的"：检索到表面相似但无关的内容。纯向量相似度检索容易遇此问题，混合检索+重排序是关键解法。

6. 记忆检索策略：在海量记忆中精准调取

当长期记忆中积累了成千上万条记忆后，检索什么和检索多少就成了影响 Agent 表现的核心变量。检索太少→Agent 缺乏上下文；检索太多→噪声淹没信号+浪费 token。

多因子加权评分机制

评分维度含义典型实现
语义相关性 (Relevance) 记忆内容与当前查询的语义匹配度 向量余弦相似度 + 重排序模型打分
新鲜度 (Recency) 离现在越近的记忆默认越相关 指数衰减：score = base * e(-lambda * days)
重要性 (Importance) 该记忆被标记或推断的重要程度 LLM 打分 1-10，或从用户行为中推断
使用频率 (Frequency) 该记忆被检索和使用过的次数 访问计数器 + 时间归一化
来源权威性 (Authority) 记忆来源的可信度 人工标注权重(官方文档 > 社区帖子)

综合得分 = w1*相关性 + w2*新鲜度 + w3*重要性 + w4*频率 + w5*权威性。权重系数根据任务类型动态调整——事实查询侧重于相关性，个性化推荐侧重于新鲜度和重要性。

查询改写 (Query Rewriting) — 检索质量的隐形杠杆

  • 指代消解：将"那个"、"它"、"上次的方案"等指代词替换为具体实体名称。需注入短期记忆中的上下文。
  • 多角度查询生成：将一个用户查询改写为 2-3 个不同表述，分别检索后合并结果去重。例："Agent 如何管理记忆"改写为"AI Agent 记忆管理最佳实践"和"LLM 上下文窗口 管理策略"。
  • HyDE (Hypothetical Document Embeddings)：让 LLM 先生成假设的理想答案，用假想答案(而非原始查询)做向量检索。原理：假想答案在语义空间中更接近真实文档。

混合检索 (Hybrid Search)

向量检索擅长语义理解但对精确匹配(产品编号、API 名称)无能为力。混合检索 = 向量检索 + 关键词检索(BM25)，结果加权融合排序。这是大多数生产场景的最优选择——兼具语义广度与精确匹配深度。

实践路径

起步阶段不必过度追求复杂策略。先做好三件事：(1) 合理的语义分块 + overlap；(2) 混合检索(向量+BM25)；(3) 重排序。这三者即可将检索质量提升到可用水平。之后再逐步引入加权评分、查询改写等高级技巧。

7. 记忆压缩与淘汰：在有限空间中保持无限智慧

记忆不会无限增长——磁盘空间有限，检索速度随数据量增加而下降。记忆压缩和淘汰机制是维持系统长期健康的关键。

记忆压缩策略

  • 周期性摘要压缩：定时任务对新增记忆聚类→每类生成摘要→用摘要替换原始条目。压缩比可达 10:1~50:1。例：用户一天中 20 次提到"天气"，压缩为"用户频繁关注北京和上海天气，偏好当天实时数据"。
  • 层级化压缩：三层存储——热数据(7天内，原始记忆)、温数据(7-30天，摘要)、冷数据(30天以上，元摘要)。时间越远越概括。
  • 实体融合：同实体多条相关记忆合并。如"用户喜欢蓝色"+"用户偏好深色主题"融合为"用户偏好：颜色方面倾向蓝色和深色系"。

时间衰减淘汰 存储空间 > 80% 按复合得分排序(新鲜度权重加倍)，淘汰末尾 10-20%
重复淘汰 检测到语义高度相似的记忆 嵌入余弦相似度 > 0.95 视为重复，保留最新一条
矛盾裁决 检测到相互冲突的记忆 标记冲突→LLM 裁决→被否决方标记为"过时"并降权
用户显式遗忘 用户请求删除 支持"忘掉 X"和"忘掉关于 X 的全部"两种粒度
压缩的代价

压缩不可避免地丢失细节。好的做法：压缩时生成"可追溯标记"——当 LLM 发现摘要信息不足以回答问题时，可"展开"摘要回到原始记忆。类似 Git——日常工作看 commit message(摘要)，需要时 git diff 查看完整改动。

8. 代码实战：完整的三层记忆系统实现

以下是可直接运行的生产级记忆系统实现：滑动窗口+摘要压缩的短期记忆、结构化状态追踪的工作记忆、ChromaDB 驱动的 RAG 长期记忆，以及统一的三层编排器。

Python — 三层记忆系统完整实现 
import json, time, hashlib, logging
from dataclasses import dataclass, field
from typing import Any, Optional
from collections import deque
import numpy as np
import chromadb
from chromadb.utils import embedding_functions

logging.basicConfig(level=logging.INFO)
logger = logging.getLogger("MemorySystem")


# ============================================================
# 1. 短期记忆：滑动窗口 + 摘要压缩
# ============================================================

class ShortTermMemory:
    """管理 LLM 上下文窗口中的对话历史"""

    def __init__(self, max_tokens: int = 8000, summary_threshold: float = 0.7):
        self.messages: deque[dict] = deque(maxlen=200)
        self.max_tokens = max_tokens
        self.summary_threshold = summary_threshold  # 窗口占用 70% 时触发压缩
        self.summary: str = ""

    def add(self, role: str, content: str, importance: float = 0.5):
        msg = {"role": role, "content": content,
               "importance": importance, "tokens": self._est_tokens(content)}
        self.messages.append(msg)
        self._maybe_compress()

    def get_context(self) -> list[dict]:
        llm_messages = []
        if self.summary:
            llm_messages.append({"role": "system", "content": self.summary})
        for m in self.messages:
            llm_messages.append({"role": m["role"], "content": m["content"]})
        return llm_messages

    def _maybe_compress(self):
        total = sum(m["tokens"] for m in self.messages)
        if total > self.max_tokens * self.summary_threshold:
            # 压缩最旧的 30% 消息为摘要
            cutoff = int(len(self.messages) * 0.3)
            old = [self.messages.popleft() for _ in range(cutoff)]
            # 实际应用需调 LLM 生成高质量摘要
            self.summary = f"[历史摘要({len(old)}条消息)] " + self.summary

    def _est_tokens(self, text: str) -> int:
        chinese = sum(1 for c in text if '一' <= c <= '鿿')
        return int(chinese * 0.4 + (len(text) - chinese) * 0.3)


# ============================================================
# 2. 工作记忆：任务状态 + 便签区 + 事实缓存
# ============================================================

class WorkingMemory:
    """任务执行过程中结构化状态追踪"""

    def __init__(self):
        self.task_state: dict = {}        # 当前任务状态机
        self.scratchpad: dict = {}        # 临时便签
        self.fact_cache: dict = {}        # 已验证事实
        self.plan_trace: list = []         # 执行计划追踪
        self.task_stack: list = []        # 多任务上下文栈

    def start_task(self, task_id: str, goal: str, plan: list):
        if self.task_state:
            self._push_current_task()
        self.task_state = {"task_id": task_id, "goal": goal,
                            "phase": "planning", "current_step": 0}
        self.plan_trace = [{"step": i, "action": s, "status": "pending"}
                          for i, s in enumerate(plan)]
        self.scratchpad = {}

    def step_complete(self, idx: int, result: Any):
        self.plan_trace[idx]["status"] = "completed"
        self.plan_trace[idx]["result"] = result
        self.task_state["current_step"] = idx + 1

    def get_state_summary(self) -> str:
        if not self.task_state:
            return ""
        completed = sum(1 for p in self.plan_trace if p["status"] == "completed")
        return f"[任务: {self.task_state['goal']} | 进度: {completed}/{len(self.plan_trace)}步]"

    def _push_current_task(self):
        self.task_stack.append({
            "state": self.task_state, "scratchpad": self.scratchpad.copy(),
            "plan": self.plan_trace, "facts": self.fact_cache.copy()
        })


# ============================================================
# 3. 长期记忆：ChromaDB RAG 管线
# ============================================================

class LongTermMemory:
    """基于向量数据库的长期记忆 + RAG 检索"""

    def __init__(self, collection: str = "agent_memory"):
        self.client = chromadb.PersistentClient(path="./memory_db")
        self.ef = embedding_functions.SentenceTransformerEmbeddingFunction(
            model_name="BAAI/bge-large-zh-v1.5"
        )
        self.col = self.client.get_or_create_collection(
            name=collection, embedding_function=self.ef
        )

    def store(self, content: str, meta: dict = None, importance: float = 0.5) -> str:
        mid = hashlib.md5(content.encode()).hexdigest()[:16]
        m = meta or {}
        m.update({"importance": importance, "created_at": time.time()})
        self.col.upsert(ids=[mid], documents=[content], metadatas=[m])
        return mid

    def retrieve(self, query: str, top_k: int = 5) -> list[dict]:
        # 向量检索(取2倍候选用于重排序)
        raw = self.col.query(query_texts=[query], n_results=top_k * 2)
        entries = []
        for i, doc in enumerate(raw["documents"][0]):
            meta = raw["metadatas"][0][i]
            age = (time.time() - meta.get("created_at", time.time())) / 86400
            recency = np.exp(-0.1 * age)
            importance = meta.get("importance", 0.5)
            score = 0.5 * 1.0 + 0.25 * recency + 0.25 * importance
            entries.append({"content": doc, "_score": score})
        entries.sort(key=lambda x: x["_score"], reverse=True)
        return entries[:top_k]


# ============================================================
# 4. MemorySystem — 三层统一编排器
# ============================================================

class MemorySystem:
    """协调三层记忆，生成完整 LLM 上下文"""

    def __init__(self, stm_tokens: int = 8000):
        self.short_term = ShortTermMemory(max_tokens=stm_tokens)
        self.working = WorkingMemory()
        self.long_term = LongTermMemory()

    def build_context(self, user_input: str) -> list[dict]:
        ctx = []
        # 长期记忆检索
        memories = self.long_term.retrieve(user_input, top_k=3)
        if memories:
            ltm = "\n".join([f"- {m['content']}" for m in memories])
            ctx.append({"role": "system", "content": f"[长期记忆]\n{ltm}"})
        # 工作记忆状态
        s = self.working.get_state_summary()
        if s:
            ctx.append({"role": "system", "content": f"[工作记忆] {s}"})
        # 短期记忆(对话历史)
        ctx.extend(self.short_term.get_context())
        return ctx

代码设计要点

  • ShortTermMemory：滑动窗口 deque + 自动摘要压缩(超阈值时压缩最早 30% 消息) + 中英文区分 token 估算。摘要阈值设为窗口 70%，给摘要过程预留操作空间。
  • WorkingMemory：支持多任务上下文切换(task_stack 推入/恢复)、每一步状态追踪、50-200 token 精简摘要生成。
  • LongTermMemory：ChromaDB+BGE 模型、2倍候选取回+多因子重排序(新鲜度衰减+重要性加权)、sha256 内容去重避免重复存储。
  • MemorySystem：build_context() 按长期→工作→短期有序编排 LLM 上下文，每层信息清晰分隔便于 LLM 理解来源。

9. 记忆缓存与性能优化策略

在频繁的 LLM 调用中，记忆系统的 I/O 操作可能成为瓶颈。合理的多层缓存大幅降低延迟和成本。

缓存层位置延迟容量内容
L1: 查询缓存进程内存 / Redis微秒级数百条重复查询的检索结果(精确匹配或语义相似>0.95视为命中)
L2: 嵌入缓存本地 SQLite / Redis毫秒级数万条已计算过的文本嵌入向量(SHA256 哈希索引，避免重复 API 调用)
L3: 磁盘持久化向量数据库数毫秒~数十毫秒百万~十亿级完整长期记忆(最终数据源)

性能优化点

  • 批量嵌入：将多个待嵌入文本打包为批次(32-128条)，一次 API 调用处理，单条延迟降低 5-10 倍。
  • 增量索引：文档更新时仅重新嵌入变化的 chunk，基于文档版本号追踪变更，而非全量重建索引。
  • 缓存 TTL：事实类查询(如天气)TTL 短(数分钟)，知识类查询(如技术文档)TTL 长(数天甚至不限)。
  • 异步写入：长期记忆写入采用异步队列，避免阻塞主 Agent 循环。新记忆先入队列，后台线程批量写入向量数据库。
性能指标参考

优化良好的记忆系统应达到：检索 P95 延迟 < 50ms、嵌入批处理吞吐 > 100 条/秒、嵌入缓存命中率 > 60%(在重复文档内容场景下)、索引构建速度 > 1000 条/秒。
