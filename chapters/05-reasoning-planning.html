第 5 章

推理与规划

深入掌握AI Agent的多种推理策略——从ReAct循环到思维树，学会为不同任务选择最佳思考方式

1. ReAct模式：思想与行动的交响

ReAct (Reasoning + Acting)由Yao et al.在2023年提出，ReAct将推理(Thought)和行动(Action)交替进行。模型在每一步先输出推理轨迹，再决定执行什么工具操作，最后观察结果。这个循环使Agent具备可解释的多步推理能力。是目前最经典也是应用最广泛的Agent推理范式。它的核心思想极其优雅：让模型在思考中行动，在行动中思考。不同于传统的"一次性回答"模式，ReAct将问题求解展开为一个逐步推进的过程：

  • Thought (思考)：模型分析当前状态和已有信息，判断下一步应该做什么。思考是纯内部推理，不产生外部副作用。例如："用户想查询北京的天气，我需要调用 weather API。"
  • Action (行动)：模型根据思考结论，选择并调用具体工具。行动输出是结构化的函数调用，包含工具名称和参数。例如：get_weather(city="Beijing")。
  • Observation (观察)：工具执行完毕后，Agent收到外部环境的反馈。这个观察结果被追加到上下文中，作为下一轮思考的依据。例如：{"temp": 22, "condition": "晴"}。

这三步循环形成了一个自我纠错的闭环：如果行动结果不符合预期，Agent可以在下一轮思考中调整策略，重新选择工具或修改参数。这就是ReAct模式相比单轮推理的巨大优势——通过反复感知环境来修正推理路径。

核心洞察

ReAct的本质是将LLM的推理能力与外部工具的执行能力通过"观察-反馈"回路耦合在一起。思考没有工具执行就会变成空中楼阁，工具执行没有思考指导就会变成盲目试错。

2. ReAct执行轨迹：逐步剖析一个完整案例

仅理解概念是不够的——让我们通过一个真实任务的完整执行轨迹，窥探ReAct在每一步究竟发生了什么。

任务："帮我查一下北京和上海明天的天气，如果北京下雨就提醒我带伞，并帮我规划一个适合天气的户外活动。"

步骤 阶段 内容
1 Thought 用户需要两个城市的天气信息。我需要先并行获取两地的天气数据，然后再基于结果做判断。先查北京。
1 Action get_weather(city="北京", date="tomorrow")
1 Observation { city: "北京", temp: 18, condition: "小雨", humidity: 85% }
2 Thought 北京明天有小雨。现在查询上海的天气，然后综合判断。
2 Action get_weather(city="上海", date="tomorrow")
2 Observation { city: "上海", temp: 25, condition: "多云", humidity: 55% }
3 Thought 北京下雨（需要带伞），上海多云（适合户外活动）。我已有足够信息回答用户的问题了。北京建议带伞并选择室内活动，上海可以推荐户外活动如公园散步。
3 Final 最终回答：北京明天有小雨(18°C)，记得带伞！建议安排室内活动如博物馆参观。上海明天多云(25°C)，适合户外活动，推荐世纪公园散步或滨江骑行。

注意这个轨迹中几个关键点：(1) Agent没有一次性查询两个城市，而是在第一轮观察后再决定第二步——这不是低效，而是条件依赖的体现；(2) 思考步骤包含了推理过程，使得Agent的行为可解释、可调试；(3) 最终回答综合考虑了两个数据源，展现了多步信息融合的能力。

3. Chain-of-Thought：让模型"大声思考"

思维链 (Chain-of-Thought, CoT)由Wei et al.(2022)提出，通过让模型生成中间推理步骤来提升复杂问题的解决能力。CoT不需要模型训练，仅通过prompt设计即可触发。其核心假设是：将复杂问题分解为中间步骤，每个步骤都更简单可解。是ReAct的推理基础——没有CoT，ReAct中的"Thought"就失去了内涵。CoT有三种主要形式：

3.1 Zero-shot CoT

最简单的形式：在prompt末尾加上一句"Let's think step by step"（让我们一步一步思考）。这神奇地触发模型展开推理过程。研究表明，仅这一句话就能在数学推理任务上提升10-30%的准确率。其原理是：这句话改变了模型的解码路径，使其更倾向于生成结构化、分步骤的输出。

3.2 Few-shot CoT

提供2-5个带有完整推理过程的示例，模型会模仿这种推理模式。关键设计原则：(1) 示例要覆盖不同的推理类型；(2) 每个步骤的粒度要适中——太细碎则冗余，太跳跃则不可跟随；(3) 示例中要包含错误修正的场景，教会模型"犯错后如何自我纠正"。

3.3 Automatic CoT (Auto-CoT)

手动编写Few-shot示例费时费力。Auto-CoT自动化地：(1) 将问题聚类为不同类型；(2) 从每个聚类中选择代表性样本；(3) 使用Zero-shot CoT为这些样本自动生成推理链；(4) 将生成的推理链作为Few-shot示例。这种方法将人工构建成本降至零，同时保持接近手动Few-shot的性能。

实战技巧

在Agent系统的System Prompt中嵌入CoT指令（如"在每次做决策前，先分析当前状态、已有信息和可能的行动方案"）比每次在user message中写"think step by step"更稳定可靠。

4. Plan-and-Execute：先谋后动

当任务涉及十几个甚至几十个步骤时，ReAct的"边走边看"策略就会暴露问题：容易在局部迷失方向，缺乏全局视野。Plan-and-Execute模式将规划和执行解耦，形成两阶段架构：

阶段一：规划 (Planning)

  • 任务分析：理解用户意图、确定成功标准和约束条件
  • 步骤拆解：将任务递归分解为子任务，构建依赖关系图(DAG)
  • 资源预估：每个步骤需要哪些工具、权限和数据
  • 计划输出：生成结构化的执行计划，包含步骤顺序、预期输入输出和验证条件

阶段二：执行 (Execution)

  • 顺序执行：按照计划的拓扑顺序逐步执行每一步
  • 中间验证：每步执行后检查输出是否符合预期，不符合则触发重新规划
  • 动态调整：当环境变化或新信息出现时，更新剩余计划
  • 结果汇总：收集所有步骤的输出，生成最终结果

Replanning触发条件——何时需要重新规划？

触发条件 说明 策略
工具执行失败 API返回错误、超时或权限拒绝 重试3次后降级处理或更换工具
结果意外 输出格式/内容与预期严重不符 检查假设是否正确，调整参数
依赖断裂 某步骤的输出无法满足后续步骤的输入要求 插入补充步骤或修改下游计划
目标变更 用户在中途提出新需求或修改目标 全面重新规划，保留已完成步骤的结果

5. Tree of Thoughts：在可能性森林中寻路

思维树 (Tree of Thoughts, ToT)由Yao et al.(2023)提出，将推理过程建模为树形结构。在每个推理节点，模型生成多个候选的"下一步思考"，然后通过评估函数筛选最有希望的路径继续探索。ToT支持BFS和DFS两种搜索策略，在需要探索性推理的任务上显著优于链式推理。将线性推理升级为树形探索。如果说CoT是一条思维的"直线"，ToT就是一棵枝繁叶茂的"思维树"——在每个推理节点，模型同时探索多条可能的路径，评估后剪枝，保留最有希望的分支继续深入。

ToT的四个核心操作：

  • 分解 (Decomposition)：将问题拆解为中间步骤。对于"24点游戏"(用4个数字通过加减乘除得到24)，每一步就是选两个数做运算。
  • 生成 (Generation)：在每个节点，LLM生成k个候选的下一步（通常是5-10个）。
  • 评估 (Evaluation)：对每个候选进行打分。有两种方法：(a) 让LLM直接评估"这个状态离目标有多远"；(b) 模拟前瞻——从该节点向前模拟几步看能否成功。
  • 搜索 (Search)：使用BFS或DFS遍历树结构，保留得分最高的b个分支（beam size，通常为3-5）。

BFS vs DFS 搜索策略对比：

维度 BFS (广度优先) DFS (深度优先)
探索模式 逐层展开，每层保留top-b个节点 沿一条路径深入到底，再回溯
LLM调用量 高（每层k*b次生成+b次评估） 较低（只深入当前最佳路径）
适用的任务 全局最优重要、并行度高的任务 路径深度大、有明确目标状态的任务
风险 API成本高，token消耗大 可能陷入局部最优，错过更好的全局解

剪枝策略：剪枝是ToT效率的关键。常见策略包括：(1) 阈值剪枝：得分低于某一阈值的分支直接丢弃；(2) 相对剪枝：只保留得分前b名的分支（beam search）；(3) 置信度剪枝：当某个分支的评估置信度很低时提前终止；(4) 前瞻剪枝：快速模拟一到两步，如果模拟结果差则剪枝。

适用场景

ToT特别适合需要探索和规划的推理任务：数学证明、逻辑谜题、创意写作（评估多个故事发展方向）、代码生成（探索不同的架构方案）。对于简单的检索或计算任务，ToT的额外开销是不必要的。

6. Self-Reflection：让Agent学会自省

人类专家之所以高效，很大程度归功于元认知能力——能审视自己的思考过程并加以改进。Self-Reflection赋予Agent类似的能力：在输出最终答案之前，先对自己的推理进行批判性审查。

反思的核心机制：

  • Critique Generation (生成批评)：LLM对自己的初始输出进行审查，找出逻辑漏洞、事实错误、遗漏信息和不一致之处。关键在于设计一个专注于"找茬"的critique prompt。
  • Error Analysis (错误分析)：将错误分类——是知识错误（模型不知道）？推理错误（逻辑跳跃）？还是格式错误（输出不符合要求）？不同类别需要不同的修正策略。
  • Iterative Improvement (迭代改进)：基于批评和错误分析结果，重新生成修正后的答案。这个过程可以迭代2-3轮，每轮都会提升输出质量。

Self-Reflection的实际效果：研究表明，在不需要外部工具或额外训练数据的情况下，仅通过Self-Reflection就能将Agent在复杂推理任务上的准确率提升10-30%。值得注意的是，反思的质量取决于反思prompt的设计——一个好的反思prompt需要明确指出"从哪些角度审视答案"（事实准确性、逻辑一致性、完整性、清晰度等）。

注意事项

Self-Reflection也有局限性：(1) 如果模型的初始输出有根本性错误，反思可能也发现不了——这是"盲点"问题；(2) 反思可能引入新的错误——"过度修正"问题；(3) 每轮反思增加一次LLM调用，成本翻倍。建议为反思设置置信度阈值，只对低置信度的输出进行反思。

7. Self-Consistency：从多角度验证

Self-Consistency (自一致性)由Wang et al.(2023)提出，核心思想是：对于同一个问题，让LLM独立生成多条推理路径（通过调整temperature引入随机性），然后通过投票选择最一致的答案。这种方法用"统计共识"弥补了单条推理的不确定性。的直觉来自一个简单的观察：如果一个人从多个不同的角度思考同一个问题都得出相同的结论，那么这个结论很可能就是正确的。应用到LLM上：

  1. 多样本生成：使用非零temperature（通常0.5-0.8），让模型对同一问题独立生成N条推理路径（N通常取5-20）。较高的temperature确保了推理路径的多样性。
  2. 答案提取：从每条推理路径中提取最终答案。对于选择题，就是选项字母；对于开放式问题，需要通过解析提取核心主张。
  3. 投票聚合：对提取出的答案进行多数投票。得票最多的答案作为最终输出。
  4. 加权变体：更高级的做法是加权投票——根据每条推理链的连贯性、长度、是否引用证据等因素赋予不同权重。

Self-Consistency vs Self-Reflection 对比：

维度 Self-Consistency Self-Reflection
核心机制 多次采样+投票 自我批评+修正
LLM调用次数 N次（并行） 2-3次（串行）
适合的任务 有明确答案的选择题、数学题 开放生成、写作、代码
最大优势 利用统计降低单次采样的随机错误 引入元认知，能发现深层逻辑问题

值得注意的是，Self-Consistency和Self-Reflection可以叠加使用：先用Self-Consistency生成并投票选出最佳答案，再对这个答案进行Self-Reflection修正。这种组合在多个benchmark上创造了SOTA结果。

8. 推理策略选择框架：何时用什么？

面对不同的任务，如何选择最合适的推理策略？以下是一个系统化的决策框架——按照任务特征匹配最佳策略：

任务特征 推荐策略 理由
需要外部工具交互 ReAct Thought-Action-Observation循环天然适合工具调用
纯推理、无工具调用 Chain-of-Thought 简单高效，token消耗低
多步骤、步骤多(>10) Plan-and-Execute 先全局规划再执行，避免迷失
需要探索多种方案 Tree of Thoughts 树形搜索探索多条路径，找到最优解
准确率要求高、容错低 Self-Consistency 多次采样投票，降低随机错误概率
输出质量可迭代提升 Self-Reflection 自我批评+修正，逐步优化
实时交互、动态环境 ReAct + Replanning ReAct响应环境变化，失败时触发重规划
预算有限、延迟敏感 Direct / Zero-shot CoT 最小化LLM调用，单次推理完成

组合策略：实际生产中很少只用一种策略。例如，一个代码Agent的系统prompt可能同时包含：(1) Plan-and-Execute用于高层任务分解；(2) ReAct用于单个子任务的工具调用；(3) Self-Reflection用于最终代码的审查修正。关键在于根据任务阶段动态切换策略。

决策口诀

"工具交互用ReAct，纯推理用CoT；步骤多用Plan，方案多用Tree；要准确用Consistency，要优化用Reflection；简单任务别折腾，直接回答最高效。"

9. 完整代码示例：带详细日志的ReAct循环

下面是一个生产级别的ReAct Agent实现，包含了完整的日志记录、错误处理、重试机制和token统计：

Python — 完整的ReAct Agent实现 
import json, time, logging
from dataclasses import dataclass, field
from typing import Any, Callable

logging.basicConfig(level=logging.INFO, format="%(asctime)s [%(levelname)s] %(message)s")
logger = logging.getLogger("ReActAgent")

@dataclass
class ToolCall:
    name: str
    args: dict
    result: Any = None
    error: str = None
    duration_ms: float = 0

@dataclass
class ReActStep:
    step_num: int
    thought: str
    tool_call: ToolCall = None
    observation: str = ""
    is_final: bool = False

@dataclass
class ReActAgent:
    llm: Callable                          # LLM调用函数
    tools: dict                              # 工具名 -> 工具函数
    max_steps: int = 15
    max_retries: int = 3
    verbose: bool = True
    trace: list = field(default_factory=list)

    def run(self, task: str) -> str:
        """主执行循环——ReAct的核心"""
        messages = [{"role": "user", "content": task}]
        total_tokens = 0

        for step_num in range(1, self.max_steps + 1):
            self._log(f"========== Step {step_num} ==========", "HEADER")

            # === THOUGHT: 模型推理 ===
            response = self.llm(messages, tools=self.tools)
            total_tokens += response.usage.total_tokens
            thought = response.content
            self._log(f"💭 THOUGHT: {thought}", "THINK")

            # 检查是否已得出最终答案
            if response.tool_calls is None:
                self.trace.append(ReActStep(step_num, thought, is_final=True))
                self._log(f"✅ 任务完成 (tokens: {total_tokens})", "DONE")
                return thought

            # === ACTION: 执行工具调用 ===
            for tc in response.tool_calls:
                tool_call = ToolCall(name=tc.name, args=tc.args)

                # 带重试的工具执行
                for attempt in range(self.max_retries):
                    try:
                        self._log(f"🔧 ACTION: {tc.name}({json.dumps(tc.args)})", "ACTION")
                        t0 = time.perf_counter()
                        result = self.tools[tc.name](**tc.args)
                        tool_call.duration_ms = (time.perf_counter() - t0) * 1000
                        tool_call.result = result
                        break
                    except Exception as e:
                        tool_call.error = str(e)
                        self._log(f"⚠️ 重试 {attempt+1}/{self.max_retries}: {e}", "WARN")
                        if attempt == self.max_retries - 1:
                            tool_call.result = {"error": str(e)}

                # === OBSERVATION: 观察结果 ===
                obs_text = json.dumps(tool_call.result, ensure_ascii=False)
                self._log(f"👁️ OBSERVATION: {obs_text[:200]}", "OBSERVE")
                self._log(f"⏱️ 耗时: {tool_call.duration_ms:.1f}ms", "DEBUG")

                # 将工具结果反馈到消息历史
                messages.append({"role": "tool", "content": obs_text})

                class _Step:
                    step_num=step_num; thought=thought; tool_call=tool_call; observation=obs_text
                self.trace.append(_Step)

        self._log("⛔ 达到最大步数限制", "ERROR")
        return "任务超时：无法在最大步数内完成。"

    def _log(self, msg: str, tag: str):
        if self.verbose:
            logger.info(f"[{tag}] {msg}")

    def print_trace(self):
        """打印完整的执行轨迹——用于调试和分析"""
        for step in self.trace:
            print(f"\n--- 步骤 {step.step_num} ---")
            print(f"  💭 {step.thought}")
            if step.tool_call:
                print(f"  🔧 {step.tool_call.name}({step.tool_call.args})")
                print(f"  👁️ {step.observation[:100]}")
                if step.tool_call.error:
                    print(f"  ⚠️ 错误: {step.tool_call.error}")

代码要点解读：

  • Trace记录：每一步的思考、行动和观察都被完整记录在self.trace中。这在调试Agent行为时无价——你能精确定位Agent在哪个步骤做出了错误决策。
  • 重试机制：工具调用可能因网络问题临时失败。3次重试策略（带指数退避更佳）大大提高了系统的鲁棒性。
  • Token统计：追踪每次LLM调用的token消耗，为成本优化提供数据支撑。
  • 分层日志：使用不同TAG区分Thought、Action、Observation，日志输出清晰可读。
  • 最终判断：当LLM不再返回tool_calls时，意味着它认为任务已完成——这是ReAct的"停止条件"。

10. 推理策略的实战技巧与常见陷阱

基于大量Agent实践经验，以下是一些关键教训：

技巧：

  • 早停机制：ReAct循环中设置"思考收敛检测"——如果连续2轮的Thought内容高度相似（可用embedding余弦相似度检测），说明Agent陷入循环，应强制终止或注入新信息。
  • 思考预算：为不同类型的任务设置不同的max_steps。简单查询3步，中等任务7步，复杂任务15步。动态调整比固定上限更高效。
  • 工具描述即文档：每个工具的description不是写给开发者看的，而是写给LLM看的。要像写微缩版API文档一样精心撰写，包含"何时使用"、"输入约束"、"输出格式"。
  • 渐进式复杂度：先用最简单的策略（Direct/Zero-shot CoT）尝试，失败后再升级到ReAct，再失败升级到Plan-Execute，最后才用ToT。不要上来就用最复杂的策略。

常见陷阱：

  • 无限循环：Agent在Thought-Action-Observation中反复做同样的事。解决方法是设置步数上限 + 内容新鲜度检测。
  • 过早终止：Agent在第1-2步就认为任务完成了，实际还差得远。解决方法是在system prompt中明确"做完X、Y、Z之后再给最终答案"。
  • 规划过度：Plan-and-Execute中，Agent花了5步去规划一个3步就能完成的任务。解决方法是限制规划阶段的步数和token预算。
  • 工具幻觉：Agent调用了不存在的工具或传了错误参数。解决方法是严格的Schema验证 + 工具调用前的参数检查。
关键教训

推理策略是手段而非目的。用户关心的是任务是否完成、结果是否准确，而非Agent用了多高级的推理策略。选择策略的唯一标准是：在满足质量要求的前提下，最小化延迟和成本。
