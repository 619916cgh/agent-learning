第 2 章

Agent 核心架构

深入理解 Agent 的内部构造、运行循环和分层设计。本章将为你揭开 Agent 系统的"黑箱"，让你看清它的每一个组成部分如何协同工作。

🏗
Agent 全景解构：从外到内看透 Agent
一张思维导图式描述，建立对 Agent 完整架构的直观印象

在深入到每个组件之前，我们先用一个全景视角审视 Agent 的整体结构。想象 Agent 是一个有五层楼的建筑，每一层都有独立的职责，层与层之间通过明确的接口通信：

Agent 五层架构全景图
🛡 第5层 — Guardrails（安全护栏）
输入验证 | 输出过滤 | 权限检查 | 审计日志 | 速率限制 | 内容审核
💭 第4层 — Memory（记忆系统）
短期记忆（对话窗口）| 工作记忆（任务状态）| 长期记忆（向量库）
⚙ 第3层 — Orchestrator（编排引擎）
主循环控制 | 步骤调度 | 错误恢复 | 超时管理 | 并发协调
🛠 第2层 — Tools（工具层）
工具注册 | Schema管理 | 执行适配 | 结果转换 | 错误标准化
🧠 第1层 — Model（LLM推理引擎）
模型选择 | 推理调用 | Token管理 | 温度控制 | 流式输出

这个五层架构的"数据流向"遵循一个清晰的模式：用户输入从最外层（Guardrails）进入，逐层向内传递到 Model 层进行推理，然后 Model 的决策经过 Orchestrator 编排后向外层（Tools）传递并执行，执行结果再回传到内层形成下一个推理循环。

架构设计理念

五层架构的核心设计理念是关注点分离（Separation of Concerns）。每一层只做自己的事，不跨层干涉。这种设计让你可以单独替换任一层而不会破坏整体。例如，你可以将 LLM 从 GPT-4 换成 Claude，只需要修改第1层的适配代码，其他四层完全不需要变动。

🔃
灵魂循环：感知-规划-行动-观察
Agent 与其他 AI 系统最本质的区别就在于这个持续运行的循环

如果只能用一个概念来理解 Agent，那就是感知-规划-行动-观察循环（Perceive-Plan-Act-Observe Loop）。这个循环是 Agent 的"心跳"，只要 Agent 还在运行，这个循环就在持续转动。下面我们通过一个"用户请求 Agent 分析销售数据"的真实案例来追踪循环的每一步。

第1阶段：感知 (Perceive)

触发：用户输入 "帮我分析上个月各区域的销售数据，找出表现最好的产品"

Agent 做了什么：

  • 接收原始用户输入文本
  • 结合系统提示词（system prompt）理解任务目标和约束
  • 识别出这是一个数据分析任务，需要访问数据库、进行计算、生成结论
  • 从工作记忆中检查是否有相关的历史上下文（如之前查询过类似数据）
  • 从长期记忆中检索是否有相关的业务知识（如"区域"的定义、"产品"的分类）

输出：结构化的任务理解 — "目标：分析销售数据；维度：区域、产品；指标：销售额/利润；时间范围：上个月；产出：最佳产品清单"

第2阶段：规划 (Plan)

Agent 做了什么：

  • 拆分子任务：① 查询数据库获取销售数据；② 按区域汇总计算；③ 按产品汇总计算；④ 排序找出 Top 产品；⑤ 生成分析报告
  • 为每个子任务匹配合适的工具：查询用 SQL 工具、汇总用计算工具、报告用文本生成
  • 规划执行顺序：先查数据，再计算，最后生成报告（顺序依赖关系）
  • 评估风险：数据量是否过大？查询是否会超时？是否需要用户确认？

输出：执行计划 — 一个有顺序依赖关系的步骤列表

第3阶段：行动 (Act)

Agent 做了什么：

  • 依次调用计划中的工具：执行 SQL 查询 -> 对结果进行聚合计算 -> 排序筛选
  • 每次工具调用都等待结果返回，验证结果的有效性
  • 如果某步失败（如 SQL 语法错误），进入错误恢复流程
  • 将中间结果存入工作记忆，供后续步骤使用

第4阶段：观察 (Observe)

Agent 做了什么：

  • 检查每个工具的执行结果是否符合预期格式和内容
  • 判断任务是否已经完成：数据是否完整？结论是否充分？
  • 如果完成，生成最终回答并退出循环
  • 如果未完成，带着新的观察结果回到"感知"阶段，开始新一轮循环
循环的关键特征

注意三个关键点：1) 循环不是线性的——Agent 可能在"行动"之后发现需要重新"规划"；2) 每一次循环都在消耗 token 和时间，所以高效的 Agent 会尽量减少无意义的循环；3) 终止条件的设计非常关键——它决定了 Agent 是"懂得适时收手"还是"陷入死循环"。

🧠
LLM 作为推理引擎：Prompt 如何变成 Action
理解 LLM 在 Agent 架构中扮演的独特角色和运作机制

在 Agent 架构中，LLM 的角色不是"聊天"，而是推理决策。每一次 LLM 调用的输入和输出都是精心设计的结构化信息，而非自由对话。理解这个过程是掌握 Agent 架构的关键。

输入：比"用户消息"丰富得多

当 Agent 调用 LLM 时，发送的不是一条简单的用户消息，而是一个复合上下文包：

  1. 系统提示词 (System Prompt)：定义 Agent 的角色、行为准则和可用工具
  2. 历史消息 (Message History)：用户输入、Agent 之前的回答、工具调用记录
  3. 工具定义 (Tool Definitions)：完整的工具列表及其 JSON Schema
  4. 记忆注入 (Memory Context)：从长期记忆中检索到的相关信息
  5. 规划状态 (Plan State)：当前任务的执行计划和已完成步骤（可选）

推理过程：多路径决策树

LLM 在收到这个上下文包后，内部经历一个多路径的决策过程：

  • 理解路径：解析用户意图，识别任务类型和复杂度
  • 工具匹配路径：在所有可用工具中评估哪个最适合当前步骤
  • 参数推理路径：从上下文和历史中提取/推断工具所需的参数值
  • 终止判断路径：评估是否应该停止循环并输出最终结果

输出：结构化的行动指令

LLM 的输出不是自由文本，而是两种可能的形式之一：

  • 工具调用请求：包含 tool_name 和 arguments JSON，告诉编排器"我要调用这个工具，参数是这些"
  • 最终文本回答：表示任务完成，输出给用户的自然语言结果
常见误区

很多人误以为 Agent 的 LLM 在"控制一切"——实际上，LLM 只负责推理和决策，真正的执行由编排器和工具层完成。LLM 说"查询数据库"，但它不亲自查询数据库；编排器接收这个指令，调用数据库工具，再把结果送回给 LLM。这个"决策与执行分离"的架构是 Agent 设计的精髓所在。

🗒
上下文窗口机制：Agent 的"视野"限制
上下文窗口的大小和管理方式深刻影响着 Agent 的行为能力

上下文窗口（Context Window）是 LLM 处理信息的基本单位，也是 Agent 设计中最重要的硬约束之一。当前的顶级模型支持 128K 到 200K token 的上下文窗口（约等于一本中等长度的小说），但这个空间需要被很多内容共享。

上下文窗口的"分配图"

假设你的 LLM 有 128K token 的上下文窗口，在一次典型的 Agent 任务中，这个空间是这样分配的：

内容 典型占用 说明
系统提示词 2K - 8K 固定开销，每次请求都携带
工具定义 Schema 3K - 10K 随工具数量线性增长；10个复杂工具可占 10K+
对话历史 10K - 50K 随交互轮数增长；是上下文压力的主要来源
工具调用结果 5K - 30K 每次工具返回的数据都在这里累积
记忆检索注入 2K - 8K RAG 注入的相关文档片段
输出预留空间 2K - 16K LLM 生成回答所需的空间

上下文窗口压力带来的挑战

  • 信息丢失：当总内容超出窗口限制时，最早的信息会被截断。如果关键信息正好在"被遗忘"的部分，Agent 的行为就会出错。
  • 注意力稀释：即使没有超出窗口，过多的内容也会稀释 LLM 对关键信息的注意力。这就是著名的"中间丢失"（Lost in the Middle）现象。
  • 成本线性增长：每次请求携带的上下文越多，token 消耗越大，API 成本越高。一个长对话的 Agent 任务可能消耗上百万 token。
  • 延迟增加：更大的上下文意味着更长的处理时间，直接影响用户体验。

应对策略

  • 滑动窗口：只保留最近 N 轮对话，旧消息自动丢弃
  • 摘要压缩：对历史对话进行定期摘要，用短摘要替代长历史
  • 智能裁剪：基于相关性评分，优先保留最重要的信息
  • 分层存储：重要信息存入长期记忆，通过检索按需加载
  • 结构化输出：减少不必要的大段文本，用结构化数据替代
设计启示

上下文窗口是 Agent 架构中一个"看不见的设计约束"，几乎所有高级特性（记忆管理、工具设计、对话策略）都在某种程度上是为了应对上下文窗口的局限性而存在的。理解了上下文窗口，你就能理解为什么 Agent 需要记忆系统、为什么 RAG 很重要、为什么工具设计要简洁。

五层架构深度解析：每层都怎么运转
逐层剖析 Model、Tools、Orchestrator、Memory、Guardrails 的内部机制

第1层 — Model（LLM推理引擎）

Model 层封装了与 LLM 的所有交互。它的核心职责：

  • 模型抽象：提供统一的调用接口，屏蔽不同 LLM 提供商（OpenAI、Anthropic、本地模型）的差异
  • 参数管理：控制 temperature、top_p、max_tokens 等推理参数
  • 流式处理：支持 streaming 输出，让用户实时看到 Agent 的思考过程
  • 重试与容错：处理 API 超时、限流、服务不可用等异常情况

第2层 — Tools（工具层）

工具层是 Agent "能做之事"的目录。它的核心职责：

  • 工具注册：维护一个工具注册表，每个工具都有名称、描述、Schema 和实现
  • Schema 生成：将工具的 Python/JS 函数签名自动转换为 LLM 可理解的 JSON Schema
  • 执行适配：将 LLM 传回的参数映射到实际函数调用
  • 结果标准化：将各种工具的不同返回格式统一为 LLM 可处理的标准格式

第3层 — Orchestrator（编排引擎）

编排器是 Agent 的"大脑皮层"，控制着整个执行流程：

  • 主循环驱动：驱动感知-规划-行动-观察循环的运转
  • 步骤调度：决定下一步做什么、调用哪个工具
  • 错误恢复：当工具执行失败时，判断是重试、换个工具还是向用户求助
  • 终止判断：在合适的时机结束循环，避免无限执行
  • 并发协调：在高级 Agent 中管理多个并行工具调用

第4层 — Memory（记忆系统）

记忆系统让 Agent 拥有"完整的记忆链"：

  • Short-term：管理对话历史窗口和滑动窗口策略
  • Working：维护当前任务的状态、中间结果和执行计划
  • Long-term：持久化重要信息，通过向量检索按需召回

第5层 — Guardrails（安全护栏）

安全护栏贯穿所有层次，是最后一道防线：

  • 输入门禁：过滤恶意 prompt、限制输入长度、检测注入攻击
  • 执行沙箱：在隔离环境中执行高风险操作（代码执行、文件删除）
  • 权限管理：工具级别的权限控制，某些操作需要用户确认
  • 输出审查：检查输出是否包含敏感信息、是否符合安全规范
📝
系统提示词剖析：Agent 的"DNA"
一个好的 System Prompt 决定了 Agent 的行为模式和能力边界

系统提示词（System Prompt）是 Agent 设计中最重要但也最容易被低估的部分。它就像是 Agent 的"DNA"——定义了 Agent 是谁、能做什么、不能做什么、以及如何与用户和工具交互。

高效 System Prompt 的四要素

① 角色定义 (Persona)

明确告诉 LLM "你是谁"。不是模糊的"你是一个 AI 助手"，而是具体的：

你是一个专业的数据分析 Agent，隶属于 ProductInsight 团队。你的主要职责是帮助用户从数据库中提取数据、执行分析计算并生成可操作的洞察报告。你擅长 SQL、Python 数据处理和统计分析方法。

② 行为约束 (Constraints)

划定 Agent 的"行为红线"。这些约束应该是明确的、可执行的：

规则：
1. 涉及数据删除或修改的操作，必须先向用户确认
2. 单次查询返回超过 10000 行时，先告知用户并建议筛选
3. 如果同一个工具连续失败 3 次，停止尝试并解释原因
4. 不要在回答中编造数据——如果不确定，明确告知
5. 优先使用最直接的工具，避免不必要的多步调用

③ 工具使用指南 (Tool Usage)

告诉 LLM 每个工具适用的场景和选择优先级：

工具选择指南：
- 查询数据库：使用 query_database（SQL）
- 数据统计分析：使用 run_python（pandas, scipy）
- 可视化：使用 generate_chart
- 文件操作：使用 file_read / file_write
- 需要最新信息：使用 web_search
注意：query_database 和 web_search 可以并行调用来加速

④ 输出格式 (Output Format)

定义 Agent 最终输出的结构和风格：

回答格式：
- 先给出 2-3 句的 TL;DR 总结
- 然后分点列出关键发现
- 如果涉及数据，以 Markdown 表格呈现
- 最后给出可操作的下一步建议
- 语气：专业但平易近人，避免过度使用技术术语

系统提示词工程的核心原则

1) 越具体越好——模糊的指令会导致不可预测的行为；2) 先给规则再给例子——规则建立框架，例子填充细节；3) 用"不要做什么"来划定边界——负面约束往往比正面指令更有效；4) 测试-迭代-再测试——没有一个系统提示词是一次就写对的。

代码实战：完整 Agent 编排器实现
一个可直接运行的 Agent 编排器类，包含所有核心机制

下面是一个生产级 Agent 编排器的完整实现。它展示了五层架构如何在实际代码中落地运行。请仔细阅读每个方法的实现和注释，它们对应着我们前面讨论的每一个架构概念。

Python — Agent Orchestrator 完整实现 
import json
import time
from typing import Any, Callable, Optional
from dataclasses import dataclass, field


# ============================================================
# 数据结构定义
# ============================================================

@dataclass
class ToolCall:
    """LLM 返回的工具调用请求"""
    id: str
    name: str
    arguments: dict


@dataclass
class LLMResponse:
    """LLM 的标准化返回结果"""
    text: Optional[str] = None
    tool_calls: Optional[list[ToolCall]] = None

    def has_final_answer(self) -> bool:
        return self.text is not None and self.tool_calls is None

    def has_tool_calls(self) -> bool:
        return self.tool_calls is not None and len(self.tool_calls) > 0


@dataclass
class ExecutionStep:
    """单步执行的完整记录（用于调试和审计）"""
    step_number: int
    thought: str
    tool_name: Optional[str]
    tool_args: Optional[dict]
    tool_result: Optional[Any]
    error: Optional[str]
    timestamp: float


# ============================================================
# Agent 编排器主类
# ============================================================

class AgentOrchestrator:
    """完整的 Agent 编排器，实现五层架构的协同运转。
    
    这个类展示了：
    - 主循环控制 (Orchestrator 层)
    - LLM 推理调用 (Model 层)
    - 工具注册与执行 (Tools 层)
    - 短期记忆管理 (Memory 层)
    - 安全边界控制 (Guardrails 层)
    """

    def __init__(
        self,
        llm_client,
        system_prompt: str,
        tools: dict[str, Callable],
        max_steps: int = 15,
        context_limit: int = 50,
        verbose: bool = True
    ):
        self.llm = llm_client
        self.system_prompt = system_prompt
        self.tools = tools                    # {name: callable}
        self.tool_schemas = self._build_tool_schemas()
        self.max_steps = max_steps
        self.context_limit = context_limit    # 滑动窗口大小
        self.verbose = verbose

        # 运行时状态
        self.messages = []                    # 短期记忆：对话历史
        self.execution_trace = []             # 完整的执行轨迹（可观测性）
        self.working_state = {}               # 工作记忆：任务状态
        self.tool_error_counts = {}           # 工具错误计数

    # ================================================================
    # 公共接口
    # ================================================================

    def run(self, user_input: str) -> str:
        """启动 Agent 执行主循环。返回最终答案。"""
        self._initialize(user_input)

        for step in range(1, self.max_steps + 1):
            self._log(f"\n{'='*40}\n  第 {step} 步\n{'='*40}")

            # --- 感知阶段 ---
            context = self._build_context()

            # --- 规划与推理阶段 ---
            response = self._call_llm(context)

            # --- 检查是否完成 ---
            if response.has_final_answer():
                self._log(f"任务完成: {response.text[:100]}...")
                self.messages.append({
                    "role": "assistant", "content": response.text
                })
                return response.text

            # --- 行动阶段 ---
            if response.has_tool_calls():
                for tool_call in response.tool_calls:
                    result = self._execute_single_tool(tool_call, step)

                    # --- 观察阶段：将结果反馈给上下文 ---
                    self.messages.append({
                        "role": "tool",
                        "tool_call_id": tool_call.id,
                        "content": json.dumps(result, ensure_ascii=False)
                    })

        # 超出步数限制：生成降级回答
        return self._graceful_degradation()

    # ================================================================
    # 内部方法
    # ================================================================

    def _initialize(self, user_input: str):
        """初始化：清空状态，设置系统提示词和用户输入"""
        self.messages = [
            {"role": "system", "content": self.system_prompt},
            {"role": "user", "content": user_input}
        ]
        self.execution_trace = []
        self.working_state = {"started_at": time.time(), "completed_steps": []}
        self.tool_error_counts = {}

    def _build_context(self) -> list:
        """构建 LLM 的输入上下文（含滑动窗口裁剪）"""
        # 始终保留 system 消息
        system_msg = self.messages[0]
        # 对非 system 消息应用滑动窗口
        recent = self.messages[1:][-self.context_limit:]
        return [system_msg] + recent

    def _call_llm(self, context: list) -> LLMResponse:
        """调用 LLM 进行推理（带重试机制）"""
        for attempt in range(3):
            try:
                raw = self.llm.chat(
                    messages=context,
                    tools=self.tool_schemas,
                    tool_choice="auto"  # 让模型自行判断是否需要工具
                )
                return self._parse_llm_response(raw)
            except Exception as e:
                if attempt == 2:
                    raise RuntimeError(f"LLM 调用失败（已重试3次）: {e}")
                time.sleep(2 ** attempt)  # 指数退避

    def _execute_single_tool(self, tool_call: ToolCall, step: int):
        """执行单个工具调用（含安全校验和错误处理）"""
        trace = ExecutionStep(
            step_number=step,
            thought="",
            tool_name=tool_call.name,
            tool_args=tool_call.arguments,
            tool_result=None,
            error=None,
            timestamp=time.time()
        )

        # Guardrail 1: 工具存在性检查
        if tool_call.name not in self.tools:
            trace.error = f"工具 '{tool_call.name}' 不存在"
            self.execution_trace.append(trace)
            return {"error": trace.error, "hint": "请从可用工具列表中选择"}

        # Guardrail 2: 错误频率检查
        if self.tool_error_counts.get(tool_call.name, 0) >= 3:
            trace.error = f"工具 '{tool_call.name}' 已连续失败3次，已禁用"
            self.execution_trace.append(trace)
            return {"error": trace.error, "hint": "请换一种方式完成此任务"}

        # 执行工具
        try:
            result = self.tools[tool_call.name](**tool_call.arguments)
            trace.tool_result = result
            self.working_state["completed_steps"].append({
                "tool": tool_call.name, "result": str(result)[:200]
            })
            return result

        except Exception as e:
            trace.error = str(e)
            self.tool_error_counts[tool_call.name] = \
                self.tool_error_counts.get(tool_call.name, 0) + 1
            self.execution_trace.append(trace)
            return {
                "error": f"执行失败: {e}",
                "hint": f"请检查参数是否正确，或尝试其他工具"
            }

        finally:
            self.execution_trace.append(trace)
            if self.verbose:
                status = "OK" if not trace.error else f"ERROR: {trace.error}"
                self._log(f"  工具: {tool_call.name} | 状态: {status}")

    def _graceful_degradation(self) -> str:
        """超出步数限制时的优雅降级处理"""
        completed = len(self.working_state["completed_steps"])
        return (
            f"任务已执行 {self.max_steps} 步，但尚未完成。
            已完成的步骤：{completed} 个。
            建议将任务拆分为更小的子任务重试。
            执行摘要：{json.dumps(self.execution_trace[-3:], default=str)}"
        )

    def _build_tool_schemas(self) -> list:
        """从注册的工具构建 LLM 工具 Schema（简化版）"""
        return [
            {
                "type": "function",
                "function": {
                    "name": name,
                    "description": fn.__doc__ or "无描述",
                    "parameters": {"type": "object", "properties": {}}
                }
            }
            for name, fn in self.tools.items()
        ]

    def _parse_llm_response(self, raw) -> LLMResponse:
        """解析 LLM 的原始响应为标准格式"""
        # 实际实现需要适配具体 LLM 的返回格式
        text = raw.get("content")
        raw_tool_calls = raw.get("tool_calls", [])
        tool_calls = [
            ToolCall(id=tc["id"], name=tc["function"]["name"],
                     arguments=json.loads(tc["function"]["arguments"]))
            for tc in raw_tool_calls
        ] if raw_tool_calls else None
        return LLMResponse(text=text, tool_calls=tool_calls)

    def _log(self, msg: str):
        if self.verbose:
            print(msg)
代码设计亮点

注意几个关键设计决策：1) 所有状态（messages, execution_trace, working_state）都封装在类实例中，方法不产生副作用；2) 每个工具调用都有完整的执行追踪（ExecutionStep），方便调试和审计；3) 错误处理不是简单粗暴地抛异常，而是返回结构化的错误信息给 LLM；4) tool_error_counts 机制防止某个"坏工具"无限拖垮整个任务。

⚖
架构决策指南：在实践中做选择
面对不同场景时，如何进行合理的架构选择和权衡

决策1：同步 vs 异步工具调用

当一个步骤中需要调用多个相互独立的工具时（如同时查天气和查股票），你有两个选择：

  • 同步顺序调用：简单可靠，但总耗时 = 每个工具耗时之和。适合工具少、对延迟不敏感的场景。
  • 异步并行调用：总耗时 = 最慢工具耗时。效率高，但增加编排复杂度。适合工具多、需要快速响应的场景。

建议：从同步开始，确认工具稳定后再引入并行。

决策2：单模型 vs 多模型

  • 单模型方案：所有推理由一个 LLM 完成。架构简单，上下文一致性好。
  • 多模型方案：不同任务走不同模型。如用轻量模型做意图分类，用强模型做深度推理。成本和延迟更优，但增加协调复杂度。

建议：大部分场景下单模型已足够。只有当有明确的"轻任务-重任务"区分时，多模型才有意义。

决策3：固定流程 vs 动态规划

  • 固定流程：预定义执行步骤和顺序（如 LangChain 的 Chain）。可控性强，适合标准化任务。
  • 动态规划：让 Agent 自主决定每一步做什么（如 ReAct 模式）。灵活性强，适合开放性任务。

建议：混合使用。将可预测的部分固定流程化，不可预测的部分交给 Agent 自主决策。

架构设计的黄金法则

"先让它工作，再让它优雅，最后让它快速"。面对架构选择时，优先选择最简单可行的方案。过早优化是 Agent 工程中的常见陷阱——你可能花了大量时间优化的并行调用，最终发现瓶颈在 LLM 推理延迟上。
