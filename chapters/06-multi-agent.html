第 6 章

多Agent系统

探索多个AI Agent如何协作、通信和编排——从顺序管道到去中心化Swarm，构建群体智能

1. 为什么需要多Agent系统？

单个Agent虽然强大，但面对复杂、大规模、跨领域的任务时，单Agent架构逐渐暴露其瓶颈。多Agent系统(Multi-Agent System, MAS)通过专业化分工与协作，突破了单一Agent的能力边界：

维度 单Agent 多Agent系统
任务复杂度 受限于单一prompt的承载能力 任务自然分解，每个Agent聚焦子任务
专业深度 "通才"——广度有余，深度不足 每个Agent可成为特定领域的"专才"
鲁棒性 单点故障，出错即整体失败 某个Agent失败可不影响整体，可重试
可扩展性 垂直扩展困难 水平扩展——添加新Agent即可增强能力
视角多样性 单一视角，可能有盲点 多Agent从不同角度审视问题，减少盲点

核心驱动力——任务分解(Task Decomposition)：多Agent系统的理论基础是"分而治之"。一个"构建电商网站"的任务可以自然分解为：前端Agent(UI设计+代码) + 后端Agent(API+数据库) + 测试Agent(质量保障) + 文档Agent(编写文档)。每个Agent只需要在自己的专业领域内做到最好。

核心洞察

多Agent系统的价值不仅在于"做更多的事"，更在于"以更高的质量做更复杂的事"。一个由专业Agent组成的团队，其输出质量往往远超一个"万能"Agent。

2. Agent间通信协议：如何"对话"？

多个Agent要协同工作，首先需要解决通信问题。Agent之间的通信协议决定了信息如何流动、被谁访问、何时可用。以下是三种主流通信模式：

2.1 共享内存 (Shared Memory)

所有Agent读写同一个全局上下文。这是最简单也是最常用的模式：一个全局的消息列表或键值存储，Agent A写入结果，Agent B随后读取。优点是实现简单、延迟低；缺点是随着Agent数量增加，上下文会急剧膨胀，且缺乏访问控制——任何Agent都能看到所有信息。LangChain中的ConversationBufferMemory本质上就是共享内存模式。

2.2 消息传递 (Message Passing)

Agent之间通过显式的消息发送和接收来通信——类似Actor模型。每个Agent有独立的收件箱(Inbox)，消息路由器(Message Router)负责将消息投递到目标Agent。优点是可实现点对点通信、支持异步处理、可追踪消息链路；缺点是增加了系统复杂度和通信延迟。消息传递源自Actor模型(1973年由Carl Hewitt提出)，每个Actor是一个独立的计算实体，通过异步消息通信。这种模式被Erlang/Elixir发扬光大，在分布式系统中广泛应用。是构建大规模多Agent系统的基础通信范式。

2.3 黑板架构 (Blackboard Architecture)

一个公共的数据空间(Blackboard)，Agent可以：(1) 查看黑板上已有的信息；(2) 写入新的发现或中间结果；(3) 当黑板上的信息满足特定条件时被唤醒。黑板模式特别适合数据驱动的协作——多个Agent各自贡献不同维度的分析，逐渐拼凑出完整的解决方案。例如在智能诊断系统中：症状采集Agent写入症状、检验Agent写入化验结果、影像Agent写入影像分析，诊断Agent综合所有信息给出判断。

选择建议

小型系统(2-5个Agent)首选共享内存；中等规模(5-20个Agent)推荐消息传递；需要多轮信息聚合的场景选择黑板架构。实际系统中经常混合使用多种通信模式。

3. 拓扑模式：Agent如何组织？

拓扑结构决定了多Agent系统中权力、信息流和依赖关系的分布。不同的拓扑适用于不同的问题结构：

拓扑模式 结构 控制方式 典型场景
顺序管道 A → B → C → D，单向流水线 数据驱动，前一个输出=后一个输入 CI/CD流水线、文档生成(研究→撰写→审校→翻译)
星型/Hub 一个中心Agent连接多个外围Agent 中心Agent路由和调度 客服系统(路由Agent+各领域专家)
网状(Mesh) 每个Agent都可以与任何其他Agent通信 完全去中心化，协商决策 学术辩论、方案评审、头脑风暴
层级(Hierarchical) 树形结构，上级Agent管理下级Agent 指令自上而下，结果自下而上 大型项目管理、企业级自动化
Swarm/群体 动态无结构，Agent按需加入和退出 基于Handoff的自主调度 开放域对话、动态问题处理

组合拓扑——真实世界的混合架构：实际生产系统很少使用单一拓扑。例如一个代码助手平台可能同时使用：(1) 层级拓扑管理项目(项目经理Agent → 前端/后端Agent)；(2) 顺序管道在每个子团队内部(设计Agent → 编码Agent → 审查Agent)；(3) 星型拓扑处理用户请求(路由Agent分发请求到对应的项目团队)。

设计原则

选择拓扑不是越复杂越好，而是越匹配问题结构越好。如果问题是流水线型的(输出-输入-输出)，就用管道；如果问题需要多方论证和协商，就用网状或Swarm。

4. Agent角色设计：如何打造高效的专业Agent？

在多Agent系统中，每个Agent的角色定义直接决定了系统的整体效能。一个设计精良的Agent角色需要覆盖以下五个维度：

角色设计的五个核心要素：

  • 身份 (Identity)：Agent的"我是谁"——包括名称、专业领域、资历描述。例如："你是一位拥有15年经验的资深后端架构师，专注于分布式系统和微服务设计。" 清晰的身份定位为Agent的行为提供了稳定的参照系。
  • 职责 (Responsibility)：Agent的"我要做什么"——明确的任务范围，包含必须做什么和不应该做什么。职责边界不清是多Agent系统最常见的问题根源。
  • 能力 (Capability)：Agent的"我能用什么工具"——可调用的工具集、可访问的数据源、拥有的权限。例如CoderAgent可以调用read_file、write_file、execute_code；而ReviewerAgent只有read_file和add_comment。
  • 约束 (Constraint)：Agent的"我不能做什么"——行为边界和规则。例如："不要修改未与你讨论的代码"、"不要执行会修改生产环境的命令"。
  • 协作协议 (Collaboration Protocol)：Agent的"我如何与他人配合"——何时将任务handoff给谁、如何请求帮助、如何汇报结果。这是多Agent系统运转的润滑剂。

反例——糟糕的角色设计：

避免这些错误
  • 职责重叠：两个Agent都能做同一件事，导致重复劳动或互相等待。
  • 能力不匹配：给CoderAgent配置了只有Reviewer才需要的工具，增加了安全风险。
  • 角色过于宽泛：一个Agent被定义为"Helper"——没有明确的专业领域，在实际执行中表现不稳定。
  • 缺乏协作协议：Agent不知道何时该把任务交给别人，导致任务卡住或产生错误结果。

5. 任务分配策略：谁来做这件事？

当一个任务进入多Agent系统，第一个要解决的问题就是"由谁来执行"。任务分配策略直接影响系统的效率和负载均衡：

5.1 静态分配 (Static Allocation)

通过预定义的规则将任务分配给特定Agent。例如："所有前端相关的任务都交给FrontendAgent"。规则可以是基于关键词匹配、任务类型标签、或用户显式指定。优点是简单、可预测、调试方便；缺点是缺乏灵活性——当FrontendAgent过载或失败时，系统无法自动调整。

5.2 动态分配 (Dynamic Allocation)

根据Agent的当前状态实时决定任务归属。调度器维护每个Agent的状态信息（空闲/忙碌、已完成任务数、平均处理时间、成功率等），选择最优Agent。常见策略包括：轮询(Round-Robin)、最少连接(Least Connection)、加权轮询(Weighted Round-Robin)。动态分配提高了资源利用率，但调度器本身成为关键路径和潜在瓶颈。

5.3 拍卖式分配 (Auction-Based)

受经济学启发——任务发布后，各个Agent根据自身能力和当前负载"出价"(Bid)，调度器选择"性价比"最高的出价。每个Agent的出价可以包含：预估完成时间、置信度评分、所需资源。这种方法能最大化整体系统效用，但在Agent数量多时通信开销较大。在多Agent强化学习(MARL)中，拍卖机制常被用于实现去中心化的任务分配。

策略 延迟 负载均衡 实现复杂度
静态分配 最低 差 低
动态分配 中等 较好 中等
拍卖式 较高 最优 高

6. 冲突解决：当Agent意见不一致时

多个Agent意味着多种观点。当Agent对同一问题给出不同答案时，系统如何裁决？冲突解决机制是多Agent系统可靠性的关键保障：

6.1 投票机制 (Voting)

最简单的冲突解决方式——少数服从多数。N个Agent(通常为奇数)各自独立给出答案，得票最多的答案胜出。投票适用于有明确对错的决策(如判断代码是否有bug)。变体包括：加权投票(资深Agent的票权重更高)、否决投票(安全Agent拥有一票否决权)、共识投票(需要超过某一阈值才算通过)。

6.2 辩论机制 (Debate)

两个持有不同观点的Agent进行多轮辩论：Agent A阐述观点与论据 → Agent B反驳并提出己方观点 → Agent A回应反驳 → 重复若干轮 → 裁判Agent(或人类)做出最终裁决。辩论机制的优势在于：它不仅能找出"哪个答案更好"，还能暴露每个答案背后的推理逻辑，产生更全面的分析。

6.3 仲裁机制 (Arbitration)

引入一个专门的仲裁Agent(或人类仲裁者)，听取各方意见后做出最终裁决。仲裁Agent通常拥有更高的权限和更全面的视角。关键设计：仲裁者自身如何保证公正？实践中常用方法有——仲裁者使用不同的底层模型(避免与参辩Agent同质化)、仲裁者看到所有辩论记录但不参与辩论、仲裁决策附带理由。

实践建议

对于安全相关的冲突（如安全Agent认为某操作危险而执行Agent认为安全），应始终偏向更保守的一侧——"宁可误报，不可漏报"。对于创意相关的冲突（如两个Agent给出不同的设计方案），可以将两个方案都呈现给用户，由用户最终决定。

7. Swarm架构深度解析：Handoff机制

Swarm架构由OpenAI在2024年提出，是一种轻量级的多Agent编排模式。其核心是Handoff(交接)机制——Agent在执行过程中，可以将对话控制权和上下文无缝转移给另一个更合适的Agent。Swarm强调去中心化、动态路由和最小框架开销。是目前最受关注的多Agent编排模式。与传统的层级架构不同，Swarm中的Agent是平等且自主的——没有中央调度器，Agent通过Handoff机制自行决定"现在该由谁来处理"。

Handoff机制的工作原理：

  1. 意图识别：当前Agent分析用户请求，判断是否需要Handoff。例如，通用Agent发现用户开始深入讨论数据库问题，意识到需要转给DBAgent。
  2. 目标选择：Agent从已知Agent列表中选择最适合的下一个Agent。选择依据包括：Agent的角色描述、当前对话的上下文、用户在对话中表达的需求。
  3. 上下文传递：将完整的对话历史、中间状态、已完成的步骤等信息打包传递给目标Agent。良好的上下文传递是Handoff成功的关键——信息遗漏会导致目标Agent"重新开始"，浪费时间和token。
  4. 控制转移：目标Agent接管对话，从传递的上下文继续推理和执行。原Agent退出当前会话。
  5. 返回Handoff（可选）：目标Agent完成任务后，可以将控制权交回原Agent，实现"借调"模式。

Handoff的三种模式：

  • 完全交接 (Full Handoff)：原Agent完全退出，目标Agent从头接管。适合任务性质发生根本变化的场景。
  • 借调 (Secondment)：原Agent暂时将任务委托给目标Agent，完成后返回。适合"帮我查一下某个专业信息"的场景。
  • 并行Handoff：原Agent将任务同时委托给多个专业Agent，汇总结果后继续处理。适合需要多维度信息的场景。
Swarm vs 传统编排

Swarm的核心哲学是"去中心化智能"——每个Agent只需关注自己的专业领域和何时应该Handoff，系统的整体智能从Agent之间的交互中涌现出来。与需要精心设计调度逻辑的层级架构相比，Swarm更适合动态、开放、难以预先规划的任务场景。

8. 完整代码示例：多Agent编排器

下面是一个可运行的多Agent编排系统，实现了Agent注册中心、消息路由器、和Handoff机制：

Python — 多Agent编排器完整实现 
from dataclasses import dataclass, field
from enum import Enum
from typing import Any, Callable, Optional
import uuid, json

class HandoffType(Enum):
    FULL = "full"           # 完全交接
    SECONDMENT = "secondment"  # 借调（完成后返回）

@dataclass
class AgentInfo:
    name: str
    role: str
    capabilities: list
    handler: Callable
    status: str = "idle"

@dataclass
class HandoffRequest:
    from_agent: str
    to_agent: str
    handoff_type: HandoffType
    context: dict
    return_agent: str = None

@dataclass
class Message:
    id: str
    sender: str
    receiver: str
    content: Any
    msg_type: str  # "task","result","handoff","query"

class AgentRegistry:
    """Agent注册中心——管理所有可用的Agent"""
    def __init__(self):
        self._agents: dict = {}

    def register(self, info: AgentInfo):
        self._agents[info.name] = info

    def find_by_capability(self, capability: str) -> list:
        return [a for a in self._agents.values()
                if capability in a.capabilities]

    def get(self, name: str) -> AgentInfo:
        return self._agents.get(name)

class MessageRouter:
    """消息路由器——Agent间的通信中枢"""
    def __init__(self, registry: AgentRegistry):
        self.registry = registry
        self.inboxes: dict = {}
        self.message_log: list = []

    def send(self, msg: Message):
        self.message_log.append(msg)
        inbox = self.inboxes.setdefault(msg.receiver, [])
        inbox.append(msg)

    def receive(self, agent_name: str) -> list:
        msgs = self.inboxes.get(agent_name, [])
        self.inboxes[agent_name] = []
        return msgs

class Orchestrator:
    """编排器——多Agent系统的总指挥"""
    def __init__(self):
        self.registry = AgentRegistry()
        self.router = MessageRouter(self.registry)
        self.shared_context = {}  # 共享黑板

    def execute_pipeline(self, agent_chain: list, task: Any):
        """顺序管道执行——A的输出作为B的输入"""
        current = task
        for agent_name in agent_chain:
            agent = self.registry.get(agent_name)
            if not agent:
                raise ValueError(f"Unknown agent: {agent_name}")
            agent.status = "busy"
            current = agent.handler(current, self.shared_context)
            agent.status = "idle"
            self.shared_context[f"{agent_name}_result"] = current
        return current

    def select_agent(self, requirement: str) -> AgentInfo:
        """根据需求动态选择最佳Agent"""
        candidates = self.registry.find_by_capability(requirement)
        # 优先选择空闲Agent，同状态选择第一个
        idle = [c for c in candidates if c.status == "idle"]
        return (idle or candidates)[0] if candidates else None

# ============ 使用示例 ============
orchestrator = Orchestrator()

# 注册三个专业Agent
orchestrator.registry.register(AgentInfo(
    "researcher", "研究员",
    ["research", "search"],
    lambda task, ctx: {"findings": "研究完成"}
))
orchestrator.registry.register(AgentInfo(
    "coder", "程序员",
    ["coding", "debugging"],
    lambda task, ctx: {"code": "实现完成"}
))
orchestrator.registry.register(AgentInfo(
    "reviewer", "审查员",
    ["review", "testing"],
    lambda task, ctx: {"review": "审查通过"}
))

# 执行流水线：研究 -> 编码 -> 审查
result = orchestrator.execute_pipeline(
    ["researcher", "coder", "reviewer"],
    {"task": "实现用户认证模块"}
)
print(f"最终结果: {json.dumps(result, ensure_ascii=False)}")

代码设计要点：

  • AgentRegistry：Agent注册中心是系统的"电话簿"——所有Agent在此注册其名称、角色、能力列表和处理函数。这使得系统可以动态发现和选择Agent，而不是硬编码Agent列表。
  • MessageRouter：消息路由器实现了收件箱模式——每个Agent有独立的消息队列，路由器负责投递和分发。消息日志提供完整的通信审计追踪。
  • Orchestrator.execute_pipeline：流水线执行引擎，将Agent链式串联——上一个Agent的输出自动成为下一个Agent的输入。同时维护shared_context作为共享黑板。
  • select_agent：动态Agent选择——根据能力需求找到匹配的Agent，优先选择空闲的。这是实现动态任务分配的基础。
  • shared_context：共享上下文字典扮演了黑板架构的角色，Agent可以在此读写中间结果。

9. 多Agent系统的实战经验与避坑指南

架构决策清单：

  • Agent粒度：一个Agent应该负责"一个可独立描述的专业领域"。太细(一个Agent只做一件事)导致通信开销过大；太粗(一个Agent做所有事)退化回单Agent。
  • Agent数量：对于大多数任务，3-7个Agent是最佳区间。超过10个Agent时，协调成本开始超过分工收益。可以通过"合并相似角色"和"引入子编排器"来控制Agent数量。
  • 通信频次：Agent之间不应每步都通信。设计原则是"只在有新的有价值信息时才通信"——避免空转和上下文膨胀。
  • 失败隔离：一个Agent的失败不应导致整个系统崩溃。设计原则：(1) 每个Agent有独立的错误处理；(2) 编排器有超时和重试策略；(3) 关键路径上的Agent应有热备。

常见陷阱：

  • 上下文雪崩：每个Agent往共享内存添加信息，上下文迅速膨胀到超出窗口限制。解决方法：上下文压缩——每个Agent只传递摘要而非完整输出，或使用滑动窗口保留最近N条记录。
  • 握手死锁：Agent A等待Agent B的结果，Agent B等待Agent A的确认。解决方法：设置超时、引入死锁检测、对于非关键依赖使用异步通信。
  • 回音室效应：多个Agent使用相同的模型和相似的系统prompt，导致它们给出相似的回答，丧失多视角优势。解决方法：为不同Agent使用不同的模型参数(temperature, top_p)或在prompt中注入不同的"人格"倾向。
  • 无限Handoff：在Swarm架构中，Agent A Handoff给B，B又Handoff给C，C又Handoff回A。解决方法：设置最大Handoff次数、追踪Handoff链、检测循环。
黄金法则

"不要因为能用多Agent就使用多Agent。" 如果一个单Agent能高效完成的任务，引入多Agent只会增加复杂度和成本。多Agent系统的最佳应用场景是：任务天然可分解、需要多种专业视角、或单Agent输出质量和可靠性不足的情况。
