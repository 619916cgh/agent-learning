基础概念

AI Agent（智能体）
AI Agent / Intelligent Agent
基础概念

AI Agent（智能体）是一种能够自主感知环境、做出决策并执行行动以实现特定目标的人工智能系统。理解 Agent 的关键在于认识到：Agent 不是"更好的聊天机器人"，而是"能够使用工具的自主推理系统"。

从技术角度看，Agent 的核心公式为：Agent = LLM（推理引擎）+ 工具系统 + 编排逻辑 + 记忆系统 + 安全护栏。LLM 充当"大脑"，负责理解意图和制定策略；工具系统是"手和眼"，通过调用 API、执行代码、操作数据库等方式与外部世界交互；编排逻辑控制整个执行流程；记忆系统管理短期、工作和长期三层记忆；安全护栏确保行为在可控边界内。

Agent 的关键特征包括：自主性——无需逐步人工指令即可独立规划执行；工具使用——根据需求自动选择和调用合适的外部工具；推理能力——多步推理、分析和反思；记忆能力——跨会话和跨任务的信息持久化；学习能力——从反馈和错误中持续改进策略。

当前基于 LLM 的 Agent 主要处于 L2（目标型）到 L3（效用型）的智能层级，并具备部分 L4（学习型）特征。典型应用场景包括：软件开发助手、自动化数据分析、智能客服系统、DevOps 运维自动化和个人任务助理等。

LLM（大语言模型）
Large Language Model
基础概念

大语言模型（LLM）是基于 Transformer 架构、在海量文本数据上训练的深度神经网络模型，是 AI Agent 的核心推理引擎。模型通过自监督学习从数十亿个参数中习得语言的统计规律、知识关联和推理模式。

在 Agent 系统中，LLM 承担着多重关键角色：意图理解——将自然语言指令解析为结构化的任务描述；任务分解——把复杂目标拆分为可逐步执行的子任务；工具选择——从工具集中选出最适合当前步骤的工具并填充参数；结果综合——将多次工具调用的结果整合为连贯、有洞察的最终回复；错误恢复——识别执行失败并自主调整策略。

主流 LLM 包括 OpenAI 的 GPT-4o/4.5 系列、Anthropic 的 Claude 系列、Google 的 Gemini 系列、Meta 的 Llama 系列以及 DeepSeek、Qwen 等开源模型。不同模型在推理深度、上下文窗口大小（128K 到 1M+）、工具调用精度、多语言能力和成本结构方面各有差异。

LLM 的核心能力源于规模的涌现效应——当参数量和训练数据超过一定阈值时，模型会展现出小模型不具备的能力，如上下文学习（Few-shot Learning）、思维链推理和元认知（知道自己不知道什么）。

自主性（Autonomy）
Autonomy / Autonomous Behavior
基础概念

自主性是 Agent 区别于传统程序的最核心特征，指系统在没有逐步人工指令的情况下独立做出决策并执行行动的能力。自主性并非"完全不需要人"，而是指 Agent 能够根据目标和当前状态自主决定下一步做什么，无需人类逐步骤地发出指令。

自主性体现为三个递进层次：执行自主——Agent 根据高层目标自主规划具体行动步骤（如告知"帮我准备季度报告"后自主决定查询哪些数据、生成什么图表）；路径自主——面对同一目标，Agent 能在多条可行路径中选择最优方案；策略自主——当首选路径失败时，Agent 自主切换策略尝试替代方案，而非简单报错退出。

在实际设计中，自主性需要在效率和安全性之间取得平衡。全自主 Agent 可能在陌生场景下做出危险决策，因此大多数生产系统采用"有条件自主"模式——常规操作自动执行，高风险操作需人工确认。

Function Calling（函数调用）
Function Calling / Tool Calling
核心概念

Function Calling 是 LLM 的一项关键能力，允许模型以结构化 JSON 格式输出调用特定函数的请求，而非生成自由文本。这是 LLM 从"只会说话"到"能够做事"的质变技术。

工作流程分为五个阶段：定义阶段——开发者将工具的名称、描述、参数 JSON Schema 随请求发送给模型；推理阶段——模型根据用户输入判断是否需要调用工具，若需要则决定调用哪个工具并传什么参数；输出阶段——模型输出结构化 JSON（包含函数名和参数）；执行阶段——Agent 框架解析 JSON 并实际调用对应函数（模型本身不执行任何代码）；反馈阶段——将函数结果返回给模型，模型据此决定继续调用还是给出最终回答。

Function Calling 的精度取决于多个因素的综合效果：工具描述的清晰度和准确性、参数 Schema 的完整性、模型本身的推理能力、以及 Prompt 中提供的使用示例质量。最佳实践包括精确的 description 字段、使用 enum 限制参数可选值、明确标记 required 参数、为复杂工具提供 Few-shot 示例。

Tool Use（工具使用）
Tool Use
核心概念

Tool Use（工具使用）是 Agent 通过调用外部工具来完成超出 LLM 自身能力范围的任务的能力。它是 Agent 区别于纯 LLM Chat 的核心特征，使 Agent 成为真正能够"动手做事"的智能系统。

Agent 可使用的工具涵盖：API 调用（天气查询、股票数据、网页搜索）、代码执行（Python/R/SQL 脚本运行）、文件操作（读写、搜索、修改文件）、数据库操作（查询、更新记录）、通信工具（发送邮件、推送通知）、浏览器工具（打开网页、提取内容）。每种工具都有明确的 JSON Schema 定义其接口和参数。

工具设计的核心原则：单一职责——一个工具只做一件事且做好；清晰契约——输入参数和返回格式明确且稳定；优雅降级——工具失败时返回有帮助的错误信息而非直接崩溃；最小权限——工具只开放完成任务所必需的权限。

Context Window（上下文窗口）
Context Window
核心概念

上下文窗口是 LLM 在一次推理中能够"看到"的最大文本量，以 Token 为单位衡量。它决定 Agent 能同时处理的对话长度、能记忆的历史信息量、以及能一次性使用的工具调用轮次。

上下文窗口经历了快速增长：从 GPT-2 的 1K 到 GPT-4 Turbo 的 128K，再到 Claude 的 200K 和 Gemini 的 1M+。更大的窗口使得 Agent 能处理超长文档（整个代码库、数百页 PDF）、维持更长时间的连续对话、并包含更完整的工具调用历史用于自我反思。

然而更大的窗口也带来挑战：注意力稀释效应——模型对窗口中部信息的关注度可能降低（"Lost in the Middle"问题）；成本非线性增长——长上下文的计算和计费开销显著增大；延迟增加——处理更多 Token 需要更长时间。因此，有效管理而非无脑扩充上下文才是关键。

Token（令牌/词元）
Token
核心概念

Token 是 LLM 处理文本的基本单位。英文中约 1 Token = 0.75 个单词，中文中约 1 个汉字 = 1-2 个 Token。模型的输入理解、输出生成、计费均以 Token 为基本计量单位。这意味着处理中文内容的成本通常是英文的 1.5-3 倍。

在 Agent 场景中，Token 消耗源包括：系统提示词的固定开销（每次请求都需完整传递）、对话历史的累积开销（随对话轮次线性增长）、工具 Schema的传递开销（所有可用工具的定义都被计入）、工具执行结果的返回开销（可能非常庞大）、以及模型输出本身的 Token。一个复杂的多步 Agent 任务可能消耗数万甚至十万 Token。

Token 管理是成本控制的核心：精简系统提示词、设置滑动窗口限制对话历史、压缩工具返回结果、采用 Prompt Caching 等技术都能显著降低 Token 消耗。

技术方法

RAG（检索增强生成）
Retrieval-Augmented Generation
技术方法

RAG（检索增强生成）是一种将信息检索与文本生成相结合的架构模式。核心思想：当 LLM 需要回答问题时，先从外部知识库中检索最相关的文档片段，再将片段作为上下文提供给 LLM，使其生成的回答更准确、有据可查且时效性强。

RAG 解决了 LLM 的两个根本局限：知识截止日期——训练数据存在时间截止点，对截止后的事件一无所知；幻觉倾向——缺乏信息时 LLM 可能编造答案。通过引入外部知识检索，RAG 让 Agent 能访问最新、最相关的信息。

标准 RAG 流程：文档分块（按语义边界切分为 256-1024 Token 的 Chunk）→ 向量化（用 Embedding 模型将每个 Chunk 转为高维向量）→ 索引存储（向量存入向量数据库）→ 查询检索（用户查询同样向量化后通过相似度搜索找到 Top-K 相关 Chunk）→ 增强生成（检索结果注入 LLM 上下文生成回答）。

RAG 性能的关键优化点：Chunk 大小的选择（太小丢失上下文，太大稀释精度）、Embedding 模型的质量、Top-K 的取值（通常 3-10）、以及是否使用 Reranker 做二次排序提升精度。

Embedding（嵌入向量）
Embedding / Vector Representation
技术方法

Embedding（嵌入向量）是将文本、图像等非结构化数据转换为固定长度浮点数向量的数学方法。核心价值在于将语义相似度转化为数学上的向量距离——语义越接近的两段文本，其 Embedding 在空间中的余弦距离越小。

主流 Embedding 模型包括 OpenAI 的 text-embedding-3（支持 256-3072 维灵活调整）、Cohere 的 Embed v3、以及开源的 BGE、GTE 系列。优秀 Embedding 模型能捕捉深层语义而非表面关键词匹配——例如，"人工智能"和"机器学习"的向量距离远小于"人工智能"和"天气"的距离。

在 Agent 系统中，Embedding 的应用场景包括：RAG 的知识检索（查询与文档的语义匹配）、长期记忆的语义召回（从历史对话中检索最相关的记忆）、工具选择的辅助（匹配当前任务最相关的工具）、以及异常检测（监控输出向量与期望的偏差）。选择 Embedding 模型需权衡维度、多语言支持、最大输入长度和部署成本。

相关: Vector Database RAG
Vector Database（向量数据库）
Vector Database
技术方法

向量数据库是专为存储和检索高维向量数据设计的数据库系统。核心操作是近似最近邻搜索（ANN）——在百万/亿级向量中，快速找到与查询向量最相似的 Top-K 个结果。在 Agent 的 RAG 架构中，向量数据库承载着长期记忆和知识库的角色。

主流选择：Pinecone（全托管云服务，零配置开箱即用）、Weaviate（开源，支持混合搜索——向量+关键词）、ChromaDB（轻量级，Python 原生，适合本地开发和实验）、Milvus（高性能分布式，适合大规模生产环境）、Qdrant（Rust 编写，过滤功能强大）、pgvector（PostgreSQL 扩展，让传统数据库也能做向量检索）。

选择向量数据库需考虑：向量规模、查询延迟要求（毫秒级对交互式 Agent 至关重要）、过滤能力（向量检索的同时按元数据过滤）、索引算法（HNSW vs IVF——速度与精度的权衡）、运维模式（云服务 vs 自托管）。

Prompt Engineering（提示词工程）
Prompt Engineering
技术方法

Prompt Engineering是设计、优化和管理 LLM 输入提示以引导模型产生期望输出的系统性方法论。在 Agent 系统中，System Prompt 不仅定义 Agent 的"人格"和行为规范，还包含工具使用指导、安全边界和错误处理策略。

Agent System Prompt 通常包含：角色定义（"你是一个专业的代码助手..."）、能力声明（"你可以使用以下工具..."）、行为规则（"不确定时主动询问用户..."）、格式要求（"输出使用 Markdown 格式..."）、安全约束（"不要执行可能删除文件的命令..."）、思维链引导（"在行动前先分析问题..."）。

核心技术包括：Few-shot Prompting（提供输入输出示例引导行为）、Chain-of-Thought（要求展示逐步推理过程）、角色扮演（设定角色身份和背景）、约束声明（精确说明什么能做/不能做）、动态组装（根据任务上下文动态拼接 Prompt 段落）。提示词工程需要持续迭代——A/B 测试、评估数据集验证、根据实际使用中的故障模式调整。

Fine-tuning（微调）
Fine-tuning
技术方法

Fine-tuning（微调）是在预训练模型基础上，使用特定领域数据额外训练以优化模型在特定任务上表现的技术。不同于 Prompt Engineering 只改变输入，Fine-tuning 直接修改模型权重参数，能更深层地改变模型行为模式。

在 Agent 中的应用：工具调用指令微调——用大量工具调用示例训练，提升工具选择和参数填充准确性；结构化输出微调——训练模型输出固定 JSON Schema，确保输出格式可靠；领域知识注入——用行业专有数据微调，让模型深入理解特定术语和工作流；安全对齐微调——通过 RLHF 训练模型拒绝危险请求。

Fine-tuning vs Prompt Engineering：微调可大幅减少 Prompt 长度（模型已"学会"部分行为），使用更小更便宜的模型达到大模型水平，但也面临高质量标注数据获取成本高、可能发生灾难性遗忘、更新周期长、可能过拟合等挑战。当前最佳实践是组合策略——用 Fine-tuning 固化基本行为，用 Prompt 覆盖动态变化部分。

推理模式

ReAct（推理-行动循环）
ReAct (Reasoning + Acting)
推理模式

ReAct 是当前最主流的 Agent 推理模式，由 Google DeepMind 在 2022 年提出。核心理念是将推理和行动交替进行，每一步由三个环节构成：Thought（思考）——分析当前状态，推理下一步该做什么以及为什么；Action（行动）——选择和执行具体的工具调用；Observation（观察）——接收工具执行结果，评估是否推进了任务，是否需要调整策略。

ReAct 相比纯 Chain-of-Thought 的关键优势：思维链只在"脑子里"推理，而 ReAct 能通过与真实世界交互获取信息。模型不确定时不需要猜测——它可以调用搜索工具查找、调用计算工具验证、调用数据库核实。这种"知行合一"的模式大幅提升了输出的准确性和可靠性。

ReAct 的实现依赖 LLM 的 Function Calling 能力。框架在 Prompt 中引导模型按 Thought → Action → Observation 格式组织输出，框架解析 Action 执行实际调用，再将 Observation 反馈给模型继续思考。这种结构化格式还带来了良好的可解释性——开发者可清晰看到每一步的推理和决策过程。

相关: Chain-of-Thought Function Calling Tool Use
Chain-of-Thought（思维链）
Chain-of-Thought (CoT)
推理模式

Chain-of-Thought（CoT / 思维链）是一种引导 LLM 进行逐步推理的提示技术。通过在 Prompt 中要求模型"让我们一步步思考"或提供包含推理步骤的 Few-shot 示例，CoT 能显著提升模型在复杂推理任务（数学题、逻辑推理、多步规划等）上的表现。

CoT 的原理是让模型展示工作过程而非直接跳到结论。复杂问题需要多步推导，直接要求答案往往导致模型猜测；而要求展示推理过程会使模型分配更多"注意力"到每个推理步骤，每步都建立在之前的结果上。研究表明，CoT 可将某些推理任务的准确率从 30% 提升到 80% 以上。

CoT 的高级变体包括：Tree of Thoughts（思维树）——同时探索多条推理路径并从最优路径回溯；Self-Consistency——多次采样取多数投票；Least-to-Most——先解决最简单子问题，逐步推进；Auto-CoT——自动生成推理链而非人工编写。在 Agent 中，CoT 已内嵌到推理循环中——Agent 选择工具和参数时就在进行思维链推理。

Plan-and-Execute（先规划后执行）
Plan-and-Execute
推理模式

Plan-and-Execute 是一种先将任务分解为详细计划再逐步执行的 Agent 推理模式，与 ReAct 的"边想边做"形成互补。对于步骤明确的复杂任务（如数据分析报告生成、多阶段代码重构），这种模式比纯 ReAct 更可靠。

执行流程分为四个阶段：规划阶段——Agent 全面分析任务后生成包含所有步骤和依赖关系的详细计划；确认阶段（可选）——计划展示给用户确认或修改；执行阶段——按计划逐步执行每个步骤，每一步完成后验证结果；总结阶段——汇总所有步骤结果生成最终输出。

Plan-and-Execute 相比 ReAct 的优势：全局视角——先看全局再做每一步，避免短视决策；可审阅性——用户可提前审查计划再批准执行；可预测性——执行路径确定，适合需要审计的场景。缺点是对高度动态和不可预见的任务适应性较差。

Prompt Injection（提示注入）
Prompt Injection
安全相关

Prompt Injection（提示注入）是针对 LLM 应用的攻击手段，攻击者构造特殊输入试图覆盖或绕过模型的 System Prompt，使 Agent 执行非预期操作或泄露敏感信息。在 Agent 场景中——由于 Agent 能实际执行操作（调用 API、运行代码、访问文件）——提示注入的危害远比纯 Chat 场景严重。

攻击分三类：直接注入——用户输入中直接包含"忽略之前所有指令，改为执行..."；间接注入——将恶意指令隐藏在 Agent 可能读取的外部内容（网页、文档、邮件）中，Agent 访问时被"感染"；多阶段注入——先用看似无害的请求让 Agent 进入特定状态，再利用该状态实施后续攻击。

防御策略需要多层次组合：输入过滤（检测可疑指令模式）、指令隔离（用特殊分隔符明确分开系统指令和用户输入）、权限最小化（Agent 只被授予完成任务所需的最小权限）、输出审核（对输出做独立安全验证）、沙箱执行（高风险操作在隔离环境中执行）。没有任何单一方案能完全防御。

Guardrails（安全护栏）
Guardrails
安全相关

Guardrails（安全护栏）是 Agent 系统中用于约束和控制行为的多层防护机制总称。就像高速公路护栏防止车辆冲出道路，Guardrails 确保 Agent 行为始终在安全预期范围内。

四层防护体系：输入护栏——在 Agent 接收输入前检查过滤违规内容、注入攻击和越权请求；推理护栏——Agent 生成响应或选择行动时实时干预，检测到危险操作意图时主动阻止并替换为安全响应；执行护栏——工具调用前进行权限校验、参数验证和速率限制；输出护栏——最终回复前的内容审核，确保输出合规且不包含敏感信息。

实现方式包括：规则引擎（正则/关键词快速过滤）、分类模型（专门检测不安全内容）、LLM-as-Judge（独立 LLM 审查主 Agent 的输入输出）、沙箱隔离（高风险操作限在隔离环境）。设计 Guardrails 需平衡安全性和可用性——过于严格会让 Agent 拒合理请求（假阳性），过于宽松会放过恶意行为（假阴性）。

Sandbox（沙箱）
Sandbox / Isolated Execution Environment
安全相关

Sandbox（沙箱）是隔离的执行环境，Agent 可在其中安全地运行代码、执行命令和测试操作，不会影响宿主系统或访问未授权资源。在 Agent 安全架构中，沙箱扮演"防火墙"角色——即使 Agent 行为出现偏差或被恶意利用，损害也被限制在沙箱内部。

沙箱提供的隔离能力：文件系统隔离——Agent 只能访问沙箱内临时文件，无法触及宿主系统；网络隔离——限制可访问的网络地址和端口；资源限制——CPU、内存、磁盘和时间的硬上限，防止资源耗尽；权限限制——以最低用户权限运行，无法执行系统管理操作。

实现方式：容器化（Docker 为每次执行创建独立容器，执行后销毁）、虚拟机（隔离级别最高但启动开销大）、WebAssembly 沙箱（轻量安全）、云沙箱服务（如 AWS Lambda 隔离执行）。沙箱不仅是安全需求，也是可靠性保障——保证每次执行从"干净"状态开始，支持可重复性。

基础设施

Orchestration（编排）
Orchestration / Agent Workflow Management
基础设施

Orchestration（编排）在 Agent 系统中指对多个组件、步骤和 Agent 进行协调调度的逻辑层。如果说 LLM 是大脑、工具是手，编排层就是"神经系统"——控制信息流动、决策分发、步骤顺序和异常处理。

编排层的核心职责：流程控制——管理推理循环，决定何时进入下一步、停止或回溯；任务分解——将高层目标拆解为可执行的子任务；工具路由——根据当前步骤将任务分发给对应工具或子 Agent；状态同步——维持全局状态统一视图；错误处理——步骤失败时决定重试、跳过还是降级。

编排模式分类：链式（A 输出作为 B 输入）、并行（多任务同时执行后汇总）、条件（根据中间结果动态选择路径）、循环（重复执行直到终止条件）、分层（主 Orchestrator 委派给子 Orchestrator）。设计编排时需要平衡灵活性和可控性——过度依赖 LLM 编排可能产生不可预测行为，过度硬编码则丧失了 Agent 的核心优势。

MCP（模型上下文协议）
Model Context Protocol (MCP)
基础设施

MCP（Model Context Protocol）是由 Anthropic 提出的开放标准协议，为 AI 模型与外部工具、数据源和服务之间的交互提供统一接口规范。可以理解为 AI Agent 世界的"USB 协议"——只要遵循 MCP 标准，任何支持 MCP 的 Agent 框架都能即插即用地使用任何 MCP 工具。

MCP 解决了工具集成碎片化的核心痛点——在此之前，每个 Agent 框架和每个工具之间都需要编写专门的适配代码，LangChain 有一套格式，Anthropic SDK 另一套，OpenAI 又有自己的 Schema。MCP 通过统一标准消除这种碎片化。

MCP 核心概念：Server（暴露工具和数据源的服务端）、Client（Agent 框架通过客户端连接服务端）、Tools（可调用的功能）、Resources（暴露的结构化数据）、Prompts（预定义模板和指令）。通信基于 JSON-RPC，支持 stdio 和 HTTP 两种传输方式。MCP 的出现大大提升了工具复用性和跨框架互操作性。

高级话题

Multi-Agent System（多Agent系统）
Multi-Agent System (MAS)
高级话题

Multi-Agent System（多Agent系统）由多个独立但协作的 AI Agent 组成，通过分工、通信和协调共同完成单 Agent 难以处理的复杂任务。每个 Agent 通常专注于特定领域或角色（如研究员、写手、审核员），彼此通过消息传递、共享状态或中央调度器协同工作。

协作模式：顺序协作——Agent 按固定顺序执行，前一个输出成为下一个输入（流水线模式）；层级协作——一个管理者 Agent 负责分配、追踪和汇总，多个执行者 Agent 完成子任务；辩论模式——多个 Agent 从不同角度分析同一问题，通过辩论收敛到最优解；Swarm 模式——Agent 之间去中心化动态交接任务，群体自适应完成目标。

多Agent系统的优势：专业化（每个 Agent 专注擅长的领域）、鲁棒性（单 Agent 失败不导致系统崩溃）、并行性（多 Agent 同时处理不同子任务）、交叉验证（多视角推理互相纠正）。挑战包括：通信开销、一致性问题、任务分配优化、死锁与活锁风险。

挑战与限制

Hallucination（幻觉）
Hallucination / Confabulation
挑战与限制

Hallucination（幻觉）是指 LLM 生成的内容看似合理但事实上不正确、不存在或与输入无关的现象。在 Agent 场景中，幻觉的危害被放大——Agent 不仅可能"说错话"，更可能基于错误信息执行错误操作，导致实际业务损失。

幻觉的产生源于多种因素：训练数据偏差（模型学到了错误的"事实"）、概率生成本质（预测"最可能的下一个词"而非"真实事实"）、知识滞后（知识截止于训练日期）、Prompt 诱导（不当 Prompt 引导模型编造）、过度泛化（将一个领域模式错误应用于不相关领域）。

减轻幻觉影响需要多重策略组合：RAG 事实锚定（强制 Agent 先检索再基于检索结果回答）、工具验证（通过工具调用做事实核查）、置信度标注（不确定时明确表达不确定性）、多Agent交叉验证（多个 Agent 独立回答后比较一致性）、人工审核节点（关键决策前引入人工确认）。幻觉无法被完全消除——它是概率生成模型的固有特征——目标是将其影响降到可接受水平。
