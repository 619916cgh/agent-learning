📡 API 与协议集成深度指南

10 大核心技术专题 · REST/GraphQL/gRPC/SDK · MCP/Webhook/SSE · 认证与限流

一个 AI Agent 的能力边界，很大程度上取决于它所集成的外部系统。API、SDK、通信协议是 Agent 感知世界和执行行动的"神经系统"。本指南深入覆盖 Agent 开发中所有的集成范式——从经典的 REST API 到现代的 MCP 协议，从 Anthropic/OpenAI SDK 到流式处理和速率限制。每个专题不仅解释概念，更提供可直接用于生产环境的完整代码示例。无论你是在构建第一个 Agent 工具，还是在设计企业级 Agent 集成架构，这里都是你的技术地图。

🌐 REST API 集成深度指南

HTTP Methods · Headers · Pagination · Rate Limiting · Response Filtering

REST (Representational State Transfer) 是 Agent 最常集成的 API 范式。Agent 通过标准 HTTP 协议与外部服务通信，使用 GET 获取数据、POST 创建资源、PUT/PATCH 更新资源、DELETE 删除资源。对于 Agent 开发者来说，理解 REST 不仅仅是会发 HTTP 请求，更重要的是掌握 API 设计中影响 Agent 可靠性的关键要素：如何优雅处理网络失败、如何高效分页获取大数据集、如何在 API 限流下生存，以及如何将 API 返回的冗余数据压缩到 LLM 能够有效处理的 Token 量级。

在为 Agent 设计 REST API 工具时，需要关注几个核心设计原则。幂等性：对于因为网络原因不确定是否成功的操作（如 POST 创建订单），应使用幂等键（Idempotency-Key）确保重试不会创建重复资源。响应压缩：API 返回的大型 JSON 响应不应完整塞入 LLM 上下文——应在 Agent 工具层预先过滤，仅保留相关字段。错误码分类处理：4xx 错误通常来自 Agent 的错误输入，不应重试（需要 LLM 修正参数后重新调用）；5xx 错误是服务端问题，应采用指数退避重试；429 表示限流，需读取 Retry-After 头精确等待。

分页模式是 REST 集成中另一个关键话题。Offset-based pagination（offset/limit 参数）最简单但不可靠——数据在两次分页请求间可能增删，导致重复或遗漏。Cursor-based pagination（返回 next_cursor）更适合 Agent 场景：每次请求返回一个不透明的游标，下次请求从该游标继续，保证不会重复或遗漏数据。Page-based pagination（page/page_size）介于两者之间，适合数据相对稳定的场景。Agent 工具应抽象掉分页细节——对外暴露为简单的"获取所有匹配记录"，内部自动遍历所有分页直到 next_cursor 为空或达到结果上限。

速率限制是生产环境中 Agent 工具面临的最大挑战之一。API 提供商通常以"每分钟请求数 (RPM)"或"每秒请求数 (RPS)"为单位实施限流。当 Agent 在推理循环中频繁调用工具时，很容易触及限制。实现策略包括：预检机制（调用前检查本地速率计数器，超过阈值则等待）、自适应降级（接近限额时切换到更精简的工具调用模式）、请求合并（将多个独立查询合并为批量查询 API）。关键是在返回 LLM 的错误信息中明确说明"因速率限制，请稍后重试"而非模糊的"调用失败"，让 LLM 能做出正确的重试决策。

Agent 工具返回结果的大小需要仔细管理。一个未过滤的 API 响应可能轻松消耗数千 Token——而这些 Token 中大部分对 Agent 的任务无实际帮助。最佳实践：在工具函数中进行结果过滤（仅保留 schema 中标记为 relevance: "high" 的字段）、设置结果最大长度限制（超出部分截断并告知 LLM"结果已截断，可调整查询参数缩小范围"）、在工具描述中鼓励 LLM 使用精确的查询条件减少不需要的数据。对于需要展示全部数据的场景，应考虑使用文件引用而非内联传输完整数据。

REST API 集成的关键决策点

何时使用 GET vs POST：GET 请求可缓存且幂等，适合纯查询操作；POST 用于创建资源和复杂查询（查询参数超出 URL 长度限制时）。何时使用同步 vs 异步：执行时间超过 1 秒的操作应提供异步模式（返回 task_id，后续通过 polling 或 webhook 获取结果），避免 Agent 长时间等待。何时使用 SDK vs 直接 HTTP：官方 SDK 通常处理好了认证、重试、分页等细节——优先使用 SDK，仅在 SDK 不存在或受限时直接调用 HTTP。

import time
import json
import hashlib
import requests
from functools import wraps
from typing import Optional, Dict, Any, Callable

class RateLimiter:
    """滑动窗口速率限制器"""
    def __init__(self, max_requests: int, window_seconds: float = 60.0):
        self.max_requests = max_requests
        self.window_seconds = window_seconds
        self.requests = []

    def acquire(self) -> float:
        now = time.time()
        self.requests = [t for t in self.requests if t > now - self.window_seconds]
        if len(self.requests) >= self.max_requests:
            wait = self.requests[0] + self.window_seconds - now + 0.1
            return max(0, wait)
        self.requests.append(now)
        return 0

class RESTTool:
    """Agent REST API 工具包装器"""

    def __init__(self, base_url: str, api_key: str,
                 max_retries: int = 3, rpm: int = 60,
                 max_result_tokens: int = 1500):
        self.base_url = base_url
        self.session = requests.Session()
        self.session.headers.update({
            "Authorization": f"Bearer {api_key}",
            "Content-Type": "application/json",
            "Accept": "application/json",
        })
        self.max_retries = max_retries
        self.rate_limiter = RateLimiter(max_requests=rpm)
        self.max_result_tokens = max_result_tokens

    def _filter_response(self, data: Any, schema: Dict) -> Any:
        """过滤API响应，仅保留高相关性字段"""
        relevant_fields = {
            k for k, v in schema.get("properties", {}).items()
            if v.get("relevance") == "high"
        }
        if isinstance(data, dict):
            return {
                k: self._filter_response(v, schema.get("properties", {}).get(k, {}))
                for k, v in data.items() if k in relevant_fields
            }
        elif isinstance(data, list):
            items_schema = schema.get("items", {})
            return [self._filter_response(item, items_schema) for item in data]
        return data

    def _estimate_tokens(self, text: str) -> int:
        """粗略估算 token 数量（中文 1 字约 1.5 token，英文 1 词约 1.3 token）"""
        return len(text.encode("utf-8")) // 2

    def _truncate_if_needed(self, result: str) -> str:
        """截断过长的结果并附加提示"""
        if self._estimate_tokens(result) <= self.max_result_tokens:
            return result
        truncated = result[: self.max_result_tokens * 4]
        return truncated + "\n[结果已截断 - 请缩小查询范围获取完整数据]"

    def call(self, method: str, path: str,
             params: Optional[Dict] = None,
             body: Optional[Dict] = None,
             idempotency_key: Optional[str] = None,
             response_schema: Optional[Dict] = None) -> Dict[str, Any]:
        """
        执行API调用，内置重试、限流、响应过滤
        返回格式: {"success": bool, "data": ..., "error": ..., "metadata": {...}}
        """
        headers = {}
        if idempotency_key:
            headers["Idempotency-Key"] = idempotency_key

        for attempt in range(self.max_retries + 1):
            # 速率限制
            wait = self.rate_limiter.acquire()
            if wait > 0:
                time.sleep(wait)

            try:
                resp = self.session.request(
                    method=method,
                    url=f"{self.base_url}{path}",
                    params=params,
                    json=body,
                    headers=headers,
                    timeout=30,
                )

                # 4xx 错误 - Agent 输入问题，不重试
                if 400 <= resp.status_code < 500:
                    return {
                        "success": False,
                        "error": json.dumps(resp.json(), ensure_ascii=False),
                        "status_code": resp.status_code,
                        "message": (
                            f"请求参数有误({resp.status_code})。"
                            f"请检查参数格式并重新调用。错误: {resp.text[:300]}"
                        ),
                    }

                # 429 - 速率限制
                if resp.status_code == 429:
                    retry_after = int(resp.headers.get("Retry-After", 5))
                    if attempt < self.max_retries:
                        time.sleep(retry_after)
                        continue
                    return {
                        "success": False,
                        "error": "rate_limited",
                        "message": "API 速率限制已达上限，请等待后重试。",
                    }

                # 5xx - 服务端错误，重试
                if resp.status_code >= 500:
                    if attempt < self.max_retries:
                        backoff = (2 ** attempt) + (time.time() % 1)
                        time.sleep(backoff)
                        continue
                    return {
                        "success": False,
                        "error": f"server_error_{resp.status_code}",
                        "message": f"服务端错误({resp.status_code})，已重试{self.max_retries}次仍失败。",
                    }

                # 成功
                data = resp.json()
                if response_schema:
                    data = self._filter_response(data, response_schema)

                return {
                    "success": True,
                    "data": data,
                    "metadata": {
                        "status_code": resp.status_code,
                        "request_url": resp.url,
                        "retries": attempt,
                    },
                }

            except (requests.ConnectionError, requests.Timeout) as e:
                if attempt < self.max_retries:
                    time.sleep(2 ** attempt)
                    continue
                return {
                    "success": False,
                    "error": "network_error",
                    "message": f"网络连接失败，已重试{self.max_retries}次。请检查API地址是否正确。",
                }

    def paginate_all(self, path: str, params: Dict,
                     max_results: int = 500) -> list:
        """遍历所有分页，收集全部结果"""
        all_items = []
        cursor = None
        page = 0

        while len(all_items) < max_results:
            page_params = {**params}
            if cursor:
                page_params["cursor"] = cursor
            page_params["limit"] = min(100, max_results - len(all_items))

            result = self.call("GET", path, params=page_params)
            if not result["success"]:
                break

            data = result["data"]
            items = data.get("data", data.get("results", []))
            all_items.extend(items)

            cursor = data.get("next_cursor")
            if not cursor or not items:
                break
            page += 1

        return all_items[:max_results]

📊 GraphQL 集成

精确数据获取 · Schema 内省 · 查询构建 · Nested Queries · DataLoader 批处理

GraphQL 为 Agent 工具集成提供了一种与 REST 完全不同的数据获取范式。在 REST 模型中，Agent 调用一个固定端点，接收服务端预定义的完整响应——可能包含大量无关字段。在 GraphQL 模型中，Agent 在查询中精确声明所需字段，服务端仅返回这些字段，不会多也不会少。这种"按需取数"的特性对 LLM 上下文窗口极为友好——Agent 只为真正需要的信息付出 Token 成本。对于需要从复杂嵌套数据结构中提取特定信息的 Agent 来说，GraphQL 能节省 50-80% 的响应体积。

GraphQL 最强大的特性之一是 Schema 内省 (Introspection)。通过查询 __schema 元字段，Agent 可以动态发现 API 的完整类型系统——有哪些类型、每个类型有哪些字段、字段类型是什么、哪些参数是必填的。这意味着 Agent 可以在运行时自动生成 GraphQL 查询，无需开发者硬编码查询模板。这种"自发现"能力让 Agent 能够适应 API 的变化——API 新增字段后，Agent 只需重新内省即可获知。这是 REST API（需要开发者提前知道端点结构）所不具备的动态能力。

GraphQL 查询构建策略对 Agent 来说是个有趣的挑战。让 LLM 直接生成 GraphQL 查询字符串是可行的，但有风险——LLM 可能生成语法错误的查询或请求不存在的字段。更稳健的策略是 Schema-Guided Query Construction（基于 Schema 引导的查询构建）：Agent 先从内省结果中提取可用字段列表，将这些字段发送给 LLM，LLM 选择需要的字段，Agent 框架再根据选择构建合法的 GraphQL 查询。这种"人机协作"模式既利用了 LLM 的语义理解能力（"哪些字段与用户的查询相关"），又保证了查询的语法正确性。

对于涉及多实体关联的复杂查询，GraphQL 的 嵌套查询 能力可以大幅减少请求次数。REST 模式下获取"用户的订单及其订单项"可能需要 3 次 API 调用（获取用户 → 获取订单列表 → 逐个获取订单项），而 GraphQL 在单次请求中即可完成。不过这也带来了 N+1 查询问题——GraphQL resolver 可能对每个子对象独立查询数据库。企业级 GraphQL 服务通常使用 DataLoader 模式解决此问题：将同一请求中的多个独立数据加载操作自动合并为批量查询，减少数据库往返次数。

在实践中，GraphQL 和 REST 并非相互排斥的替代品。许多现代 API 架构同时提供两种接口——REST 用于简单 CRUD 操作，GraphQL 用于需要灵活字段选择的复杂数据聚合。Agent 工具设计时应评估每个数据源的特点：数据字段多且 Agent 只需其中少数 → GraphQL 优势明显；API 结构简单且 Agent 总是需要完整数据 → REST 更简单高效；需要流式实时数据 → 优先考虑 WebSocket 或 SSE。关键原则是：让数据获取方式适配 Agent 的需求，而不是让 Agent 忍受数据获取方式的限制。

维度GraphQLREST
响应体积精确控制，仅返回请求字段服务端预定义，可能包含冗余数据

工具发现通过内省动态发现所有数据能力需预知端点（或通过OpenAPI spec）

学习曲线中等（需理解类型系统和查询语法）低（标准化程度高，文档丰富）

Python
import json
import requests
from typing import Dict, List, Optional, Any
from datetime import datetime


class GraphQLIntrospectionTool:
    """Schema 内省驱动的动态 GraphQL 查询工具"""

    INTROSPECTION_QUERY = """
    query IntrospectType($name: String!) {
      __type(name: $name) {
        name
        description
        fields(includeDeprecated: false) {
          name
          description
          type { name kind ofType { name kind ofType { name kind } } }
          args { name description type { name kind } }
        }
        inputFields { name description type { name kind } defaultValue }
      }
    }
    """

    SCHEMA_QUERY = """
    query {
      __schema {
        queryType { name fields { name description } }
        mutationType { name fields { name description } }
        types {
          name kind description
          fields { name description type { name kind } }
        }
      }
    }
    """

    def __init__(self, endpoint: str, api_key: Optional[str] = None):
        self.endpoint = endpoint
        self.headers = {"Content-Type": "application/json"}
        if api_key:
            self.headers["Authorization"] = f"Bearer {api_key}"
        self._schema_cache: Optional[Dict] = None

    def _execute(self, query: str, variables: Dict = None) -> Dict:
        """执行 GraphQL 查询"""
        resp = requests.post(
            self.endpoint,
            json={"query": query, "variables": variables or {}},
            headers=self.headers,
            timeout=15
        )
        resp.raise_for_status()
        data = resp.json()
        if "errors" in data:
            raise ValueError(f"GraphQL Error: {json.dumps(data['errors'], ensure_ascii=False)}")
        return data["data"]

    def introspect_schema(self) -> Dict:
        """获取完整 Schema 信息（自动缓存）"""
        if self._schema_cache:
            return self._schema_cache
        self._schema_cache = self._execute(self.SCHEMA_QUERY)
        return self._schema_cache

    def get_type_fields(self, type_name: str) -> List[Dict]:
        """获取指定类型的所有字段及其描述"""
        introspection = self._execute(
            self.INTROSPECTION_QUERY,
            {"name": type_name}
        )
        if not introspection.get("__type"):
            return []
        return introspection["__type"].get("fields", [])

    def build_field_options_for_llm(self, type_name: str) -> str:
        """生成可供 LLM 做字段选择的文本"""
        fields = self.get_type_fields(type_name)
        lines = [f"Type: {type_name} 可用字段："]
        for f in fields:
            desc = f.get("description") or "(无描述)"
            type_info = f.get("type", {})
            type_str = self._type_to_string(type_info)
            lines.append(f"  - {f['name']} ({type_str}): {desc[:120]}")
        return "\n".join(lines)

    def _type_to_string(self, type_info: Dict) -> str:
        """递归解析类型嵌套结构"""
        kind = type_info.get("kind", "SCALAR")
        if kind == "NON_NULL":
            return f"{self._type_to_string(type_info.get('ofType', {}))}!"
        if kind == "LIST":
            return f"[{self._type_to_string(type_info.get('ofType', {}))}]"
        return type_info.get("name", "Unknown")

    def build_and_execute(self, type_name: str,
                          operation: str,
                          selected_fields: List[str],
                          variables: Dict = None,
                          where: Dict = None,
                          pagination: Dict = None) -> Dict:
        """
        根据 LLM 选择的字段构建并执行查询
        这是 Agent 工具暴露的主方法：
          1. LLM 看到 field_options
          2. LLM 选择 relevant_fields
          3. 本方法构建合法 GraphQL 查询并执行
        """
        fields_str = "\n    ".join(selected_fields)

        var_defs = ""
        var_input = {}
        if where:
            var_defs = "($where: " + type_name + "FilterInput)"
            var_input["where"] = where

        if pagination:
            var_input.update({
                "first": pagination.get("first", 20),
                "after": pagination.get("after"),
            })

        query = f"""
        query {operation}{var_defs} {{
          {operation}({self._build_args_str(where, pagination)}) {{
            {fields_str}
          }}
        }}
        """
        return self._execute(query, var_input)

    def _build_args_str(self, where: Dict, pagination: Dict) -> str:
        """构建查询参数字符串"""
        parts = []
        if where:
            parts.append("where: $where")
        if pagination:
            if pagination.get("first"):

📡 gRPC 与 Protobuf

高性能RPC · Protocol Buffers · 强类型契约 · 流式支持 · 代码生成

gRPC (Google Remote Procedure Call) 是一个高性能、开源的 RPC 框架，使用 Protocol Buffers (Protobuf) 作为接口定义语言和二进制序列化格式。在 Agent 生态中，gRPC 主要适用于 服务间通信——Agent 编排层调用后端微服务、工具执行层调用 AI 推理服务（如自部署的 vLLM/TGI）、以及 Agent-to-Agent 之间的高效通信。gRPC 的核心优势在于：二进制序列化比 JSON 小 3-10 倍、严格类型系统消除解析歧义、内建双向流式通信支持。

对于 Agent 开发者来说，gRPC 的 强类型契约 是最大的价值所在。Protobuf 定义的 .proto 文件同时作为接口文档和代码生成源——运行 protoc 编译器即可生成 Python/Go/Java/TypeScript 等多种语言的客户端和服务端 stub 代码。这意味着 Agent 工具的函数签名和参数类型由 Proto 文件精准定义，编译器保证调用方和实现方的一致性。与 REST 的 JSON Schema（运行时校验、容易不一致）相比，gRPC 的契约强制在编译时检查，从根源上减少因接口不一致导致的 Agent 工具调用失败。

gRPC 提供了四种通信模式，每一种在 Agent 工具设计中有独特应用场景。Unary RPC（单次请求-响应）：最常见模式，Agent 调用一个工具并等待结果返回，适用于参数明确、结果确定的同步操作。Server Streaming RPC（服务端流式）：Agent 发送一次请求，服务端持续推送结果流——适用场景包括 LLM 推理服务的 token-by-token 输出、长时间运行任务的渐进式结果报告。Client Streaming RPC（客户端流式）：Agent 持续发送数据流，服务端汇总后一次性返回——适用于批量处理场景如批量 Embedding、批量文档分析。Bidirectional Streaming RPC（双向流式）：Agent 和服务端同时收发数据——适用于实时协作场景（如多 Agent 在线讨论和动态任务调整），实现复杂度最高。

gRPC 在 Agent 架构中应扮演的是内部基础设施通信层的角色，而非直接暴露给终端用户的 API。典型的架构分层：外部 API 层（REST/GraphQL）接收用户请求 → Agent 编排层将任务分解为内部调用 → 内部通信层（gRPC）高效连接 Agent 组件和微服务。gRPC 不直接面向外部的主要原因是：浏览器 HTTP/2 支持不足、调试工具不如 curl 方便、以及许多网络中间件对 gRPC 流量缺乏原生支持。但通过 grpc-gateway 可以自动生成与 gRPC 服务对应的 REST API，实现一套 Proto 定义同时对外提供 REST 和 gRPC 两种访问方式。

在决定是否使用 gRPC 时，关键考量因素包括：性能需求（高频/低延迟调用场景 gRPC 优势明显）、服务规模（微服务数量多、相互调用频繁时 gRPC 的类型安全优势放大）、团队技能（需要理解 Proto 语法和代码生成流程）、基础设施（负载均衡器、服务网格是否支持 gRPC）。对于大多数 Agent 原型和中小规模应用，REST + JSON 已经足够好用。当系统演进到需考虑毫秒级延迟优化和跨语言服务调用的阶段时，引入 gRPC 是自然的选择。

class SearchServiceServicer(search_pb2_grpc.SearchServiceServicer):
    """Agent 搜索工具服务实现"""

    def Search(self, request, context):
        """Unary: 单次查询"""
        results = [
            search_pb2.SearchResult(
                id=f"r_{i}",
                title=f"关于'{request.query}'的结果 {i+1}",
                snippet=f"这是搜索结果 {i+1} 的摘要...",
                url=f"https://example.com/{i}",
                relevance_score=0.95 - 0.1 * i,
                metadata={"source_type": "web", "crawled_at": "2026-05-10"}
            )
            for i in range(min(request.max_results, 10))
        ]
        return search_pb2.SearchResponse(
            results=results,
            total_count=len(results),
            took_ms=42
        )

    def SearchStream(self, request, context) -> Iterator[search_pb2.SearchResult]:
        """Server Streaming: 结果逐步推送"""
        for i in range(min(request.max_results, 10)):
            yield search_pb2.SearchResult(
                id=f"r_stream_{i}",
                title=f"流式结果 {i+1}: {request.query}",
                snippet=f"实时推送的搜索结果...",
                relevance_score=1.0 - 0.08 * i,
            )

    def InteractiveSearch(self, request_iterator, context):
        """Bidirectional Streaming: 对话式搜索"""
        context_state = {}
        for query in request_iterator:
            # 根据 LLM 的查询和 refinement_params 动态调整搜索
            refined_query = query.query_text
            if "query_rewrite" in query.refinement_params:
                refined_query = query.refinement_params["query_rewrite"]

            results = [
                search_pb2.SearchResult(
                    id=f"ir_{context_state.get('round', 0)}_{j}",
                    title=f"交互式结果: {refined_query}",
                    snippet=f"多轮交互搜索的第{context_state.get('round', 0)+1}轮",
                    relevance_score=0.88,
                )
                for j in range(3)
            ]
            context_state["round"] = context_state.get("round", 0) + 1

            for result in results:
                yield result

🤖 Anthropic Claude SDK 深度解析

Messages API · System Prompts · Tool Use · Prompt Caching · Extended Thinking · Streaming

Anthropic Claude SDK 是构建 AI Agent 的最直接路径——它不提供传统框架的抽象层，而是将 Claude API 的所有能力以简洁、直观的方式暴露给开发者。SDK 的核心是 Messages API：一个对话式接口，接收 system prompt + messages 数组 + tools 定义，返回 Claude 的文本回复或工具调用请求。与 OpenAI Chat Completions API 类似但设计哲学不同——Anthropic API 更强调安全对齐（Constitutional AI）、工具调用的清晰语义和推理过程的透明度。

System Prompt 最佳实践是 Anthropic SDK 的重要话题。Claude 的 system 参数是一个独立的顶级字段，不会被后续对话轮次截断或稀释。有效的 Agent system prompt 应包括：角色定义（"你是一个数据分析 Agent..."）、工具使用指导（"使用工具前先评估工具描述与用户问题的匹配度"）、行为约束（"如果多个工具都能完成任务，优先选择最简单高效的"）、输出格式规范（"统计数字保留两位小数，金额使用人民币表示"）。Anthropic 官方推荐将 system prompt 控制在 2000-4000 tokens 以平衡效果和成本。

Claude 的 Tool Use 机制是其 Agent 能力的核心。不同于 OpenAI 的 function calling（模型输出 JSON），Claude 的 tool use 在 API 层面返回一个结构化的 content block（type: "tool_use"），包含 tool name、tool_use_id 和 input JSON。Agent 循环中，开发者解析这个 block、执行工具、将结果包装在 tool_result block 中追加到 messages，然后继续调用 API。Claude 支持 并行工具调用——当系统判断多个工具调用之间无依赖关系时，会在单次响应中同时发起多个 tool_use，大幅减少 Agent 循环的轮次和延迟。

Prompt Caching 是 Claude SDK 最具成本效益的特性。通过在 content 上标记 cache_control: {type: "ephemeral"}，可以将 system prompt、工具定义、甚至多轮对话中的历史消息缓存起来。缓存的 Token 读取价格仅为正常读取价格的 10%，写入价格稍高（约 25% 溢价）。对于 system prompt 和 tools 在每次请求中不变的生产 Agent，Prompt Caching 可以节省高达 90% 的输入 Token 成本。最佳实践：在 system prompt 的末尾和工具定义的最后一条末尾加上 cache_control 断点，将这两大部分固定缓存；如果对话历史较长，在前 N-2 轮消息的末尾也加断点以缓存大部分上下文。

Claude 的 Extended Thinking 功能是其独特的差异化优势。通过设置 thinking: {type: "enabled", budget_tokens: 8000}，Claude 会在给出最终回复前进行多步推理，并将推理过程以结构化格式返回。与 Chain-of-Thought 提示不同，Extended Thinking 是模型内部的训练能力——可靠性远超 Prompt 诱导的推理。对 Agent 来说，这意味着在关键决策步骤（工具选择、复杂参数推理、多步规划）中可以请求更高 thinking budget 以获得更准确的结果。Budget 范围通常为 1024-32000 tokens，根据任务复杂度选择。Note：thinking tokens 不计入常规 output tokens，有自己的使用额度。

import json
import time
import anthropic
from typing import Any, Callable, Dict, List, Optional


class ClaudeAgent:
    """基于 Anthropic SDK 的完整 Agent 实现"""

    def __init__(self, api_key: str,
                 model: str = "claude-sonnet-4-20250514",
                 max_tool_rounds: int = 10):
        self.client = anthropic.Anthropic(api_key=api_key)
        self.model = model
        self.max_tool_rounds = max_tool_rounds
        self.tools: Dict[str, Dict] = {}    # tool_name -> {schema, handler}
        self.tool_schemas: List[Dict] = []
        self.messages: List[Dict] = []

    def register_tool(self, name: str, description: str,
                      input_schema: Dict, handler: Callable) -> None:
        """注册一个工具：提供 Schema 定义和实际处理函数"""
        schema = {
            "name": name,
            "description": description,
            "input_schema": input_schema,
        }
        self.tools[name] = {"schema": schema, "handler": handler}
        self.tool_schemas.append(schema)

    def _build_system_prompt(self, custom_prompt: str) -> List[Dict]:
        """构建包含缓存断点的 System Prompt"""
        base_prompt = (
            "你是一个能够使用工具完成任务的 AI Agent。"
            "核心原则：\n"
            "1. 仔细分析用户请求，确定需要哪些信息\n"
            "2. 在调用工具前，评估所有可用工具，选择最匹配的工具\n"
            "3. 如果多个工具能完成任务，选择最简单高效的那个\n"
            "4. 工具调用结果可能不全——必要时用不同参数重新调用\n"
            "5. 完成所有工具调用后，基于工具结果给出完整的自然语言回答\n"
            "6. 不要编造任何未在工具结果中出现的信息\n"
            "7. 如果工具返回错误，分析错误原因并尝试不同策略\n"
            "\n"
            f"{custom_prompt}"
        )
        return [
            {
                "type": "text",
                "text": base_prompt,
            },
            {
                "type": "text",
                "text": "[SYSTEM_PROMPT_END]",
                "cache_control": {"type": "ephemeral"},
            },
        ]

    def _build_tool_definitions(self) -> List[Dict]:
        """构建工具定义列表（末尾带缓存断点）"""
        if len(self.tool_schemas) <= 1:
            return self.tool_schemas

        cached = []
        for i, tool in enumerate(self.tool_schemas):
            entry = tool.copy()
            if i == len(self.tool_schemas) - 1:
                # 最后一个工具定义末尾加缓存断点
                entry["cache_control"] = {"type": "ephemeral"}
            cached.append(entry)
        return cached

    def _execute_tool(self, name: str,
                      input_data: Dict) -> Dict:
        """执行工具并返回结果"""
        if name not in self.tools:
            return {"error": f"未知工具: {name}"}

        try:
            result = self.tools[name]["handler"](**input_data)
            return {"result": result}
        except Exception as e:
            return {"error": f"工具执行失败: {str(e)}"}

    def run(self, user_message: str,
            system_prompt: str = "",
            thinking_budget: int = 0,
            temperature: float = 0.7) -> str:
        """
        执行 Agent 主循环

        Args:
            user_message: 用户输入
            system_prompt: 自定义系统提示词（追加到基础提示词后）
            thinking_budget: Extended Thinking token 预算（0=禁用）
            temperature: 采样温度
        """
        system = self._build_system_prompt(system_prompt)
        tools = self._build_tool_definitions()

        self.messages.append({
            "role": "user",
            "content": user_message,
        })

        for round_num in range(1, self.max_tool_rounds + 1):
            kwargs = {
                "model": self.model,
                "max_tokens": 4096,
                "system": system,
                "messages": self.messages,
                "tools": tools,
                "temperature": temperature,
            }

            if thinking_budget > 0:
                kwargs["thinking"] = {
                    "type": "enabled",
                    "budget_tokens": thinking_budget,
                }

            response = self.client.messages.create(**kwargs)

            # 分析响应
            text_parts = []
            tool_uses = []

            for block in response.content:
                if block.type == "text":
                    text_parts.append(block.text)
                elif block.type == "tool_use":
                    tool_uses.append({
                        "id": block.id,
                        "name": block.name,
                        "input": block.input,
                    })

            assistant_content = []
            if text_parts:
                assistant_content.append({
                    "type": "text",
                    "text": "\n".join(text_parts),
                })
            for tu in tool_uses:
                assistant_content.append({
                    "type": "tool_use",
                    "id": tu["id"],
                    "name": tu["name"],
                    "input": tu["input"],
                })

            self.messages.append({
                "role": "assistant",
                "content": assistant_content,
            })

            # 如果没有工具调用，Agent 完成了
            if not tool_uses:
                return "\n".join(text_parts)

            # 执行所有工具调用
            tool_results = []
            for tu in tool_uses:
                exec_result = self._execute_tool(tu["name"], tu["input"])
                result_content = json.dumps(exec_result, ensure_ascii=False)
                tool_results.append({
                    "type": "tool_result",
                    "tool_use_id": tu["id"],
                    "content": result_content,
                })

            self.messages.append({
                "role": "user",
                "content": tool_results,
            })

        return "Agent 已达到最大工具调用轮次限制。"

    def clear_history(self) -> None:
        """清空对话历史"""
        self.messages = []

    def token_count(self) -> int:
        """使用 API 计数工具估算 Token 用量"""
        resp = self.client.messages.count_tokens(
            model=self.model,
            messages=self.messages,
        )
        return resp.input_tokens

🤖 OpenAI SDK 深度解析

Chat Completions · Function Calling · Structured Outputs · Assistants API · Vector Stores

OpenAI SDK 提供了从底层 API 调用到全托管 Agent 服务的完整能力谱系。最基础的 Chat Completions API 接收 messages 数组和 tools 定义，返回文本回复或 function call。与 Anthropic 的设计不同，OpenAI 的函数调用返回的是标准的 function_call 对象（包含 name 和 arguments 字符串），arguments 是 JSON 字符串而非已解析的对象。这使得开发者需要额外解析步骤，但也提供了更灵活的格式控制空间。OpenAI 的并行函数调用（Parallel Function Calling）是默认启用的能力——模型可以在单次响应中调用多个互不依赖的函数，减少 Agent 循环的往返次数。

Structured Outputs 是 OpenAI 的重要创新功能，通过 JSON Mode 和更精确的 response_format 参数实现。Agent 可以使用 response_format={"type": "json_schema", "json_schema": {...}} 来强制模型输出符合指定 JSON Schema 的结构化数据。这在需要 Agent 与数据库、ETL 管道或下游微服务精确对接时至关重要——不再需要"尝试解析 LLM 输出的 JSON，处理可能存在的格式错误"这种不稳定的模式。Structured Outputs 在 gpt-4o 及更新模型上可用，且在 function calling 的结果中默认精确匹配 Schema，消除了传统"LLM 可能输出错误格式的工具参数"的痛点。

Assistants API 是 OpenAI 的全托管 Agent 服务，抽象掉了 Agent 循环、上下文管理、工具执行等底层细节。核心概念包括：Assistant（配置——模型、指令、工具、文件）、Thread（对话容器——自动管理消息历史和截断）、Run（执行实例——代表 Assistant 在 Thread 上的一次执行周期）。Assistants API 内置三个强大工具：Code Interpreter（沙箱 Python 环境，可执行代码、生成图表、处理文件）、File Search（内置向量存储的 RAG 检索）、Function Calling（自定义函数）。Run 的生命周期管理是关键：queued → in_progress → requires_action（需要执行函数）→ completed/failed/expired。

Vector Stores 是 Assistants API 的知识库组件。工作流程：上传文件 → 创建 Vector Store（自动分块、Embedding、索引）→ 将 Vector Store 附加到 Assistant → Assistant 在 File Search 工具启用时自动从 Vector Store 检索相关片段。单个 Assistant 可附加一个 Vector Store，单个 Vector Store 最多支持 10,000 个文件。Vector Store 支持文件级别的过期策略（expires_after 参数），可设定文件在 Store 中的保留时间（如 7 天后自动从向量索引中移除）。这对经常更新知识库的 Agent 场景尤为重要。

选择 OpenAI SDK 的哪个层次取决于场景。Chat Completions + 手写 Agent 循环：最大控制权，适合需要精确控制每一轮的场景。Assistants API：最快开发速度，适合 MVP 验证和简单 Agent 场景，但注意成本不透明（Run 的执行步数由 OpenAI 决定）和数据必须经 OpenAI 服务器处理。Batch API：适合不需要实时响应的批量处理任务（如夜间数据处理），成本为实时 API 的 50%，24 小时内完成。在 2026 年的生态中，许多团队采用分层策略：内部用 Assistants API 做快速实验，生产环境用 Chat Completions + 自建循环做精确控制。

import json
import time
from openai import OpenAI
from typing import Callable, Dict, List, Any


class OpenAIAgent:
    """基于 OpenAI Chat Completions 的完整 Agent"""

    def __init__(self, api_key: str,
                 model: str = "gpt-4o",
                 max_tool_rounds: int = 10):
        self.client = OpenAI(api_key=api_key)
        self.model = model
        self.max_tool_rounds = max_tool_rounds
        self.tools: Dict[str, Dict] = {}
        self.tool_schemas: List[Dict] = []
        self.messages: List[Dict] = []

    def register_tool(self, name: str, description: str,
                      parameters: Dict, handler: Callable) -> None:
        """注册工具"""
        self.tools[name] = {
            "handler": handler,
            "schema": {
                "type": "function",
                "function": {
                    "name": name,
                    "description": description,
                    "parameters": parameters,
                }
            }
        }
        self.tool_schemas.append(self.tools[name]["schema"])

    def run(self, user_message: str,
            system_prompt: str = "",
            temperature: float = 0.7) -> str:
        """Agent 主循环"""
        self.messages = []

        if system_prompt:
            self.messages.append({
                "role": "system", "content": system_prompt
            })
        self.messages.append({
            "role": "user", "content": user_message
        })

        for _ in range(self.max_tool_rounds):
            kwargs = {
                "model": self.model,
                "messages": self.messages,
                "temperature": temperature,
            }
            if self.tool_schemas:
                kwargs["tools"] = self.tool_schemas

            response = self.client.chat.completions.create(**kwargs)
            choice = response.choices[0]
            message = choice.message

            # 判断是否调用了工具
            if message.tool_calls:
                # 追加 assistant 消息（包含 tool_calls）
                self.messages.append({
                    "role": "assistant",
                    "content": message.content,
                    "tool_calls": [
                        {
                            "id": tc.id,
                            "type": "function",
                            "function": {
                                "name": tc.function.name,
                                "arguments": tc.function.arguments,
                            }
                        }
                        for tc in message.tool_calls
                    ]
                })

                # 执行每个工具调用
                for tc in message.tool_calls:
                    name = tc.function.name
                    args = json.loads(tc.function.arguments)

                    if name in self.tools:
                        try:
                            result = self.tools[name]["handler"](**args)
                        except Exception as e:
                            result = {"error": str(e)}
                    else:
                        result = {"error": f"未知工具: {name}"}

                    self.messages.append({
                        "role": "tool",
                        "tool_call_id": tc.id,
                        "content": json.dumps(result, ensure_ascii=False),
                    })
            else:
                # 没有工具调用，返回文本回复
                return message.content or ""

        return "达到最大工具调用轮次。"

class OpenAIAssistantAgent:
    """基于 Assistants API 的全托管 Agent"""

    def __init__(self, api_key: str):
        self.client = OpenAI(api_key=api_key)

    def create_assistant(self, name: str, instructions: str,
                         model: str = "gpt-4o",
                         tools: List[Dict] = None) -> str:
        """创建 Assistant 并返回 ID"""
        assistant = self.client.beta.assistants.create(
            name=name,
            instructions=instructions,
            model=model,
            tools=tools or [{"type": "code_interpreter"}],
        )
        return assistant.id

    def create_thread(self) -> str:
        """创建 Thread 并返回 ID"""
        thread = self.client.beta.threads.create()
        return thread.id

    def add_message(self, thread_id: str, content: str) -> None:
        """向 Thread 添加用户消息"""
        self.client.beta.threads.messages.create(
            thread_id=thread_id,
            role="user",
            content=content,
        )

    def run(self, assistant_id: str, thread_id: str) -> str:
        """执行 Run 并等待完成"""
        run = self.client.beta.threads.runs.create(
            thread_id=thread_id,
            assistant_id=assistant_id,
        )

        # 轮询等待 Run 完成
        while True:
            run = self.client.beta.threads.runs.retrieve(
                thread_id=thread_id, run_id=run.id
            )

            if run.status == "completed":
                # 获取最新消息
                messages = self.client.beta.threads.messages.list(
                    thread_id=thread_id, limit=1
                )
                text_parts = []
                for msg in messages.data:
                    if msg.role == "assistant":
                        for block in msg.content:
                            if block.type == "text":
                                text_parts.append(block.text.value)
                return "\n".join(text_parts)

            elif run.status in ("failed", "cancelled", "expired"):
                return f"Run {run.status}: {run.last_error}"

            elif run.status == "requires_action":
                # 处理 function calling
                tool_outputs = []
                for tc in run.required_action.submit_tool_outputs.tool_calls:
                    func_name = tc.function.name
                    func_args = json.loads(tc.function.arguments)
                    # 这里执行实际函数...
                    output = json.dumps(
                        {"result": f"执行 {func_name}({func_args})"}
                    )
                    tool_outputs.append({
                        "tool_call_id": tc.id,
                        "output": output,
                    })

                self.client.beta.threads.runs.submit_tool_outputs(
                    thread_id=thread_id,
                    run_id=run.id,
                    tool_outputs=tool_outputs,
                )

            time.sleep(0.5)

    def create_vector_store(self, file_paths: List[str],
                            name: str) -> str:
        """创建带文件的 Vector Store"""
        file_ids = []
        for path in file_paths:
            with open(path, "rb") as f:
                file_obj = self.client.files.create(
                    file=f,
                    purpose="assistants"
                )
                file_ids.append(file_obj.id)

        vector_store = self.client.beta.vector_stores.create(
            name=name,
            file_ids=file_ids,
            expires_after={"anchor": "last_active_at", "days": 30}
        )
        return vector_store.id

🔗 MCP 协议深入

Model Context Protocol · Client-Server架构 · stdio/SSE传输 · Tools/Resources/Prompts · 工具复用

MCP (Model Context Protocol) 是 Anthropic 于 2024 年末提出并开源的开放协议，为 AI 模型与外部工具、数据源和服务之间的交互提供了统一的标准化接口。MCP 的诞生解决了 Agent 生态中最核心的痛点——工具集成的碎片化。在 MCP 之前，每个 Agent 框架都有自己的一套工具定义格式和调用协议，工具开发者需要为每个框架编写适配器，Agent 开发者被锁定在特定框架的工具生态中。MCP 通过定义统一的客户端-服务端协议，让工具开发者和框架开发者各自独立迭代，通过标准化接口实现即插即用。

MCP 的架构基于经典的 Client-Server 模型。MCP Server 是一个轻量级服务进程，暴露一组 Tools（可调用的功能）、Resources（可读取的结构化数据）和 Prompts（预定义的提示模板）。Server 通过标准输入输出（stdio）或 HTTP Server-Sent Events（SSE）与 Client 通信，通信协议基于 JSON-RPC 2.0——请求和响应都是结构化的 JSON 消息。MCP Client 嵌入在 Agent 框架中，负责发现 Server 提供的能力（通过 tools/list、resources/list 等方法），并将这些能力转换为 Agent 框架自身的工具 Schema 格式（如 Anthropic Tool Use Schema、OpenAI Function Schema）。

MCP 的三个核心概念各有用途：Tools——模型可控的、可能产生副作用的操作（如读写文件、调用 API、执行命令）。Tools 返回结构化的结果并告知模型操作是否成功。这是 MCP 中使用最频繁的组件。Resources——只读的、模型可上下文化的数据源（如文件内容、数据库记录、API 响应）。Resources 是静态或准静态的数据快照，不会因模型读取而发生改变。Prompts——预定义的、可参数化的提示词模板，Server 暴露给 Client 的对话起点或交互模式（如"代码审查提示模板"、"系统架构分析模板"）。一个 MCP Server 可以同时提供所有三种组件，也可以只提供其中一种。

MCP 的 传输层有两种模式。stdio 模式：Client 作为父进程启动 Server 子进程，通过标准输入输出流传输 JSON-RPC 消息。适合本地工具——Server 与 Agent 运行在同一台机器上，无需网络配置，安全性高。SSE 模式：Server 作为 HTTP 服务运行，Client 通过 HTTP GET 建立 SSE 连接接收 Server 推送的消息，通过 HTTP POST 向 Server 发送请求。适合远程工具——Server 可作为微服务部署在网络中的任何位置，多个 Agent 可并发共享同一个 Server。两种模式各有适用场景：stdio 简单安全但限制单机，SSE 灵活强大但需要处理网络安全和认证。

MCP 生态在 2026 年已经显著成熟。官方和社区维护的 MCP Server 涵盖了文件系统操作、数据库查询（PostgreSQL、SQLite、MySQL）、云服务管理（AWS、GCP）、浏览器自动化（Puppeteer）、代码仓库操作（GitHub、GitLab）、搜索服务（Brave Search、Tavily）等数十个领域。构建 MCP Server 的开发体验简单——只需实现几个接口函数并遵循 JSON-RPC 协议，即可让任何支持 MCP 的 Agent 框架使用你的工具。这种"一次开发，到处使用"的模式正在重塑 AI 工具生态的经济学——工具开发者的潜在用户从单个框架扩展到了整个 Agent 生态。

# ============ MCP SERVER: 文件系统工具 ============

class FilesystemMCPServer:
    """暴露文件系统操作为 MCP Tools 的 Server"""

    def __init__(self, allowed_paths: List[str]):
        self.allowed_paths = allowed_paths
        self.server = Server("filesystem-tools")

    def _is_path_allowed(self, path: str) -> bool:
        """检查路径是否在允许列表中"""
        import os
        abs_path = os.path.abspath(path)
        return any(
            abs_path.startswith(os.path.abspath(p))
            for p in self.allowed_paths
        )

    def setup_handlers(self):
        """注册所有 Tool/Resource/Prompt handlers"""
        server = self.server

        # ---- Tools ----
        @server.list_tools()
        async def list_tools() -> List[Tool]:
            return [
                Tool(
                    name="read_file",
                    description=(
                        "读取指定文件的内容。返回文件全文。"
                        "支持 .txt, .md, .py, .json, .yaml 等文本格式。"
                    ),
                    inputSchema={
                        "type": "object",
                        "properties": {
                            "path": {
                                "type": "string",
                                "description": "文件的绝对路径或相对路径"
                            },
                            "max_lines": {
                                "type": "integer",
                                "description": "最大读取行数，默认全部",
                                "default": 500
                            }
                        },
                        "required": ["path"]
                    }
                ),
                Tool(
                    name="list_directory",
                    description="列出目录下的文件和子目录",
                    inputSchema={
                        "type": "object",
                        "properties": {
                            "path": {
                                "type": "string",
                                "description": "目录路径"
                            },
                            "pattern": {
                                "type": "string",
                                "description": "文件名过滤 glob 模式，如 *.py"
                            }
                        },
                        "required": ["path"]
                    }
                ),
                Tool(
                    name="search_files",
                    description="在指定目录中递归搜索包含关键字的文件",
                    inputSchema={
                        "type": "object",
                        "properties": {
                            "directory": {
                                "type": "string",
                                "description": "搜索的根目录"
                            },
                            "query": {
                                "type": "string",
                                "description": "搜索关键词或正则表达式"
                            },
                            "file_pattern": {
                                "type": "string",
                                "description": "限定文件类型，如 *.py"
                            },
                            "max_results": {
                                "type": "integer",
                                "description": "最大返回结果数",
                                "default": 20
                            }
                        },
                        "required": ["directory", "query"]
                    }
                ),
            ]

        @server.call_tool()
        async def call_tool(name: str,
                            arguments: dict) -> List[TextContent]:
            if name == "read_file":
                path = arguments["path"]
                if not self._is_path_allowed(path):
                    return [TextContent(
                        type="text",
                        text=f"错误: 无权访问路径 '{path}'。"
                             f"允许的路径: {self.allowed_paths}"
                    )]
                max_lines = arguments.get("max_lines", 500)
                import pathlib
                content = pathlib.Path(path).read_text(encoding="utf-8")
                lines = content.split("\n")
                if len(lines) > max_lines:
                    lines = lines[:max_lines]

class MCPAgentClient:
    """Agent 中集成 MCP 工具的客户端"""

    def __init__(self):
        self.server_processes: Dict[str, subprocess.Popen] = {}
        self.tools: List[Dict] = []

    def connect_server(self, name: str,
                       server_script: str) -> None:
        """启动一个 MCP Server 进程并发现其工具"""
        proc = subprocess.Popen(
            ["python", server_script],
            stdin=subprocess.PIPE,
            stdout=subprocess.PIPE,
            stderr=subprocess.PIPE,
            text=True
        )
        self.server_processes[name] = proc

        # 发送 tools/list JSON-RPC 请求
        request = json.dumps({
            "jsonrpc": "2.0",
            "id": 1,
            "method": "tools/list",
            "params": {}
        })
        proc.stdin.write(request + "\n")
        proc.stdin.flush()

        # 读取响应
        response_line = proc.stdout.readline()
        response = json.loads(response_line)
        server_tools = response.get("result", {}).get("tools", [])

        # 转换为 OpenAI/Anthropic 兼容的 Tool Schema
        for tool in server_tools:
            self.tools.append({
                "server": name,
                "name": tool["name"],
                "description": tool.get("description", ""),
                "parameters": tool.get("inputSchema", {}),
            })

        print(f"[MCP] 已连接 '{name}' Server, "
              f"发现 {len(server_tools)} 个工具")

    def call_tool(self, name: str,
                  arguments: Dict) -> Any:
        """调用 MCP 工具"""
        # 找到工具所属的 Server
        tool_info = next(
            (t for t in self.tools if t["name"] == name), None
        )
        if not tool_info:
            return {"error": f"未找到工具: {name}"}

        proc = self.server_processes[tool_info["server"]]

        # 发送 tools/call JSON-RPC 请求
        request = json.dumps({
            "jsonrpc": "2.0",
            "id": 2,
            "method": "tools/call",
            "params": {
                "name": name,
                "arguments": arguments,
            }
        })
        proc.stdin.write(request + "\n")
        proc.stdin.flush()

        response_line = proc.stdout.readline()
        response = json.loads(response_line)
        return response.get("result", response.get("error", {}))

    def get_tool_schemas(self) -> List[Dict]:
        """获取所有可用工具的 Schema（供 Agent 框架使用）"""
        return [
            {
                "name": t["name"],
                "description": t["description"],
                "input_schema": {
                    "type": "object",
                    "properties": t["parameters"].get(
                        "properties", {}
                    ),
                    "required": t["parameters"].get(
                        "required", []
                    ),
                }
            }
            for t in self.tools
        ]

    def shutdown(self):
        """关闭所有 MCP Server 进程"""
        for name, proc in self.server_processes.items():
            proc.terminate()
            proc.wait()
            print(f"[MCP] 已关闭 '{name}' Server")

🔔 Webhook 与事件驱动

Webhook接收触发 · HMAC签名验证 · 幂等性键 · 异步回调处理 · 事件总线路由

Webhook 是 Agent 从"主动轮询"模式切换到"事件驱动"模式的关键技术。在传统的 API 集成中，Agent 需要定期查询外部系统获取新数据——这种轮询模式既低效（大量无效请求），又有延迟（从事件发生到 Agent 感知存在时间差）。Webhook 反转了这个模式：外部系统在事件发生时主动向 Agent 发送 HTTP POST 请求，Agent 即时响应事件。常见场景包括：GitHub Webhook 触发代码审查 Agent、Stripe Webhook 触发订单处理 Agent、Slack Webhook 触发消息处理 Agent、CI/CD 系统 Webhook 触发部署验证 Agent。

Webhook 安全性的核心是 签名验证 (HMAC)。外部服务在发送 Webhook 时，会使用一个预先共享的 Secret Key 对请求体计算 HMAC-SHA256 哈希值，将此值放在 X-Signature 或类似的 Header 中。接收端使用相同的 Secret Key 重新计算哈希并比对——如果一致则确认请求确实来自该服务且内容未在传输中被篡改。这个流程必须在任何业务逻辑执行前完成，否则整个 Webhook 通道的安全性无从谈起。生产级实现还应加入时间戳验证（拒绝过旧的请求以防止重放攻击）和 IP 白名单校验（仅接受来自已知服务的请求）。

幂等性是 Webhook 处理的最关键非功能性需求。外部服务在未收到 HTTP 200 确认时，通常会重试——同一个事件可能被发送多次。如果 Agent 对同一事件做了两次相同的处理（如重复创建订单、重复发送通知），后果可能是灾难性的。解决方案是使用 Idempotency Key（幂等键）——Webhook 请求中通常携带唯一的事件 ID（如 X-Event-Id），Agent 在处理前先检查此 ID 是否已被处理过（存储在 Redis 或数据库中），如果已处理则直接返回 200 而不执行任何业务逻辑。幂等检查应是 Webhook 处理器中第一个执行的步骤。

异步处理是 Webhook 的架构必备。Webhook 发送方通常期望在几秒内收到 HTTP 响应——如果 Agent 在收到 Webhook 后立即同步执行耗时业务逻辑（如调用 LLM 进行多步推理），Webhook 响应可能会超时，导致发送方认为失败并重复重试。正确的架构是：接收 Webhook → 快速验证签名和幂等性 → 将事件放入消息队列（或数据库任务表）→ 立即返回 200 → 后台 Worker 异步处理事件。Agent 处理完成后可以通过另一次 API 调用或 Webhook 回调通知原系统结果。这种 "Receive Fast, Process Later" 模式是生产级 Agent 的标准架构。

对于需要处理大量 Webhook 的 Agent 系统，事件路由是重要的架构考量。不同的 Webhook 事件应路由到不同的 Agent 或 Agent 组——GitHub Push 事件触发代码审查 Agent、GitHub Issue 事件触发 Issue 分类 Agent、GitHub PR 事件触发自动化测试 Agent。事件路由可以基于事件类型、来源、优先级等维度进行。在实现上，可以使用消息队列的 Topic 或 Routing Key 机制（如 RabbitMQ 的 Exchange Routing、AWS SNS Topic、Google Pub/Sub Topic）将事件分发给专门的消费者 Agent。

def verify_webhook_signature(secret_key: str):
    """装饰器: 验证 Webhook HMAC 签名"""
    def decorator(f):
        @wraps(f)
        def decorated(*args, **kwargs):
            signature = request.headers.get("X-Signature-256")
            if not signature:
                return jsonify({
                    "error": "缺少签名",
                    "code": "MISSING_SIGNATURE"
                }), 401

            # 计算请求体的 HMAC-SHA256
            raw_body = request.get_data()
            expected = "sha256=" + hmac.new(
                secret_key.encode("utf-8"),
                raw_body,
                hashlib.sha256
            ).hexdigest()

            if not hmac.compare_digest(signature, expected):
                return jsonify({
                    "error": "签名验证失败",
                    "code": "INVALID_SIGNATURE"
                }), 401

            # 时间戳验证（防止重放攻击，5分钟窗口）
            ts = request.headers.get("X-Request-Timestamp")
            if ts:
                try:
                    request_time = float(ts)
                    if abs(time.time() - request_time) > 300:
                        return jsonify({
                            "error": "请求已过期",
                            "code": "REQUEST_EXPIRED"
                        }), 401
                except ValueError:
                    pass

            return f(*args, **kwargs)
        return decorated
    return decorator

class IdempotencyManager:
    """Webhook 幂等性管理器"""

    KEY_PREFIX = "webhook:idempotency:"
    TTL_SECONDS = 86400 * 7  # 7天过期

    @classmethod
    def is_duplicate(cls, event_id: str) -> bool:
        """检查事件是否已处理"""
        key = cls.KEY_PREFIX + event_id
        return redis_client.exists(key) > 0

    @classmethod
    def mark_processed(cls, event_id: str) -> None:
        """标记事件为已处理"""
        key = cls.KEY_PREFIX + event_id
        redis_client.setex(key, cls.TTL_SECONDS, "1")

    @classmethod
    def clear(cls, event_id: str) -> None:
        """清除幂等检查标记（仅在失败重试时使用）"""
        key = cls.KEY_PREFIX + event_id
        redis_client.delete(key)

🔒 认证机制深入

API Keys · OAuth 2.0 · JWT · HMAC Signing · mTLS · Token刷新管理

Agent 在集成外部服务时，认证是第一个需要解决的技术问题——没有正确的认证，再精妙的 Agent 逻辑也无法运行。Agent 认证方案的选择取决于多个因素：被调用方的安全级别要求、Agent 的运行环境（服务器 vs 客户端）、Token 的生命周期管理、以及多租户支持。与传统的"一个用户一个 Token"模式不同，Agent 可能需要代表多个用户调用 API（如用户授权 Agent 访问其 Google 日历），这引入了 OAuth 委托授权的复杂性。

API Key 是最简单的认证方式——一个固定字符串在每次请求的 Header 中发送（通常为 Authorization: Bearer sk-xxx 或 X-API-Key: xxx）。适合 Agent 调用第三方 SaaS API（如天气服务、汇率查询、LLM API）。API Key 的管理需要关注：存储安全（环境变量或 Secret Manager，绝不硬编码在代码中）、轮换策略（定期更换以减少泄露风险）、权限最小化（每个 Key 只授予 Agent 完成任务所需的最少权限范围）。对于调用多个服务的 Agent，建议使用统一的 Secret Manager（如 HashiCorp Vault、AWS Secrets Manager）集中管理所有 Key。

OAuth 2.0 是 Agent 代表用户访问第三方服务时的标准方案。OAuth 2.0 定义了多种授权流程：Authorization Code Flow——最常用，Agent 将用户重定向到服务提供商登录页，用户授权后获得短期 Authorization Code，Agent 用此 Code 换取 Access Token 和 Refresh Token。适合 Web Agent 和需要用户授权的场景。Client Credentials Flow——Agent 使用自己的 Client ID 和 Client Secret 直接获取 Access Token，不涉及用户。适合服务间调用，Agent 以自己的身份而非某个用户的身份访问 API。Device Code Flow——适合没有浏览器的设备（如 CLI Agent），用户在另一台设备上输入验证码完成授权。对于需要用户授权的 Agent 来说，OAuth 最大的复杂性在于 Refresh Token 管理——Access Token 通常在 1 小时后过期，Agent 需要使用 Refresh Token 自动续期，这要求 Agent 具备"Token 即将过期时主动刷新"的能力。

JWT (JSON Web Token) 是自包含的 Token 格式，广泛用于 API 认证和 Agent 间通信。一个 JWT 包含三个 Base64URL 编码的部分：Header（算法和类型）、Payload（声明，如用户 ID、过期时间、权限）、Signature（签名——证明 Token 未被篡改）。在 Agent 间通信场景中，JWT 非常适合实现 Service-to-Service 认证：Agent A 生成一个 JWT 声明自己的身份和权限，用私钥签名；Agent B 用公钥验证签名后信任 JWT 的声明。JWT 的优势在于无状态——验证方不需要查询数据库或调用 Auth Service，仅通过密码学操作即可验证 Token 真实性。

对于高安全性 Agent 场景，mTLS (Mutual TLS) 提供传输层的双向认证。传统的 TLS 仅客户端验证服务端（浏览器验证网站证书）；mTLS 要求双方都出示证书——服务端验证客户端证书以确认 Agent 的身份，Agent 验证服务端证书以确认目标服务的身份。mTLS 在企业 Agent 架构中尤为重要：Agent 调用内部微服务时，mTLS 确保只有经过授权的 Agent 才能访问敏感服务，且通信内容全程加密。mTLS 通常由服务网格（如 Istio、Linkerd）或 API 网关管理，Agent 开发者只需配置证书路径而无需在代码中处理证书逻辑。

import time
import json
import threading
import requests
from typing import Dict, Optional, Callable
from dataclasses import dataclass, field
from datetime import datetime, timedelta


@dataclass
class OAuthToken:
    """OAuth Token 数据模型"""
    access_token: str
    refresh_token: Optional[str]
    token_type: str = "Bearer"
    expires_at: float = 0.0   # Unix timestamp
    scope: str = ""
    raw: Dict = field(default_factory=dict)

    @property
    def is_expired(self, buffer_seconds: int = 60) -> bool:
        """检查 Token 是否过期（预留缓冲时间）"""
        return time.time() >= (self.expires_at - buffer_seconds)

    @property
    def auth_header(self) -> str:
        """构建 Authorization Header"""
        return f"{self.token_type} {self.access_token}"


class OAuth2TokenManager:
    """
    OAuth 2.0 Token 管理器
    支持 Client Credentials 和 Authorization Code 两种流程
    自动在 Token 过期前刷新
    """

    def __init__(self,
                 token_url: str,
                 client_id: str,
                 client_secret: str,
                 scope: str = "",
                 refresh_before_expiry: int = 120):
        self.token_url = token_url
        self.client_id = client_id
        self.client_secret = client_secret
        self.scope = scope
        self.refresh_before_expiry = refresh_before_expiry
        self._token: Optional[OAuthToken] = None
        self._lock = threading.Lock()
        self._on_refresh_callbacks: list[Callable] = []

    def on_refresh(self, callback: Callable):
        """注册 Token 刷新回调"""
        self._on_refresh_callbacks.append(callback)

    def get_token(self,
                  grant_type: str = "client_credentials",
                  authorization_code: str = None,
                  redirect_uri: str = None) -> OAuthToken:
        """
        获取 Token（自动处理刷新）
        如果 Token 存在且未过期则直接返回；
        否则自动刷新或获取新 Token。
        """
        with self._lock:
            if (self._token is not None and
                    not self._token.is_expired(
                        self.refresh_before_expiry)):
                return self._token

            # Token 不存在或即将过期
            if (self._token is not None and
                    self._token.refresh_token):
                # 使用 Refresh Token 续期
                new_token = self._refresh_token(
                    self._token.refresh_token
                )
            else:
                # 获取新 Token
                new_token = self._fetch_token(
                    grant_type=grant_type,
                    authorization_code=authorization_code,
                    redirect_uri=redirect_uri,
                )

            self._token = new_token

            # 触发回调
            for cb in self._on_refresh_callbacks:
                try:
                    cb(new_token)
                except Exception as e:
                    print(f"Token 刷新回调失败: {e}")

            return self._token

    def _fetch_token(self, grant_type: str,
                     authorization_code: str = None,
                     redirect_uri: str = None) -> OAuthToken:
        """请求 OAuth Token 端点获取新 Token"""
        payload = {
            "client_id": self.client_id,
            "client_secret": self.client_secret,
            "grant_type": grant_type,
        }

        if self.scope:
            payload["scope"] = self.scope
        if grant_type == "authorization_code":
            payload["code"] = authorization_code
            payload["redirect_uri"] = redirect_uri

        resp = requests.post(
            self.token_url,
            data=payload,
            headers={
                "Content-Type":
                    "application/x-www-form-urlencoded",
            },
            timeout=15,
        )
        resp.raise_for_status()
        data = resp.json()

        expires_in = data.get("expires_in", 3600)
        return OAuthToken(
            access_token=data["access_token"],
            refresh_token=data.get("refresh_token"),
            token_type=data.get("token_type", "Bearer"),
            expires_at=time.time() + expires_in,
            scope=data.get("scope", ""),
            raw=data,
        )

    def _refresh_token(self,
                       refresh_token: str) -> OAuthToken:
        """使用 Refresh Token 获取新 Token"""
        return self._fetch_token(
            grant_type="refresh_token",
            authorization_code=refresh_token,
        )

    def force_refresh(self) -> OAuthToken:
        """强制刷新 Token（如发现 Token 泄露时）"""
        with self._lock:
            self._token = None
            return self.get_token()

    def get_authenticated_session(self) -> requests.Session:
        """获取已认证的 requests Session"""
        session = requests.Session()
        token = self.get_token()
        session.headers.update({
            "Authorization": token.auth_header,
        })

        # 安装请求钩子：如遇 401 自动重试
        def handle_401(resp, *args, **kwargs):
            if resp.status_code == 401:
                # 强制刷新 Token 并重试
                new_token = self.force_refresh()
                resp.request.headers[
                    "Authorization"
                ] = new_token.auth_header
                return session.send(resp.request)
            return resp

        session.hooks["response"].append(handle_401)
        return session

import jwt  # pip install pyjwt

class JWTHelper:
    """JWT 生成和验证工具（用于 Agent 间认证）"""

    def __init__(self, private_key_pem: str,
                 public_key_pem: str,
                 algorithm: str = "RS256"):
        self.private_key = private_key_pem
        self.public_key = public_key_pem
        self.algorithm = algorithm

    def create_agent_token(self, agent_id: str,
                           audience: str,
                           permissions: list[str],
                           ttl_minutes: int = 60) -> str:
        """为 Agent 创建 JWT Token"""
        now = datetime.utcnow()
        payload = {
            "iss": "agent-auth-service",      # Issuer
            "sub": f"agent:{agent_id}",       # Subject
            "aud": audience,                  # Audience
            "permissions": permissions,       # 自定义声明
            "iat": now,                       # Issued At
            "exp": now + timedelta(minutes=ttl_minutes),  # Expiry
            "jti": f"agent_{agent_id}_{int(time.time())}", # JWT ID
        }
        return jwt.encode(
            payload,
            self.private_key,
            algorithm=self.algorithm,
        )

    def verify_agent_token(self, token: str,
                           expected_audience: str = None
                           ) -> Dict:
        """验证 Agent Token"""
        options = {"require": ["exp", "iss", "sub", "aud"]}
        if expected_audience:
            options["audience"] = expected_audience

        payload = jwt.decode(
            token,
            self.public_key,
            algorithms=[self.algorithm],
            options=options,
        )
        return payload

📺 流式处理 (SSE/WebSocket)

Server-Sent Events · WebSocket · 流式Agent输出 · 双向通信 · 重连机制

流式处理是提升 Agent 用户体验的核心技术。当一个 Agent 需要 10 秒才能完成完整推理时，如果用户看到的是一个白屏 10 秒后突然出现的完整回答，体验极其糟糕。流式处理让 Agent 的推理过程以字或词为单位逐步展示——就像 ChatGPT 的打字效果。这不仅改善了感知延迟（用户立即看到进展，等待感大幅降低），还允许用户在不满意中间结果时提前中断，节省不必要的 Token 消耗。Agent 的流式处理比纯 LLM 流式更复杂，因为它需要处理"推理-工具调用-观察-推理"循环中的流式输出。

Server-Sent Events (SSE) 是实现 Agent 流式输出的标准协议。SSE 是单向的——服务端向客户端推送事件流，客户端通过标准的 EventSource API 接收。每个事件是一条以 data: 开头的文本行，可以包含多个字段（id、event、data、retry）。在 Agent 场景中，SSE 流可以包含多种事件类型：token（LLM 输出的每个 token）、tool_call（Agent 决定调用工具的详细信息）、tool_result（工具执行完成）、thinking（推理过程的可见部分）、done（Agent 完成整个任务）、error（出错信息）。SSE 的优势在于使用标准 HTTP 协议，穿透代理和防火墙无需额外配置，浏览器原生支持 EventSource，自动处理重连。

WebSocket 提供全双工双向通信——Agent 不仅向用户推送流式输出，用户也可以随时向 Agent 发送新消息或中断指令。WebSocket 适合需要实时交互的 Agent 场景：用户不断提供额外信息或澄清需求、多用户协作通过 Agent 协调、Agent 在执行过程中需要用户确认关键步骤（Human-in-the-Loop）。WebSocket 的建立需要一次 HTTP Upgrade 握手（客户端发送 Upgrade: websocket），之后连接升级为全双工 TCP 通道。在生产环境中，WebSocket 需要处理心跳保活（定期 ping/pong 检测连接状态）、自动重连（网络波动时客户端应自动重新握手）、以及负载均衡中的 Sticky Session（WebSocket 连接必须路由到同一后端实例）。

在 Agent 架构中，流式处理的实现通常在 异步 Agent 循环 中完成。Agent 推理循环的每一步可能产生多个流式事件——LLM 在生成回复时逐 token 流式输出，直到决定调用工具；工具调用的执行可能需要几秒，执行完成后将结果作为流式事件发送；LLM 收到工具结果后继续流式生成。这个循环通过 Python 的 async generator（异步生成器）实现非常自然：每个 yield 对应一个 SSE 事件，Agent 循环是一个连续的异步生成器。客户端可以选择渲染所有中间事件（"debug 模式"）或仅渲染最终文本（"简洁模式"）。

断线重连和消息恢复是流式 Agent 不可忽视的工程细节。SSE 内置了自动重连机制（服务端可通过 retry: 3000 指定重连间隔），但重连后服务端需要知道从哪里继续——否则可能重复推送已发送的消息。解决方案：使用 Last-Event-Id——客户端重连时发送上次收到的最后一个事件 ID，服务端据此判断从哪个事件之后继续推送。对于 WebSocket，重连逻辑需要应用层自行实现：维护一个 message_id_counter，服务端将每条消息持久化到 Redis（带 TTL），客户端重连时发送 last_seen_message_id，服务端补发其后的所有消息。

⏰ 速率限制与配额管理

Token Bucket · Sliding Window · 并发限制 · Exponential Backoff · 配额追踪

速率限制 (Rate Limiting) 是 Agent 系统的"交通管制"——确保 Agent 不会因高频工具调用而压垮下游 API、超额消费配额、或触发 API 提供商的限流封禁。在 Agent 推理循环中，LLM 可能在数秒内决定调用 3-5 个工具——如果每个工具都立即执行，API 调用爆发可能在短短几秒内耗尽每分钟配额。更糟的是，Agent 可能会因为工具返回错误而持续重试，形成恶性循环：请求越多 → 限流越紧 → 更多重试 → 更严重的限流。有效的速率限制机制是 Agent 可靠性的基石。

Token Bucket（令牌桶）算法是最经典且最灵活的速率限制算法。想象一个以固定速率（如每秒 N 个）填充令牌的桶，桶有最大容量（burst 上限）。每次 API 调用需要消耗一个令牌；如果桶中有令牌则立即执行；如果没有则需等待或拒绝请求。Token Bucket 的关键特性是允许突发流量——如果桶是满的（最大容量如 10），可以立即连续执行 10 次调用，然后速率降为填充速率。这使得 Agent 可以"攒一波"爆发调用（如多轮推理后需要调用多个工具），然后回归正常速率。相比之下，Fixed Window（固定窗口）更适合简单的每分钟计数限制场景。

对于 Agent 系统，还需要并发请求限制——不仅限制请求速率，还限制同时在进行中的请求数量。为什么需要并发限制？因为 LLM API 也有并发限制（如同时最多 10 个并行请求），Agent 如果一次性触发过多工具调用并行执行，可能触发 429 错误。并发限制器通常使用 Semaphore（信号量）实现：在工具调用前 acquire()，执行完成后 release()。Semaphore 的初始值应设置为下游 API 允许的最大并发数。结合 Token Bucket（速率限制）和 Semaphore（并发限制），Agent 在所有维度上都是"温柔"的 API 消费者。

指数退避与抖动 (Exponential Backoff with Jitter) 是处理 API 错误重试的标准策略。当 Agent 遇到 429 (限流) 或 5xx (服务端错误) 时，不应立即重试——而应等待逐渐增长的时间间隔。经典策略：第一次重试等待 1 秒，第二次 2 秒，第三次 4 秒，以此类推（指数增长，上限通常 60 秒）。但纯指数退避有一个问题：如果有多个 Agent 实例同时遇到错误，它们会用相同的退避节奏同时重试，造成"雷鸣般的羊群效应 (Thundering Herd)"。解决方案是添加 Decorrelated Jitter——每次退避时间 = min(cap, random_between(base, previous_backoff * 3))。这种非确定性抖动确保多个实例的退避节奏错开，避免同步重试。

配额追踪和预警是生产 Agent 运维的必要组件。Agent 使用的 API 通常有日配额、月配额或按量计费。在生产环境中，你需要：追踪每个工具的调用次数和 Token 消耗量；定义配额告警阈值（如日配额的 80%、95%）；在接近配额时自动切换策略（如使用更便宜的模型、减少不必要的工具调用、降低采样温度减少 Token 消耗）；记录完整的配额使用日志用于成本分析和账单对账。这些数据最终汇总为 Agent 成本仪表盘——让团队了解每个 Agent 实例每天的成本、哪个工具消耗最大、是否有异常飙升。

# ==================== 1. Token Bucket 速率限制器 ====================

class TokenBucket:
    """
    令牌桶速率限制器
    支持同步和异步两种使用方式
    """

    def __init__(self, rate: float, capacity: int,
                 name: str = "default"):
        """
        Args:
            rate: 令牌填充速率（令牌/秒）
            capacity: 桶最大容量（允许的最大突发流量）
            name: 限流器名称（用于日志和监控）
        """
        self.rate = rate
        self.capacity = capacity
        self.name = name
        self.tokens = float(capacity)  # 当前令牌数
        self.last_fill = time.monotonic()
        self._lock = threading.Lock()

        # 统计
        self.total_acquired = 0
        self.total_wait_time = 0.0
        self.total_rejected = 0

    def _refill(self) -> None:
        """补充令牌"""
        now = time.monotonic()
        elapsed = now - self.last_fill
        self.tokens = min(
            self.capacity,
            self.tokens + elapsed * self.rate,
        )
        self.last_fill = now

    def acquire(self, tokens: int = 1,
                timeout: Optional[float] = None) -> bool:
        """
        获取令牌
        Args:
            tokens: 所需令牌数（复杂操作可能需要多个令牌）
            timeout: 最长等待时间（秒），None=无限等待
        Returns:
            True: 获取成功, False: 超时
        """
        with self._lock:
            deadline = (time.monotonic() + timeout
                        if timeout is not None else None)

            while True:
                self._refill()

                if self.tokens >= tokens:
                    self.tokens -= tokens
                    self.total_acquired += tokens
                    return True

                if deadline and time.monotonic() >= deadline:
                    self.total_rejected += 1
                    return False

                # 计算需要等待的时间
                wait_time = (tokens - self.tokens) / self.rate
                if deadline:
                    wait_time = min(
                        wait_time,
                        deadline - time.monotonic()
                    )

                if wait_time > 0:
                    self.total_wait_time += wait_time
                    time.sleep(wait_time)

    async def async_acquire(self, tokens: int = 1,
                            timeout: Optional[float] = None
                            ) -> bool:
        """异步获取令牌"""
        with self._lock:
            deadline = (time.monotonic() + timeout
                        if timeout is not None else None)

            while True:
                self._refill()

                if self.tokens >= tokens:
                    self.tokens -= tokens
                    self.total_acquired += tokens
                    return True

                if deadline and time.monotonic() >= deadline:
                    self.total_rejected += 1
                    return False

                wait_time = (tokens - self.tokens) / self.rate
                if deadline:
                    wait_time = min(
                        wait_time,
                        deadline - time.monotonic()
                    )

                if wait_time > 0:
                    self.total_wait_time += wait_time
                    # 释放锁后异步等待
                    # (简化实现: 实际应使用 asyncio.Condition)
                    time.sleep(wait_time)

    @property
    def stats(self) -> Dict:
        """获取统计信息"""
        return {
            "name": self.name,
            "rate": self.rate,
            "capacity": self.capacity,
            "available_tokens": self.tokens,
            "total_acquired": self.total_acquired,
            "total_wait_time_seconds": round(
                self.total_wait_time, 2
            ),
            "total_rejected": self.total_rejected,
        }


# ==================== 2. Exponential Backoff w/ Jitter ====================

def retry_with_backoff(
    max_retries: int = 3,
    base_delay: float = 1.0,
    max_delay: float = 60.0,
    backoff_factor: float = 2.0,
    jitter: bool = True,
    retryable_exceptions: tuple = (Exception,),
    on_retry: Optional[Callable] = None
):
    """
    指数退避重试装饰器（带抖动）

    退避公式（带 Decorrelated Jitter）:
      delay = min(max_delay, base_delay * (backoff_factor ** attempt))
      if jitter: delay = random.uniform(0, delay)
    """
    def decorator(func):
        @wraps(func)
        def wrapper(*args, **kwargs):
            last_exception = None

            for attempt in range(max_retries + 1):
                try:
                    return func(*args, **kwargs)
                except retryable_exceptions as e:
                    last_exception = e

                    if attempt == max_retries:
                        break

                    # 计算退避时间
                    delay = min(
                        max_delay,
                        base_delay * (backoff_factor ** attempt)
                    )
                    if jitter:
                        delay = random.uniform(0, delay)

                    if on_retry:
                        on_retry(
                            func.__name__, attempt + 1,
                            delay, str(e)
                        )

                    time.sleep(delay)

            raise last_exception
        return wrapper
    return decorator


# ==================== 3. 并发限制器 (Semaphore) ====================

class ConcurrencyLimiter:
    """并发请求限制器"""

    def __init__(self, max_concurrent: int,
                 name: str = "default"):
        self.semaphore = threading.BoundedSemaphore(
            max_concurrent
        )
        self.max_concurrent = max_concurrent
        self.name = name
        self.active_count = 0
        self.total_acquired = 0
        self.total_wait_time = 0.0

    def acquire(self, timeout: Optional[float] = None) -> bool:
        """获取并发槽位"""
        start = time.monotonic()
        result = self.semaphore.acquire(
            timeout=timeout
        )
        if result:
            self.total_wait_time += time.monotonic() - start
            self.active_count += 1
            self.total_acquired += 1
        return result

    def release(self) -> None:
        """释放并发槽位"""
        self.active_count -= 1
        self.semaphore.release()

    def __enter__(self):
        self.acquire()
        return self

    def __exit__(self, *args):
        self.release()


# ==================== 4. 组合式 Agent 调用保护器 ====================

class AgentCallProtector:
    """
    Agent API 调用的完整保护层
    组合: Token Bucket (速率) + ConcurrencyLimiter (并发) + Retry (退避)

    使用方式:
      protector = AgentCallProtector(rpm=60, max_concurrent=5)
      result = protector.call(tool_func, *args, **kwargs)
    """

    def __init__(self, rpm: int = 60,
                 max_concurrent: int = 5,
                 burst_size: int = None,
                 max_retries: int = 3):
        self.rate_limiter = TokenBucket(
            rate=rpm / 60.0,         # 转换为每秒速率
            capacity=burst_size or max_concurrent * 2,
        )
        self.concurrency_limiter = ConcurrencyLimiter(
            max_concurrent=max_concurrent,
        )
        self.max_retries = max_retries

    def call(self, func: Callable,
             *args, acquire_timeout: float = 30.0,
             **kwargs) -> Any:
        """
        安全地调用一个 Agent 工具函数
        1. 检查并发限制
        2. 获取速率令牌
        3. 带退避重试执行
        """

        # 获取并发槽位
        if not self.concurrency_limiter.acquire(
            timeout=acquire_timeout
        ):
            raise RuntimeError(
                "并发槽位获取超时——"
                "系统负载过高，请稍后重试"
            )

        try:
            # 获取速率令牌
            if not self.rate_limiter.acquire(
                timeout=acquire_timeout
            ):
                raise RuntimeError(
                    "速率限制——请求频率过高，"
                    "请降低调用频率后重试"
                )

            # 带退避重试执行
            for attempt in range(self.max_retries + 1):
                try:
                    result = func(*args, **kwargs)
                    return result

                except Exception as e:
                    if attempt == self.max_retries:
                        raise

                    # Decorrelated Jitter 计算
                    base = min(60.0, 2 ** attempt)
                    delay = random.uniform(
                        0, min(60.0, base * 3)
                    )
                    time.sleep(delay)

        finally:
            self.concurrency_limiter.release()

@dataclass
class QuotaTracker:
    """配额使用追踪器"""

    daily_limit: int        # 日配额
    warning_thresholds: list[float] = field(
        default_factory=lambda: [0.5, 0.75, 0.9, 0.95]
    )
    _usage: int = 0
    _last_reset: float = field(
        default_factory=time.time
    )

    def consume(self, amount: int = 1) -> bool:
        """消费配额"""
        # 检查是否需要重置（新的一天）
        now = time.time()
        if now - self._last_reset > 86400:
            self._usage = 0
            self._last_reset = now

        if self._usage + amount > self.daily_limit:
            return False

        self._usage += amount
        return True

    def usage_pct(self) -> float:
        """当前使用百分比"""
        return self._usage / self.daily_limit * 100

    def check_warnings(self) -> list[str]:
        """检查是否需要告警"""
        pct = self.usage_pct()
        warnings = []
        for threshold in self.warning_thresholds:
            if pct >= threshold * 100:
                warnings.append(
                    f"配额使用已达 {threshold*100:.0f}%"
                )
        return warnings

    def remaining(self) -> int:
        """剩余配额"""
        return max(0, self.daily_limit - self._usage)
